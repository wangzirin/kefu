## 1. 本片结论

本轮只完成“定位器草稿”，不是连接器。

当前可形成的真实结论只有一个：

> 抖音网页个人私信弹窗可以作为 RPA Draft-Only 研究对象，前提是操作者可见、默认不发送、探测后清空草稿、且不保存私聊内容。

不能形成以下结论：

- 这不是抖店飞鸽客服接入。
- 抖店飞鸽客服已经接通。
- 抖音企业客服已经接通。
- 抖音官方自动回复已经接通。
- 可以无人值守自动发送。
- 可以用于客户交付的高并发客服通道。

## 2. 新增 Profile

```text
research/rpa_browser_reply_feasibility/profiles/douyin_web_dm.draft_only.locator_profile.json
```

该 profile 只描述脱敏结构：

| 区域 | 记录内容 | 边界 |
|---|---|---|
| 页面身份 | `douyin.com/user/self`、私信弹窗可见 | 不记录账号 ID |
| 会话列表 | 左侧会话列表区域 | 不记录联系人、群名、未读数 |
| 当前会话 | 右侧聊天区域 | 不记录消息原文、图片、视频卡片 |
| 回复框 | 占位或可访问名为“发送消息”的输入区 | 只允许填入临时草稿 |
| 发送控件 | 输入框附近的发送按钮或图标 | Draft-Only 下必须禁止点击 |
| 清空草稿 | 聚焦、全选、删除、确认空 | 禁止按 Enter |

## 3. 为什么先做 Profile，不直接跑真实页脚本

真实平台页面和本地 mock 页面差别很大：

- 登录态可能不在 CDP 调试 Profile 里。
- 页面可能由弹窗、虚拟列表、动态节点或不可稳定选择器组成。
- 私信内容天然包含个人隐私。
- 误触发送按钮会直接外部写入。
- 个人私信页面和商家客服页面的业务语义不同。

因此下一步如果要做真实页 Smoke，也必须以这个 profile 为边界，先由操作者确认页面可见，再只做草稿填入和清空。

## 4. 安全门禁

本 profile 固定以下策略：

- `send_allowed=false`
- `auto_send_allowed=false`
- `must_not_click_send=true`
- `press_enter_blocked=true`
- `auto_clear_draft_required=true`
- `raw_private_data_persistence_allowed=false`
- `capture_screenshot_allowed=false`
- `store_cookie_token_localstorage_allowed=false`

也就是说，它不能被用于自动发送，也不能被用于沉淀私聊语料。

## 5. 后续可选路线

### 路线 A：继续 P3-06U-12I

只做一次操作者介入的 Draft-Only Smoke：

1. 用户打开测试页面并确认可以被观察。
2. 系统或操作者定位输入框。
3. 填入一条明显的临时测试草稿。
4. 不点击发送、不按 Enter。
5. 立即清空草稿。
6. 只记录“定位成功、草稿出现、草稿清空、未发送”，不记录任何私聊内容。

### 路线 B：暂停 RPA 真实页面研究

如果当前优先级转回可交付产品，应暂停个人网页私信路线，回到：

- 官方 API / 服务商授权。
- 企业微信/微信客服官方回调。
- 抖店飞鸽或商家平台的官方授权能力。
- 多渠道中台正式配置、审计、队列和人工审核。

## 6. 当前建议

建议继续做一次 P3-06U-12I，但仍然只做 Draft-Only。原因是：这一步能回答“当前电脑已登录的真实网页，能否被安全地作为 RPA 副驾驶研究对象”，而不会把它误包装成正式渠道接入。

如果 12I 失败，结论也有价值：说明该类页面不适合作为稳定 RPA 入口，应优先投入官方渠道和商家后台授权路径。
