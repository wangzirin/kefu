日期：2026-07-02

本文件把 P3-06U-25 的成熟度评分、云朵AI视频参考、RPA 研究边界、知识三页现状、渠道账号模型、前后端对齐原则和产品化总控手册合并成下一阶段施工总纲。

它不是客户宣传稿，也不是已完成能力清单。它只回答一个问题：接下来要把当前系统从“受控试点可演示”推进到“更接近真实商用客服中台”，应该按什么顺序修、修到什么验收标准、哪些边界不能越过。

## 5. 产品主链路

下一阶段所有优化必须围绕这条主链路检查，不再用“页面多、功能多”证明成熟度。

```text
渠道账号/店铺 -> 入站消息 -> 业务对象识别 -> 知识命中 -> 回复策略 -> 自动回复/人工门禁 -> 质量复盘 -> 知识修复 -> 回归验证
```

每一条真实消息至少要能回答：

1. 来自哪个渠道账号、店铺、入口和会话。
2. 识别到哪个客户、哪个业务对象、哪个问题类型。
3. 命中了哪些知识，引用是否足够。
4. 为什么可以自动回复，或者为什么要转人工。
5. 如果失败，失败归因是知识、策略、模型、渠道、外发还是人工处理。
6. 修复后能不能回放原问题并证明质量提升。

## 6. 前端优化线

### 6.1 去演示文案

目标：客户视角不再出现内部研究、演示样本和开发调试口径。

施工要点：

- 增加环境口径：内部研究环境可以显示调试边界，客户演示环境显示正式产品文案。
- 联系人中心、知识运营、质量复盘、渠道接入、管理运维统一清理“演示模式”字样。
- 真实外发继续关闭时，只在必要的状态条、按钮旁和危险动作处说明，不把页面写成说明书。

### 6.2 微信式多渠道对话台

目标：多渠道对话台成为接待主工作区，而不是人工审核中心。

施工要点：

- 左侧会话列表进一步收窄，突出头像、客户名、平台标识、最后消息、未读和风险标签。
- 主筛选只保留“全部 / 我的 / 转人工”；待审、待发、工单/SLA 降为后台状态和深链，不作为坐席主入口。
- 右侧优先展示真实对话流，普通会话展示 AI 自动回复记录，人工无需操作。
- 低置信、高风险、无知识、渠道异常、客户要求人工时才出现转人工提醒和人工接管回复区。
- 内部备注、审核意见、待发送草稿、工单 SLA 不再压在普通会话默认首屏。

### 6.3 联系人和线索减噪

目标：联系人中心只承担客户画像和历史关系，不重复工作台里的会话处理。

施工要点：

- 联系人中心保留客户基础信息、来源渠道、最近会话、线索状态、跟进记录。
- 多渠道对话台负责实时接待和 AI 回复预案。
- 线索跟进负责销售动作和下一步计划。
- 不在联系人页堆重复的对话台、审核台或质量复盘内容。

### 6.4 知识三页真实分叉

目标：知识库运营、知识缺口、知识评测三页不只是路由分叉，还要形成不同工作任务。

施工要点：

- 知识库运营：维护业务对象、对象问答卡、文档、片段、引用来源、发布记录。
- 知识缺口：收集无命中、低置信、引用不足、人工驳回、策略阻断的问题，并生成修复任务。
- 知识评测：管理题集、运行评测、比较发布前后结果、输出质量变化。
- 三页可以共享数据，但首屏、动作、筛选、指标和空状态必须不同。

### 6.5 运维页减噪

目标：管理运维服务内部维护和客户交付，不压过客服主流程。

施工要点：

- 管理运维保留运维与告警、模型路由、系统设置三类。
- 默认显示健康、风险、需要处理的告警和最近运行，不堆过多工程指标。
- 诊断包、备份恢复、升级回滚等放入明确的运维动作区。

### 6.6 总览到处理路径强化

目标：运营总览不只好看，还能直接进入处理动作。

施工要点：

- 核心指标围绕自动回复覆盖率、人工门禁率、知识缺口、渠道健康、失败复盘。
- 每个图表或卡片必须能进入相应处理页：对话台、知识缺口、质量复盘、渠道账号、运维告警。
- 弱化纯展示型表格，优先保留能驱动处理的趋势、异常和排行。

## 7. 后端优化线

### 7.1 真实服务端数据深化

前端所有核心页面应逐步减少演示样本依赖，优先读取服务端数据。无法读取时必须明确显示配置缺失、权限不足或未接入，而不是自动切成看起来很真实的假数据。

### 7.2 渠道账号配置面板

`channel_accounts` 已有后端第一片。下一阶段要把它变成可运营配置：

- 租户级列表：`GET /api/tenants/{tenant_id}/channel-accounts`
- 渠道级新增/更新：`POST /api/channels/{channel_id}/channel-accounts`
- 前端字段：平台、账号、店铺、入口、授权状态、接入状态、回复模式、健康状态、最近同步时间。
- 敏感凭据不展示，`public_profile` 只放脱敏公开信息。

### 7.3 入站消息统一信封

所有入口最终都要进入统一信封，避免官网、企微、公众号、电商平台、RPA 研究线各自生成一套数据结构。

统一信封至少包括：

- `tenant_id`
- `channel_account_id`
- `platform`
- `conversation_external_id`
- `message_external_id`
- `sender_external_id`
- `message_type`
- `text`
- `attachments`
- `business_context`
- `received_at`
- `trace_id`
- `provider_mode`

RPA 研究线也可以产生统一信封，但 `provider_mode` 必须标识为 `rpa_draft_only` 或等价研究模式。

### 7.4 回复策略状态机增强

现有 `reply_decisions` 要继续增强为完整状态机：

- `auto_reply_ready`：知识足够、低风险、可进入外发前置门禁。
- `manual_gate_required`：需要人工确认或接管。
- `draft_only`：只能生成草稿，不能外发。
- `blocked_by_policy`：政策、敏感、渠道、配置或权限阻断。
- `sent_pending_receipt`：后续真实外发白名单测试才允许出现。
- `send_failed`：发送失败并进入失败复盘。

状态机必须记录原因、命中知识、置信度、风险标签、业务对象、模型路由和外发门禁。

### 7.5 Outbox、回执和失败复盘闭环

真实外发继续关闭。在未来单独授权真实外发前，工程上仍要先补齐 outbox、回执和失败复盘的模型准备：

- outbox 草稿不等于已发送。
- 人工确认不等于已发送。
- 渠道计划不等于已发送。
- 已发送必须来自官方授权通道或正式服务商授权返回。
- 失败必须进入重试、人工接管、渠道配置修复或质量复盘。

## 8. 知识系统优化线

### 8.1 真实 50-100 题库

下一阶段必须有一个能导入真实脱敏题库的模板。建议字段：

| 字段 | 说明 |
| --- | --- |
| `case_id` | 题目编号 |
| `customer_question` | 客户原始问法，需脱敏 |
| `expected_answer` | 人工确认的标准答案 |
| `business_object` | 商品、服务、套餐、课程或项目 |
| `must_include` | 答案必须包含的事实点 |
| `must_not_include` | 禁止承诺、禁止编造或高风险话术 |
| `handoff_expected` | 是否应该转人工 |
| `risk_tags` | 投诉、退款、医疗、法律、价格、售后等风险标签 |
| `source_reference` | 答案依据来源 |

### 8.2 最终答案事实性评测

当前知识评测是检索评测，不是完整客服准确率。后续要把评测拆成四层：

1. 检索层：是否召回正确知识和引用。
2. 生成层：最终答案是否忠实于引用。
3. 策略层：是否正确自动回复、转人工或阻断。
4. 业务层：客户问题是否被解决，是否产生投诉或二次追问。

### 8.3 业务对象知识批量导入

云朵AI视频里最值得学习的是“商品学习”而不是界面样式。我们的抽象应是“业务对象知识”：

- 电商：商品、SKU、活动、售后规则。
- 教培：课程、班型、老师、价格、排期。
- 服务：服务套餐、交付范围、禁用承诺、退款条件。
- B2B：产品模块、合同边界、实施周期、技术规格。

### 8.4 发布前后对比

每次知识发布要能看到：

- 发布前命中率、引用覆盖、期望词覆盖。
- 发布后相同题集表现。
- 新增知识解决了哪些缺口。
- 是否引入新的错误、禁用承诺或转人工误判。

### 8.5 生产级检索路线

短期继续使用当前结构化知识和本地检索能力；中期进入 pgvector 或等价向量库；成熟期采用混合检索：

```text
结构化 FAQ / 业务对象卡 -> BM25 或关键词召回 -> 向量召回 -> 重排 -> 引用裁剪 -> 模型回答 -> 事实性评测
```

生产级检索不能只追求模型回答顺滑，必须保留引用溯源、版本、发布记录和回归题集。

## 9. 渠道接入优化线

### 9.1 正式线

正式交付只接受：

- 平台官方 API。
- 平台开放能力。
- 服务商授权。
- 客户授权的正式后台能力。
- 经过白名单和测试账号验证的 sandbox。

正式线必须具备：

- 官方授权。
- 测试白名单。
- 入站验签。
- 回执。
- 失败重试。
- 审计闭环。
- 可关闭的外发开关。

### 9.2 RPA 研究线

RPA 研究线只用于内部研究和副驾驶：

- 读取页面上下文。
- 生成草稿。
- 填入回复框但不点击发送。
- 采集动作证据。
- 验证回复策略是否合理。

RPA 不进入正式默认交付链。任何基于个人号、网页后台模拟点击、私有协议、cookie 复用或群控的路径，都不能写进正式交付方案。

### 9.3 当前外发边界

真实外发继续关闭。

后续即使进入平台 sandbox，也必须先满足：

1. 用户明确授权。
2. 平台规则单独核验。
3. 测试账号或白名单可用。
4. 出站消息进入 outbox。
5. 人工确认或策略门禁通过。
6. 平台回执落库。
7. 失败重试和审计记录完整。

## 10. 运维交付优化线

### 10.1 客户账号

下一阶段不先追求复杂组织权限，但必须有真实登录和客户环境基础账号：

- 管理员账号。
- 日常运营账号。
- 只读观察账号。
- 密码重置和禁用流程。
- 登录失败和权限不足提示。

### 10.2 部署

建议继续保持两条落地方式：

- 托管云端版：我们维护云服务器、数据库、备份、升级和监控；客户通过浏览器登录。
- 私有化部署版：部署到客户云或客户服务器；我们通过授权窗口、诊断包和远程维护流程支持。

### 10.3 备份、恢复和升级

最低交付要求：

- 数据库备份脚本。
- 文件和知识包备份说明。
- 空库迁移 smoke。
- 版本升级 checklist。
- 回滚路径。
- 恢复演练记录。

### 10.4 诊断包

诊断包应只读收集：

- 版本号。
- 配置摘要。
- 健康检查。
- 最近 worker 状态。
- 最近失败任务。
- 告警规则命中。
- 数据库连接状态。
- 不包含 secrets、token、密码、私钥。

### 10.5 月度质量复盘

面向售后运营，每月输出：

- 自动回复覆盖率。
- 人工门禁率。
- 知识缺口排行。
- 错因分布。
- 渠道健康。
- 模型成本。
- 客户投诉或高风险会话。
- 本月知识修复和下月计划。

## 13. 不变边界

这些边界在 P3-06U-26 全阶段保持不变，除非后续用户单独明确授权：

- 真实外发继续关闭。
- 当前知识评测是检索评测，不是完整客服准确率。
- RPA 不进入正式默认交付链。
- 当前中台前端只按桌面客服工作台设计和验收，不再做移动端、手机端或 390 视口验收。
- 真实平台外发需要官方授权、测试白名单、回执、失败重试和审计闭环。
- 任何客户真实数据、平台密钥、账号密码、token、cookie、私钥不得写入文档、日志、诊断包或前端。
- 任何“已接通微信/抖音/淘宝/京东/拼多多自动回复”的说法，必须以官方授权、入站验签、出站回执和白名单测试为证据。
