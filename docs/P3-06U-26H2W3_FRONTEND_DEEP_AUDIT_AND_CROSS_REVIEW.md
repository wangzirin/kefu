# P3-06U-26H2W3 前端深度审查与交叉验证

日期：2026-07-03

定位：内部工程审查文档。本文记录 H2W-3 质量闭环第一片完成后的前端全量巡检、子智能体交叉审查、脚本验证结果和下一步整改优先级。本文不是客户版宣传材料，不承诺真实外发、全渠道已接通或完整线上准确率已经完成。

## 一、当前结论

H2W-3 第一片已经完成“线上回执与准确率闭环”的前端质量页入口和门禁表达：质量页能够把本地样本质量、人工标签质量、真实线上回执拆开展示，并明确真实外发关闭、真实平台回执未接入前不展示完整线上准确率。

本轮深度审查没有发现 P0 级“客户可见按钮能点但完全无动作”的假控件，也没有发现横向溢出、运行时崩溃或“已自动发送 / 已接通全渠道 / 已完成线上准确率”这类高危越界承诺。

但当前前端仍没有达到成熟商用客服中台的标准。交叉验证后，最重要的问题集中在三类：

1. 信息架构仍有旧后台路由泄漏：运营总览、质量复盘仍把客户可见动作直达 `#reviews`、`#outbox`、`#conversations` 等隐藏后台页。
2. 多渠道对话台还不够像真实客服 IM：左侧会话列表可读余量不足，右侧仍有较强系统记录感，距离“纯粹的客服消息窗口”还有一轮重构。
3. 客户可见文案里仍有工程词：`API`、`sandbox`、`outbox`、`connector_noop`、`P3-06` 等词仍出现在部分页面，应转为中文业务语言或放入研发/运维详情。

## 四、子智能体交叉审查结果

本轮使用三个子智能体做独立审查，再由主线程交叉验证。

### 1. 功能真实性审查

结论：

- 没有发现 P0 级裸假按钮。
- 登录、总览筛选、知识写入、质量报告、签收记录、渠道账号登记、诊断包、备份和更新相关动作大多有前后端链路或明确边界。
- `FRONTEND_FUNCTION_REALITY_MATRIX.md` 已经落后于当前页面状态，需要更新。
- 渠道账号表单原文“新增 / 更新账号身份”容易让客户以为已经接入真实平台，已改为“登记 / 保存账号身份”。
- 会话搜索原文“搜索客户/聊天记录”容易让客户以为支持全局历史检索，已改为“搜索当前会话”。

主线程复核：

- H2W-0 浏览器门禁通过，确认没有回归到假控件。
- 深审脚本没有发现 P0 假按钮或未标记按钮。
- `FRONTEND_FUNCTION_REALITY_MATRIX.md` 确实需要下一轮更新。

### 2. 信息架构审查

结论：

- 左侧导航已比早期清楚，但旧的隐藏后台页仍然从总览和质量复盘直接暴露。
- 会话收件箱、人工审核、待发送、工单/SLA 的概念已经不适合继续作为客户主流程入口。
- 多渠道对话台应该承接日常客服处理，隐藏页只应作为内部路由或功能层，不应让客户感知成多个并列工作台。
- 知识三页应保留，但“知识评测”建议转成客户更容易理解的“发布前检查 / 题库回归”。

主线程复核：

- 深审脚本抓到两个 P1：运营总览和质量复盘仍直达隐藏后台路由。
- 这是当前最高优先级信息架构问题。

### 3. 视觉与文案审查

结论：

- 当前视觉约 `6.4/10`，已经不像空壳，但仍偏工程控制台。
- 总览 BI 相对较好，但首屏信息仍偏满。
- 多渠道对话台已经是双栏，但消息流还像系统记录，不像纯客服聊天气泡。
- 质量复盘过于复杂，像阶段验收台，建议拆成“运营复盘”和“审计/签收”两层。
- 渠道接入页信息全，但配置、合规和边界说明混在一起。

主线程复核：

- 深审脚本抓到 `API`、`sandbox`、`outbox`、`connector_noop` 等客户可见工程词。
- 质量页可视化通过，但页面复杂度仍偏高。
- 会话列表尺寸为 `198 x 79`，属于可用但紧张，应继续优化。

## 六、交叉验证后的真实问题清单

### P1-1 客户主流程不能继续直达隐藏后台路由

现状：

- 总览和质量复盘仍有大量动作跳往 `#reviews`、`#outbox`、`#conversations`、`#tickets`。
- 这些路由在旧设计里有意义，但当前产品已经收束到“客服接待 / 知识运营 / 质量复盘 / 渠道接入 / 运维设置”的主结构。

建议：

- 总览中的高风险、待发送、低置信、草稿池等入口统一跳到 `#live` 并带队列筛选，或跳到质量复盘的对应折叠区。
- 质量复盘中的低置信、高风险、待发送、入站会话入口统一改成“查看相关会话 / 生成修复任务 / 进入知识缺口 / 进入渠道复盘”，不要暴露旧后台路由名。
- 保留旧路由作为内部兼容，但不出现在客户可见链接中。

验收：

- 深审脚本中 `hiddenBackendLinks` 对总览和质量复盘返回空数组。

### P1-2 多渠道对话台要继续向真实 IM 收束

现状：

- 当前已经去掉顶部假按钮和附件假工具。
- 右侧主区域仍更像消息时间线，部分系统记录感过强。
- 左侧会话列表密度偏紧。

建议：

- 左侧列表进一步仿照成熟客服软件：头像、客户名、渠道小标、最后一句、时间、未读/转人工状态；减少业务对象、长状态和多重标签。
- 右侧主屏只保留客户气泡、客服气泡、AI 草稿气泡、少量系统弱提示。
- 转人工只作为会话列表小状态和输入区状态，不再大块说明。
- AI 决策、知识依据、失败回执进入右侧抽屉或质量复盘，不占据主聊天流。

验收：

- 旧脚本中“AI 自动回复 / AI 建议主屏文案”断言应删除。
- 新脚本应检查：真实气泡结构、输入区固定、会话列表不拥挤、转人工状态可识别、无假发送。

### P1-3 质量复盘需要拆日常复盘和审计签收

现状：

- 质量页同时承载错因 BI、H2W 闭环、客户报告、签收记录、人工标签和多种修复入口。
- 对内部验收有用，但客户日常使用会显得重。

建议：

- 首屏只保留：本月结论、最大错因、待修复项、待回归项、客户报告状态。
- 人工标签、客户确认记录、报告留档、签收记录归入“月度报告 / 交付记录”二级区域。
- H2W、P3 等工程阶段名不得出现在客户主屏。

验收：

- 质量首屏可以在 1440x900 内说明“哪里错、下一步修什么”。
- 工程词扫描不再命中 `H2W`、`P3-06`、`API`、`outbox` 等词。

### P2-1 客户可见工程词要统一清理

现状：

- 深审脚本发现 `API`、`sandbox`、`outbox`、`connector_noop`、`P3-06` 仍出现在客户可见区域。

建议替换：

- `API` -> `接口`、`系统连接`、`后台连接状态`
- `sandbox` -> `测试环境`、`测试接入`
- `outbox` -> `待发送记录`、`发送前确认`
- `connector_noop` -> `未配置发送器`、`暂未连接正式通道`
- `P3-06`、`H2W` -> 只保留在内部文档和证据目录，不出现在客户主屏

验收：

- 深审脚本工程术语扫描只允许在内部调试页、研发模式或 tooltip 里出现；客户主页面不出现。

### P2-2 前端功能真实性矩阵需要更新

现状：

- 矩阵覆盖了 H2W-0 当时的页面，但 H2W-2 和 H2W-3 后新增了客户知识建设中心、知识发布流程、质量闭环和更多报告签收动作。

建议：

- 下一轮更新 `FRONTEND_FUNCTION_REALITY_MATRIX.md`，把所有新增按钮、链接、导出、发布、标签、签收、报告入口补进矩阵。
- 对每一项标记：真实动作、真实跳转、禁用说明、隐藏、仍需修复。

验收：

- 矩阵覆盖当前左侧导航全部页面和质量页新增链接。
- 深审脚本发现的新链接都能在矩阵中找到对应记录。
