# P3-06UI 角色化导航与默认入口第二片

日期：2026-07-01  
范围：标准运营版客服中台前端导航显示层、默认入口、角色提示  
状态：第二片完成

## 1. 本轮目标

在 P3-06UI 第一片“7 个工作域”的基础上，继续把不同角色看到的入口收窄：

- 坐席不要默认看到模型、运维、系统设置等工程管理入口。
- 管理员保留完整运营、渠道、模型和运维入口。
- 只读角色保留总览、质量复盘、渠道健康等观察入口。
- 坐席进入系统时默认落到 `多渠道对话台`，而不是运营总览。

本轮仍然只是前端显示层和默认入口收口，不替代后端 RBAC。真实权限仍以后端 owner/admin/agent 校验为准。

## 2. 角色入口矩阵

| 角色 | 默认入口 | 可见工作域 | 说明 |
| --- | --- | --- | --- |
| owner | 运营总览 | 总览、工作台、客户、知识运营、质量复盘、渠道接入、管理运维 | 完整管理视图 |
| admin | 运营总览 | 总览、工作台、客户、知识运营、质量复盘、渠道接入、管理运维 | 完整运营管理视图 |
| agent | 多渠道对话台 | 工作台、客户、知识运营 | 日常接待、客户画像、知识查询；不默认看到模型、运维、设置 |
| viewer | 运营总览 | 总览、质量复盘、渠道接入 | 只读观察视图，不承诺后端所有接口已支持 viewer |
| 未知角色 | 运营总览 | 受限显示 | 保守 fallback |

## 3. 当前已完成

- `navigationGroups` 支持 `visibleTo`。
- 新增 `getNavigationGroupsForRoles`，按 `auth.user.roles` 过滤工作域和子入口。
- 新增 `getDefaultNavigationHrefForRoles`，owner/admin 优先默认进入 `#overview`，纯坐席默认进入 `#live`，viewer 默认进入 `#overview`。
- App 侧使用 `visibleNavigationGroups` 渲染侧边栏。
- 如果用户通过 URL hash 进入了当前角色不可见的页面，前端会回到该角色默认入口。
- 侧边栏新增当前视图提示，例如“管理员视图”“坐席视图”“只读视图”。
- 保留现有 hash 路由，不破坏原有页面链接。

## 4. 没有完成的内容

| 未完成项 | 原因 | 后续建议 |
| --- | --- | --- |
| 后端完整 RBAC 权限矩阵 | 本轮只做前端导航可见性 | 单独进入 RBAC 收口阶段 |
| 按角色禁用所有页面动作 | 许多按钮已有 `canManage`，但还未全量矩阵化 | 下一轮结合后端权限逐页审查 |
| viewer 的后端只读接口 | 当前 viewer 只是前端显示概念 | 后续补真实 viewer API 策略 |

## 5. 验收标准

- `agent` 角色看不到 `总览`、`质量复盘`、`渠道接入`、`管理运维`。
- `agent` 角色默认入口是 `#live`。
- `owner/admin` 角色能看到全部 7 个工作域。
- `viewer` 角色能看到总览、质量复盘、渠道接入。
- 访问不可见 hash 时，前端会 fallback 到角色默认入口。
- 前端构建通过。
- readiness 脚本通过。

## 6. 边界说明

这次改动提升客户演示和日常使用清晰度，但不能写成“完整权限系统已经完成”。原因是：

- 前端隐藏菜单不是安全边界。
- 真实接口权限仍以后端 `require_any_role` 和服务层校验为准。
- 有些页面动作仍需要逐页整理成完整权限矩阵。
- viewer 还没有完整后端只读能力验收。
