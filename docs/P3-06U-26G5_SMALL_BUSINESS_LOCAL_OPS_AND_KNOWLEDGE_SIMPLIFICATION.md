# P3-06U-26G5 小微企业本地化运维与知识运营简化

## 目标

本阶段把产品主线重新收束到“小型、小微企业本地化使用”的真实场景。系统不再把大型企业的复杂坐席体系、实验室入口和过重审批流放在第一优先级，优先保障三件事：

1. AI 能自动回复常见问题，遇到无把握、风险高、无知识命中的问题才转人工。
2. 知识库能被本地人员维护，也能通过我们远程提供的知识包、安全更新包进行升级。
3. 部署到客户本地后，系统能定期生成诊断包，经客户授权上传给我们，用于远程排障、知识修复和版本更新。

## 本轮已收束的内容

| 模块 | 调整 | 原因 |
| --- | --- | --- |
| RPA 副驾驶实验室 | 从中台侧边栏和主路由下线 | 该能力仍属于内部研究线，不适合放进小微企业正式工作台 |
| 知识缺口 | 改为服务端筛选：状态、严重度、来源、关键词和分页都由后端处理 | 避免只筛当前页，后续真实数据量增加后仍可用 |

| 前端文案边界 | 继续去掉实验味、演示味和研究入口 | 对客户只呈现可理解、可操作的业务功能 |

## 知识运营三页的保留判断

### 知识库运营

保留。它是客户本地维护资料的入口，负责 FAQ、课程/产品说明、价格政策、售后政策、禁用承诺、转人工规则等内容。

下一步要继续简化为：

- 新增知识。
- 批量导入。
- 发布/停用。
- 搜索。
- 查看被哪些问题命中。

不需要在小微版里做复杂的知识生命周期、多人审批和大型权限流。

### 知识缺口

保留，但改名和交互可以更接近“待补知识”。它不是一个复杂工单系统，而是客服系统自动发现的“这里答不好”的清单。

来源只保留三类主线：

- AI 无知识命中。
- AI 低置信或风险高而转人工。
- 评测题失败。

本轮已把列表筛选改为服务端查询。下一步应继续把操作收束为：

- 生成知识草稿。
- 加入回归题。
- 标记已处理。
- 标记不需要补充。

不需要做复杂分派、复杂审批和跨部门流转。

### 知识评测

保留，但定位必须改成“上线前检查”和“更新后回归”，不是研究型评测平台。

小微企业真实需要的是：

- 导入 50-100 条真实常见问题。
- 每次知识库更新后自动跑一遍。
- 标出答错、引用不足、应该转人工却自动回复的问题。
- 把失败结果沉淀到知识缺口。

当前知识评测仍偏检索与规则评测，不能等同于完整客服准确率。下一步要补“最终答案事实性、禁用承诺、转人工正确性”的评测口径。

## 本地化部署的账号策略

不建议做真正的“完全免登录”。原因很直接：本地系统里会有客户消息、渠道密钥、知识库、诊断包和更新包，如果完全无登录，客户内网任何能访问页面的人都可能改知识、看会话或导出数据。

推荐小微版采用“首次启动注册 + 长期保持登录”：

1. 第一次启动时进入初始化页面。
2. 客户创建第一个管理员账号。
3. 系统生成本地租户和管理员。
4. 默认勾选保持登录。
5. 后续在同一台内网电脑访问时近似免登录。
6. 需要换人或离职时，管理员可以重置账号。

本地演示环境可以允许免登录，但正式本地部署不建议无认证运行。

## 诊断包与远程维护机制

客户本地部署后，建议采用“客户授权上传诊断包 + 我们返回签名更新包”的运维模式。

### 诊断包包含什么

默认包含：

- 系统版本。
- 数据库迁移版本。
- 最近服务健康状态。
- 最近错误日志摘要。
- 渠道连接状态。
- 模型调用成功率和失败原因统计。
- 知识库数量、发布时间、命中统计。
- 知识缺口数量、主要错因。
- 评测通过率、失败题摘要。
- 配置指纹。

默认不包含：

- API Key。
- Cookie。
- 密码。
- 私钥。
- 原始客户完整聊天记录。
- 客户手机号、微信号等直接个人标识。

如必须排查真实对话问题，应由客户在本地手动勾选并脱敏后上传。

### 上传方式

小微版推荐两种方式：

1. 手动上传：客户点击“生成诊断包”，确认内容后上传。
2. 定期上传：客户启用后，每周或每天定时上传摘要包。

默认优先手动上传，避免客户担心隐私和联网风险。

### 我们如何远程修复

1. 我们读取诊断包。
2. 判断问题属于配置、知识库、模型、渠道、版本还是数据异常。
3. 生成修复方案。
4. 如需要改知识库，生成“知识更新包”。
5. 如需要改程序，生成“版本更新包”。
6. 更新包必须带版本号、变更说明、签名和回滚说明。
7. 客户本地管理员导入更新包。
8. 系统先备份，再执行更新。
9. 更新后自动跑健康检查和知识回归。
10. 失败时回滚到更新前版本。

## 后续工程拆分

### P3-06U-26H1：本地首次启动账号

目标：客户本地第一次启动时可以创建管理员账号，不再依赖开发者预置账号。

验收：

- 无管理员时显示初始化页。
- 创建第一个管理员后自动进入中台。
- 已有管理员时进入登录页。
- 支持保持登录。

### P3-06U-26H2：诊断包生成

目标：本地系统可以生成不含敏感凭据的诊断包。

验收：

- 可以手动生成诊断包。
- 诊断包包含版本、健康、日志摘要、知识统计、评测摘要。
- 静态检查确认不包含 API Key、密码、Token、Cookie。

### P3-06U-26H3：诊断包上传

目标：在客户授权后上传诊断包到我们的服务端。

验收：

- 上传前展示摘要和风险提示。
- 支持失败重试。
- 后端记录上传时间、包版本、校验值。
- 上传失败不影响本地客服系统运行。

### P3-06U-26H4：签名更新包导入

目标：我们可以把知识修复、配置修复或版本修复打成包，由客户本地导入。

验收：

- 更新包必须验证签名。
- 更新前自动备份。
- 更新后自动健康检查。
- 失败时可回滚。

## 当前边界

- 本轮没有实现完整登录注册系统。
- 本轮没有实现诊断包上传服务。
- 本轮没有开启真实平台外发。
- 本轮没有把 RPA 作为正式交付能力。
- 本轮完成的是产品主线收束、RPA 可见入口下线、知识缺口服务端筛选和后续本地化运维蓝图。
