# P3-05J 会话工作台与质量复盘

## Engineering Control Card

- Stage: P3-05J 会话工作台与质量复盘。
- Goal: 把商户中控台从“会话表格 + 各种运营模块”继续推进成更接近真实客服坐席日常使用的工作区，并补上用于观察回复质量变化的质量复盘页。
- User concern: 当前中控台不能只是总览页里堆模块；客服系统需要有像抖音、淘宝、京东、拼多多、微信客服那类对话处理界面，也需要有能发现回复质量下降、转人工升高、知识缺口和发送失败的运营页。
- Scope: 仅新增前端工作区和演示样例呈现；复用现有会话、人工审核、待发送、失败复盘、知识评测数据结构。
- Non-goal: 不宣称任何第三方平台已经真实接通；不打开真实外发；不把检索命中率包装成 AI 完整准确率；不新增生产 BI 数据仓库。

## 这轮新增了什么

### 多渠道对话台

新增左侧导航入口 `对话台`，对应 `#live` 工作区。

页面结构改为更接近坐席使用习惯的三栏：

- 会话列表：展示客户、渠道、最后一条消息、等待时间、SLA 轻量提示和优先级。
- 对话区：展示客户入站、AI 草稿、系统门禁、坐席操作入口和审核/待发送跳转。
- 证据与风险区：展示处理建议、知识证据、草稿状态、发送风险、失败复盘和渠道边界提示。

当前演示渠道包含：

- 微信客服（演示）
- 抖音店铺（演示）
- 淘宝客服（演示）
- 京东客服（演示）
- 拼多多客服（演示）
- 官网客服

这些渠道只用于统一会话结构和界面验收，不代表真实平台 API 已经接通。

### 质量复盘

新增左侧导航入口 `质量`，对应 `#quality` 工作区。

页面不做泛泛的漂亮大屏，而是围绕客服系统真正需要盯住的质量闭环：

- 强证据命中：有引用候选、置信度达到阈值、且不是高风险问题。
- 引用覆盖：来自知识评测运行结果。
- 转人工：进入人工审核池或需要人工处理的比例。
- 知识缺口：题库缺口和无知识命中样本。
- 待确认草稿：人工批准后仍需发送前确认的草稿。
- 失败复盘：回执异常、队列阻断和发送失败问题。
- 高风险：赔付、法务、投诉、敏感或高责任问题。
- 人工标签：当前仍未完成事实性标签，不报完整准确率。

## 为什么这样做

真实商户客服中台不是只看一张列表。坐席每天要回答的是一条条会话，运营负责人要看的也不是抽象图表，而是质量是否开始变差，以及变差后该修知识、改提示词、改路由、补人工审核还是修渠道。

所以本轮把两个关键工作区拆出来：

- 对话台解决“坐席怎么日常接待客户”的问题。
- 质量复盘解决“系统是否正在变差，变差从哪里开始”的问题。

这比继续堆总览卡片更接近可商用客服系统。

## 当前仍然不能承诺

- 不能承诺抖音、淘宝、京东、拼多多、小红书、公众号或企业微信已经完成真实自动回复。
- 不能承诺真实外发已经打开。
- 不能承诺完整 SLA、工单、主管质检、权限矩阵和多租户计费后台已经完成。
- 不能承诺 AI 准确率已经达到某个百分比；当前只有检索、人审、失败队列和题库回归信号。
- 不能把演示数据混进真实客户报表。

## 下一步建议

优先继续三条线：

1. 补真实官方渠道 smoke：企业微信/微信客服公网 HTTPS 回调、可信 IP、access token、open_kfid 和白名单测试消息。
2. 补商用中台后台能力：完整 RBAC、坐席团队、工单/SLA、主管视图、联系人画像、线索跟进。
3. 补质量闭环后端化：人工事实性标签、投诉/差评事件源、知识缺口状态流、质量事件表和客户月报口径。

本轮结论：`对话台` 和 `质量复盘` 已经达到前端工作区第一版可演示状态，但它们仍是标准运营版中台的一部分，不等于真实全渠道自动回复已经上线。
