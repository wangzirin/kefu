# P3-06U-26H2W3B 客户知识维护四步流程收束

日期：2026-07-03

## 1. 阶段目标

本阶段只处理知识运营页的客户可见入口和验收门禁，不重写后端知识模型，不新增真实渠道接入，不调用模型，不打开真实外发。

目标是让小微企业负责人能理解：以后维护知识库不是进入一堆工程模块，而是按四步把业务资料整理成客服可用知识。

四步固定为：

1. 业务对象：先建立商品、服务、套餐、课程、门店等对象和别名。
2. 标准问答：把客户会怎么问、标准答案和触发关键词绑定到业务对象。
3. 流程政策：导入售后、发票、退款、质保、服务流程等文档。
4. 禁用承诺与转人工规则：维护不能承诺的话术和必须转人工的风险词。

## 2. 本次改动

- 把“客户知识建设中心”改为“客户知识维护向导”，强调按四步维护。
- 四个知识层级改为第一步到第四步，避免“层级说明”和“更新路径”重复出现。
- 删除客户可见的旧 `knowledge-update-path` 流程卡，避免同一件事被解释两遍。
- 把“客户知识发布流程”收束为“启用与回归检查”。
- 把“选择草稿 / 样题试跑”改为“选择待启用文档 / 回归题检查”。
- 把“自动回复状态机”改为“自动回复处理方式”。
- 把“AI 回复预案”改为“回复草稿”。
- 把“编辑知识草稿”改为“导入知识文档”。

## 3. 保留的真实能力

这些动作仍然沿用已有真实链路：

- 新增和编辑业务对象：写入 `business_objects`。
- 绑定标准问答：写入对象问答卡。
- 导入知识文档：写入知识文档和分块。
- 预检和导入知识更新包：调用知识更新包预检和导入接口。
- 发布前样题试跑：调用 `publish-checks`。
- 确认发布版本：调用 `publication`。
- 自动回复策略：写入本地自动回复策略表。

## 4. 继续保留的边界

- 真实外发继续关闭。
- 知识发布只影响本地检索范围，不会自动发到微信、抖音、淘宝或其它平台。
- 知识评测仍不是完整线上客服准确率。
- 本轮不处理 XLSX、PDF、DOCX 解析签收。
- 本轮不做生产级 RAG、向量库替换、模型成本治理或真实渠道回执。

## 5. 验收门禁

本片新增门禁：

```bash
python3 scripts/check_p3_06u_26h2w3b_customer_knowledge_flow.py
```

必须满足：

- 页面存在 `data-h2w3b-customer-knowledge-flow="true"`。
- 四步入口分别存在业务对象、标准问答、流程政策、禁用承诺与转人工规则。
- `knowledge-update-path` 不再出现在客户可见前端和样式中。
- `编辑知识草稿`、`自动回复状态机`、`AI 回复预案` 不再出现在客户可见前端。
- 功能真实性矩阵更新到 H2W3B。

## 6. 下一步建议

如果继续知识方向，下一片建议做“知识导入模板和客户资料签收”：

- XLSX/CSV 第一片：客户按模板上传业务对象、标准问答和禁用规则。
- PDF/DOCX 第一片：先做文件接收、hash、预检和人工确认，不自动入库。
- 资料签收第一片：客户确认本次知识包已预检、已导入、已回归，形成留档。

如果转回产品成熟度主线，下一片可继续做云接收台、真实更新恢复或生产级 RAG/模型成本治理。
