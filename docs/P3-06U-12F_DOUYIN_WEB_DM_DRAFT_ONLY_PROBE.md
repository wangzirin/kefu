# P3-06U-12F 抖音网页私信 Draft-Only 真实页探测

日期：2026-07-02

## 1. 本片目标

本片只做 RPA 技术研究，不做官方渠道接入，不做真实自动发送。

目标是确认：在一个真实第三方网页消息窗口里，RPA 是否可以完成以下最小链路：

```text
识别当前私信窗口 -> 定位消息输入框 -> 写入测试草稿 -> 不点击发送 -> 清空草稿
```

## 2. 页面类型

本次页面是抖音网页端个人私信弹窗。

它适合验证“私信式输入框能否被 RPA 写入草稿”，但不能等同于：

- 抖店飞鸽客服后台。
- 抖音企业号私信后台。
- 抖音开放平台官方消息接口。
- 商家客服的订单、售后、工单和队列能力。

因此本片结论只覆盖输入框与草稿机制，不覆盖商用客服后台能力。

## 3. 实测方式

当前既有脚本 `scripts/run_p3_06u_12e_rpa_browser_reply_feasibility.mjs` 默认通过 Chrome DevTools Protocol 连接 `http://127.0.0.1:9227`。

本次检查发现：`9227` 只暴露了本地调试 Chrome Profile，没有看到用户当前已登录的抖音个人 Chrome 标签页。因此本次真实页探测没有用 CDP 脚本直接附着，而是使用 macOS Accessibility / Computer Use 读取和操作当前可见页面。

这意味着：

- 真实页面可被操作系统辅助功能层识别。
- 当前 RPA CDP Driver 还不能稳定接管这个已登录页面。
- 后续若要脚本化复跑，需要新增 Accessibility Adapter，或使用专门带 remote debugging 的 Chrome 测试 Profile 登录抖音。

## 4. 已验证能力

本次已验证：

| 能力 | 结果 |
|---|---|
| 识别私信弹窗 | 通过 |
| 识别会话列表区域 | 通过 |
| 识别当前会话标题 | 通过 |
| 识别消息输入框 | 通过 |
| 写入测试草稿 | 通过 |
| 写入后发送按钮变为可用 | 通过 |
| 不点击发送 | 通过 |
| 清空测试草稿 | 通过 |

测试草稿内容使用了明显的内部标识：

```text
【RPA测试草稿-请勿发送】收到，我先看一下。
```

写入成功后已清空，没有发送给任何真实联系人。

## 6. 客观结论

结论：可以作为 P3-06U-12F 的真实页 Draft-Only 探测样本。

它证明：

- 真实第三方网页私信窗口中，消息输入框可以被定位。
- RPA 可以把回复草稿写入输入框。
- 页面不会因为写入草稿自动发送。
- 在严格不点击发送的前提下，可以作为低风险研究样本。

但它没有证明：

- 抖店飞鸽客服后台可用。
- 抖音企业号客服接口可用。
- 抖音官方允许这种方式长期运行。
- 可以无人值守自动发送。
- 可以承载多坐席、多店铺、多订单售后场景。

所以对外或对工程主线的正确表述是：

```text
个人私信网页的 RPA draft-only 机制已被真实页验证；商家客服后台和官方渠道能力仍需单独验证。
```

## 7. 下一步建议

下一步进入 `P3-06U-12G RPA Adapter 抽象`，建议先做两个适配器边界：

1. `cdp_browser_adapter`
   - 面向可用 remote debugging 的测试 Chrome Profile。
   - 适合本地 mock、可控后台、后续专门登录的测试账号。

2. `accessibility_browser_adapter`
   - 面向当前这种已登录的真实 Chrome 页面。
   - 只允许 draft-only。
   - 默认写入后可自动清空。
   - 不提供自动点击发送能力。

同时继续保留硬边界：

- 默认不发送。
- 默认不保存原始私聊数据。
- 默认不读取或保存 Cookie、Token、LocalStorage。
- 真正点击发送必须由用户明确声明测试对象、测试账号和允许发送边界。
