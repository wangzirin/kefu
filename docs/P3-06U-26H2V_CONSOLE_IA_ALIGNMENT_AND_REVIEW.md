# P3-06U-26H2V 中台信息架构对齐与前后端审查

日期：2026-07-03

## 1. 本轮结论

本轮围绕“七个点的完善优化”做了一次前端、后端和多渠道对话台的交叉审查，并修复了用户点名的三个结构问题：

1. 运营总览的时间范围和渠道筛选增加了口径证明，明确当前结果是服务端聚合已对齐，还是本地汇总兜底。
2. 渠道接入页改为四层结构：账号与入口、企业微信步骤、其它官方平台、接入边界，避免一页堆满所有信息。
3. 管理运维拆为三个真实页面：运维与告警、模型路由、账号安全，不再使用一个共用页面加内部 tabs 的结构。
4. 多渠道对话台的文案收紧为“AI 回复建议 / 真实外发关闭”，不再暗示已经完成真实平台自动发送。

严格按生产闭环口径判断：七个点里，第三点“真实渠道官方接入 / 真实外发”仍按原计划暂缓；其余六点都有第一片或多片能力，但仍不能说已经完整生产闭环完成。

1. 真实 IM 消息流闭环。
2. 线上回执与真实准确率闭环。
3. 质量报告 XLSX/PDF/DOCX 与正式签收模型。
4. 云接收台、真实程序更新和在线恢复运维链路。
5. 生产级 RAG、模型能力契约和成本治理。

## 3. 子智能体交叉审查摘要

### 后端审查

后端并不是空壳，但多个模块仍是第一片或 dry-run：

- `reply_decision` 能做策略判断，但 `external_write_allowed=false`、`force_draft_only=true`，不能说已经自动发给客户。
- 渠道连接器能配置账号和生成发送计划，但发送器仍是 noop / dry-run，生产回执 reconciliation 尚未完成。
- 管理运维能做只读观测、本地诊断包、备份和更新包预检，但自动上传、真实程序更新、在线恢复、告警通知还没做。
- 知识系统支撑度最高，但当前知识评测仍主要是检索和客服答案质量第一片，不是完整线上客服准确率。

### 前端信息架构审查

- 总览筛选有“按钮已切换但服务端数据未必同步”的风险，本轮已补筛选口径证明。
- 渠道接入页内容过密，本轮已拆成四层。
- 管理运维左侧有三个入口，但右侧原本共用一个内部 tabs，本轮已拆成三个真实页面并删除旧共用组件。
- 仍需后续处理：隐藏路由、伪分页、部分列表的真分页和更多服务端分页。

### 多渠道对话台审查

当前方向合理，但还不是成熟客服 IM：

- 左侧会话列表方向正确。
- 右侧现在仍偏“事件流 + AI 决策记录”，不是完整消息流。
- 后端已有会话详情接口雏形，下一步应让前端按会话加载真实 messages。
- 文案不能再写“自动回复中”，因为真实外发没有打开。本轮已改为“AI 回复建议 / 真实外发关闭”。

## 6. 下一步建议

下一步不要继续泛化做新页面，优先进入“真实 IM 消息流闭环第一片”：

1. 前端在多渠道对话台选中会话后调用会话详情接口。
2. 右侧只渲染真实 inbound / outbound / system 消息。
3. AI 回复建议、转人工接管、知识依据和外发状态作为轻量抽屉或底部记录，不再挤占主对话区。
4. 后端给会话和消息补最小渠道身份字段：`channel_account_id`、外部联系人标识、外部会话标识、消息类型、消息状态。
