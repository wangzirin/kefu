# P3-06F 中台信息架构与布局审查

日期：2026-07-01  
范围：标准运营版客服中台当前前端、P3-06F 运维告警规则第一片、导航结构、核心工作区布局。  
结论：当前视觉风格已经稳定，整体美观层面可以继续保留；真正需要处理的是信息架构偏散、导航层级过长、角色视角混杂和部分首屏密度偏高。

## 2. 总体评分

| 维度 | 当前评分 | 判断 |
| --- | ---: | --- |
| 视觉完成度 | 8.0 / 10 | 色彩、留白、卡片边界、字体层级已经像一套正式中台，不是临时 demo。 |
| 功能必要性 | 7.5 / 10 | 大部分模块有商业理由，但有些模块不适合继续作为同级菜单暴露给所有角色。 |
| 信息架构 | 6.2 / 10 | 16 个一级入口过多，坐席、运营、管理员、工程运维混在一层。 |
| 工作流顺畅度 | 7.0 / 10 | 对话台方向正确，但“会话、审核、待发送、对话台”仍像同一条回复链被拆成多个入口。 |
| 布局稳定性 | 7.0 / 10 | 桌面端主要布局可用；高内容页首屏略长，部分底部内容会被系统 Dock 或窗口高度压住。 |
| 可商用中台感 | 7.2 / 10 | 已经有商用产品雏形；下一步需要做角色化导航和运营闭环收束。 |
| 可访问性风险 | 6.5 / 10 | 文本和对比度基本可读；仍需系统验证键盘焦点、表格语义、非颜色提示和移动端长页。 |

综合分：7.1 / 10。  
这不是“视觉差”的问题，而是“产品结构还没有完全收口”的问题。

## 3. 功能必要性评估

### 必须保留为正式能力

| 能力 | 判断 | 原因 |
| --- | --- | --- |
| 总览 | 保留 | 管理员进入系统后的业务状态入口。 |
| 对话台 | 强保留 | 这是坐席真正工作的主界面，应该成为日常默认入口。 |
| 审核 | 保留，但应并入工作台二级 | AI 草稿人审是我们区别 FAQ 工具的核心能力。 |
| 待发送 | 保留，但应并入工作台二级 | outbox 门禁是防误发的关键边界。 |
| 工单 | 保留 | SLA、售后问题和跨天跟进需要结构化承载。 |
| 联系人 / 线索 | 保留，但可合并到“客户”分组 | 客户画像和销售跟进有业务价值，但不必长期占两个同级入口。 |
| 知识库 / 缺口 | 强保留，但应合并为“知识运营” | 知识库更新和缺口修复是准确率维护的根。 |
| 质量 | 强保留 | 用来发现错因、投诉、转人工异常和知识缺口，是运营复盘核心。 |
| 渠道 | 保留，管理员入口 | 官方渠道接入、健康、失败复盘必须存在，但不应打扰普通坐席。 |
| 运维与告警 | 保留，管理员/运维入口 | P3-06F 放在运维页里是正确选择，不应单独新增一级菜单。 |

### 应降级或合并的入口

| 当前入口 | 建议 | 理由 |
| --- | --- | --- |
| 会话 | 并入“工作台”二级页 | 与对话台、审核、待发送强相关，不应割裂。 |
| 审核 | 并入“工作台”二级页 | 坐席日常应在对话台或工作台完成，不应频繁跳多个一级页面。 |
| 待发送 | 并入“工作台”二级页 | 与审核和 outbox 是同一条回复链。 |
| 缺口 | 并入“知识运营”二级页 | 缺口不是独立业务域，是知识库维护流程的一部分。 |
| 评测 | 并入“知识运营”或“质量”二级页 | 普通客户不需要把评测作为一级菜单理解。 |
| 模型 | 放入“管理设置”或仅管理员可见 | 商户日常不应频繁接触模型路由细节。 |
| 设置 | 保留，但只露出已落地项 | 当前仍有规划态内容，正式演示时容易显得未完成。 |

## 4. 布局与交互问题

### 问题 1：一级导航过长

当前左侧 16 个入口在桌面窗口里已经接近底部，系统 Dock 显示时下方入口容易被压住。这个问题不是单纯加滚动条能解决，因为它会让用户觉得“这套系统太散”。

建议下一轮改成 7 个主分组：

| 新一级入口 | 包含内容 |
| --- | --- |
| 总览 | 今日队列、风险摘要、渠道状态、质量摘要 |
| 工作台 | 对话台、会话池、审核、待发送、工单 |
| 客户 | 联系人、线索、标签、跟进记录 |
| 知识运营 | 知识库、缺口、发布、回归题库、评测 |
| 质量复盘 | 错因 BI、转人工趋势、投诉/高风险、修复动作 |
| 渠道接入 | 官网、企微、公众号、电商渠道、授权状态 |
| 管理运维 | 模型、运维告警、外发开关、权限、设置 |

### 问题 2：对话台方向正确，但首屏仍偏“监控台”

对话台现在已经接近成熟客服软件的思路：左侧队列、中间会话、右侧检查器。问题在于顶部指标和提示条占用了较多首屏高度，真正的消息流被压到偏下位置。

建议：

- 保留 4 个核心指标，不再横向堆 8 个筛选卡。
- 把筛选条件收进二级筛选条或可折叠区域。
- 坐席默认进入时，应优先看到“当前会话列表 + 选中会话消息 + AI 草稿/证据”。
- 顶部演示说明在正式客户版本中应只保留轻量状态提示，不应长期占一整条大横幅。

### 问题 3：质量页的价值方向正确，但需要更像“修复驾驶舱”

用户真正关心的是：错在哪里、为什么错、谁来修、修完有没有变好。质量页不能只是漂亮图表，必须有修复路径。

建议质量页固定四段：

1. 当前健康度：准确率、转人工率、投诉/高风险、知识缺口。
2. 错因排行：无知识、错引、政策风险、渠道失败、模型不确定。
3. 修复队列：缺口补知识、FAQ 调整、提示词调整、渠道配置修复。
4. 修复后回归：本周修复项、回归结果、未通过项。

### 问题 4：P3-06F 告警规则放置正确，但运维页后续不能继续堆

P3-06F 没有新增一级菜单，而是放在“运维与告警”里，这是对的。告警规则是运维的一部分，不应该让普通坐席看到。

后续运维页建议改为二级标签：

| 标签 | 内容 |
| --- | --- |
| 进程健康 | worker heartbeat、运行记录、最近错误 |
| 告警规则 | P3-06F 规则、触发状态、runbook |
| 指标出口 | P3-06G 指标快照、Prometheus 草案 |
| 外发边界 | kill switch、白名单、外部写入审计 |
| 诊断包 | 只读导出、环境检查、部署检查 |

## 5. 不建议继续新增的功能

短期不建议继续新增新的一级功能。当前更重要的是收口。

| 不建议新增 | 原因 |
| --- | --- |
| 新的“报表中心”一级页 | 质量页已经承担 BI，继续拆会更散。 |
| 新的“告警中心”一级页 | 目前没有真实通知通道，放进运维即可。 |
| 新的“模型实验室”一级页 | 会把客户注意力带到工程细节，不利于商用演示。 |
| 新的“平台市场/插件市场” | 当前核心链路还没正式试点，不该提前扩张。 |
| 新的独立“评测报告中心” | 评测是门禁和复盘，不应重新成为主线。 |

## 6. 可访问性与交互风险

| 风险 | 当前证据 | 建议 |
| --- | --- | --- |
| 长导航键盘遍历成本高 | 16 个一级入口，底部入口在当前窗口下难触达 | 改为分组导航，减少一级入口。 |
| 表格语义不完整 | 部分表格使用 `div role=table` | 后续用真实 `table` 或补齐 rowgroup、aria-sort、键盘焦点。 |
| 色彩承载状态过多 | 红、黄、蓝、绿用于多类状态 | 每个状态必须同时有文本标签，不只靠颜色。 |
| 首屏有效工作区被提示条占用 | 总览和对话台都有大块演示说明 | 正式模式移除大横幅，演示模式保留。 |
| 截图验证覆盖不足 | 运维告警页未成功捕获截图 | 下一轮用专门浏览器 QA 脚本或开启可控浏览器环境补截图。 |

## 7. 下一轮建议

建议下一轮单独做 `P3-06UI：信息架构收口第一片`，不要继续叠新功能。

交付目标：

- 把 16 个一级入口收束为 7 个主分组。
- 对话台作为坐席默认工作入口。
- 会话、审核、待发送合并进工作台二级导航。
- 知识库、缺口、评测合并进知识运营。
- 模型、运维、设置只对管理员可见，普通坐席隐藏。
- 正式模式移除大块演示提示，演示模式保留。
- 为总览、对话台、质量、运维四个页面补桌面和移动端截图验收。

优先级判断：

| 任务 | 是否应该立刻做 |
| --- | --- |
| P3-06F 告警规则第一片 | 已完成 |
| P3-06G 指标出口第一片 | 可以做，但会继续偏工程运维 |
| P3-06UI 信息架构收口 | 建议尽快做，能显著提升客户演示质量 |
| 企业微信公网 HTTPS smoke | 公网 URL、Token、AESKey 具备时优先 |

最终建议：如果下一轮目标是“继续工程底座”，走 P3-06G；如果下一轮目标是“让客户一看就觉得成熟”，先做 P3-06UI。
