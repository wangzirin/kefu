# P3-06UI 中台信息架构收口第一片

日期：2026-07-01  
范围：标准运营版客服中台前端导航与工作区入口组织  
状态：第一片完成；P3-06UI 第二片角色化导航已完成，详见 `P3-06UI_ROLE_BASED_NAVIGATION.md`；P3-06UI 第三片管理运维内部 Tab 已完成，详见 `P3-06UI_ADMIN_OPS_TABS.md`

## 1. 本轮目标

把原来 16 个平铺一级入口收束成 7 个对商户更容易理解的工作域，让中台更像可商用客服产品，而不是研发功能清单。

本轮只处理信息架构和导航呈现，不改后端业务逻辑、不改接口、不启用真实外发、不新增真实平台发送能力。

## 2. 调整前问题

| 问题 | 表现 | 风险 |
| --- | --- | --- |
| 一级入口过多 | 总览、会话、工单、对话台、审核、待发送、联系人、线索、知识库、缺口、渠道、模型、评测、质量、运维、设置全部平铺 | 客户演示时难以判断日常工作从哪里开始 |
| 角色视角混杂 | 坐席、主管、知识运营、管理员、运维功能混在同一层 | 普通坐席会看到过多工程和配置入口 |
| 商用叙事不足 | 导航像模块列表，不像客服业务流程 | 对外演示时成熟度被低估 |
| 后续扩展风险 | 每新增一个能力都容易继续加一级菜单 | 中台会越来越难用 |

## 3. 调整后导航结构

| 工作域 | 子功能 | 面向角色 | 主要用途 |
| --- | --- | --- | --- |
| 总览 | 运营总览 | 负责人、主管、管理员 | 看今日队列、风险、渠道健康和关键动作 |
| 工作台 | 多渠道对话台、会话收件箱、人工审核、待发送草稿、工单/SLA | 坐席、主管 | 日常接待、草稿审核、发送前门禁、服务闭环 |
| 客户 | 联系人中心、线索跟进 | 坐席、销售、主管 | 聚合客户画像、线索阶段和下一步动作 |
| 知识运营 | 知识库运营、知识缺口闭环、知识评测 | 运营、主管、管理员 | 管理文档、补知识缺口、用评测守住回复质量 |
| 质量复盘 | 质量诊断 | 主管、运营 | 发现错因、定位知识缺口、推动修复 |
| 渠道接入 | 渠道健康 | 管理员、运营 | 查看官方入口接入状态、入站出站风险 |
| 管理运维 | 运维与告警、模型路由、系统设置 | 管理员、运维 | 管理模型策略、worker 健康、告警规则和系统配置 |

## 4. 当前已完成

- 新增 `navigationGroups`，保留所有原有页面路由和子功能。
- 侧边栏改为“工作域 + 子功能”的两级结构。
- 桌面侧边栏改为独立滚动，避免两级导航在部分高度下压到页面底部。
- 一级导航从 16 个平铺入口收束为 7 个工作域。
- `总览` 阶段标识更新为 `P3-06UI`。
- `运维与告警` 仍保留在 `管理运维` 下，不新增单独“告警中心”一级入口。
- 子功能链接仍使用现有 hash 路由，避免破坏已验证页面。

## 6. 验收标准

- 侧边栏显示 7 个主工作域，而不是 16 个平铺入口。
- 原有 16 个页面入口仍能从子功能进入。
- 不出现新的独立“告警中心”一级入口。
- 管理员/运维类功能归入 `管理运维`。
- 坐席日常动作集中在 `工作台`。
- 知识相关能力集中在 `知识运营`。
- 前端构建通过。
- 静态检查脚本通过。

## 7. 对商用演示的影响

这次改动不会新增客服能力，但会明显降低客户首次理解成本：

- 坐席知道从 `工作台` 进入接待。
- 运营知道从 `知识运营` 和 `质量复盘` 处理知识与错因。
- 管理员知道从 `渠道接入` 和 `管理运维` 看配置、健康和告警。
- 对外演示时，系统结构更接近成熟客服 SaaS 的业务分层。
