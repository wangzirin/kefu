# P3-06U-26H2W 推荐版工程门禁计划 V3

日期：2026-07-03

## 1. 文档定位

本文件是后续智能客服中台继续工程推进时建议采用的主目标文档。

它不是客户宣传材料，不是阶段总结，也不是“把所有想法列出来”的路线图。它的用途只有一个：把当前智能客服中台从“功能已经很多，但真实闭环参差不齐”的状态，收束成可以逐段施工、逐段验收、逐段停止的工程目标。

本文件相对 V2 的主要变化：

1. 每个阶段都补齐进入门禁、验收门禁、停止门禁、证据门禁和最低验证命令。
2. 把“前端图标或按钮只有样子、没有真实功能”的问题升为全局阻断项。
3. 把“演示模式、截图好看、dry-run、静态检查”明确列为不能替代真实完成的证据。
4. 把多渠道对话台、知识建设中心、线上质量闭环、报告签收、云接收台、本地更新恢复、生产级检索和成本治理拆成可独立完成的目标。
5. 保持真实 IM 消息流暂缓，不把企业微信、公众号、抖音、淘宝、京东、拼多多真实外发写入本轮默认目标。

## 2. 当前真实边界

本阶段必须先承认以下事实：

| 事项 | 当前状态 | 不能夸大的说法 |
|---|---|---|
| 多渠道对话台 | 已有客服工作台形态，但仍有部分控件和真实动作需要逐项核验 | 不能说已经完成全平台真实接待和自动回复 |
| 知识运营 | 已有业务对象、知识卡、知识缺口、知识评测、题库导入等基础 | 不能说客户已经可以完整自助运营成熟知识库 |
| 质量复盘 | 已有质量标签、报告、签收和部分错因闭环 | 不能说已经完成线上真实准确率监控 |
| 渠道接入 | 有官方接入路线、provider/webhook 基础和渠道账号模型 | 不能说微信、抖音、淘宝、京东、拼多多都已真实接通 |
| 诊断和更新 | 已有诊断包、签名知识包、策略包、备份恢复和 dry-run 演练 | 不能说已经完成客户环境自动升级系统 |

| 移动端 | 当前主线已经明确不做移动端 | 不能继续用 390px 或手机 QA 作为当前验收目标 |

## 4. 全局工程门禁

### 4.1 全局进入门禁

每次进入任何 H2W 阶段前，必须先完成：

1. 读取本文件。
2. 读取 `PRODUCTIZATION_MASTER_PLAN_LITE_AND_STANDARD.md`。
3. 读取上一阶段阶段卡和输出证据。
4. 确认真实外发是否仍关闭。
5. 确认本阶段是否涉及真实客户数据、真实平台账号、付费模型、生产数据库或真实外部写入。
6. 如果涉及高风险事项，必须拿到用户明确授权后再执行。
7. 在阶段文档中写明本片“做什么、不做什么、怎么验收、何时停止”。

### 4.2 全局验收门禁

任何阶段声称完成，必须同时满足：

1. 代码或文档已经落到固定路径，不是只在对话里说明。
2. 至少有一条机器可重复验证命令通过。
3. 前端改动必须有浏览器 smoke 和截图证据。
4. 后端写动作必须有权限、数据变化、失败反馈和测试。
5. 客户可见页面不得出现超过真实能力的文案。
6. 真实外发关闭时，任何页面不得暗示已经发到微信、抖音、淘宝、京东、拼多多等外部平台。
7. dry-run 只能写 dry-run，不能写已执行。
8. draft-only 只能写草稿，不能写已发送。
9. 本地确认或签收记录不能写成正式电子签章。
10. 检索评测不能写成完整客服准确率。
11. 阶段完成后必须写回 Project_012 的 `执行记录.md` 和 `文件索引.md`；重大取舍写回 `关键决策.md`；踩坑写回 `复盘与采坑.md`。

### 4.3 全局停止门禁

出现以下任一情况，当前阶段必须停止，不允许继续堆新功能：

1. 客户可见按钮点击后无响应，也没有禁用说明。
2. 前端存在写动作，但后端没有接口。
3. 后端有接口，但前端按钮没有接上，却显示为可用。
4. 写动作没有权限校验。
5. 写动作没有数据变化、审计事件或失败反馈。
6. 筛选、分页、排序只改变样式，不改变真实数据口径。
7. 浏览器 smoke 只截首页，不验证关键按钮和关键页面。
8. 演示样本被写成真实客户数据。
9. dry-run 被写成已执行。
10. draft-only 被写成已发送。
11. 真实外发关闭，但页面写成自动发送或外部发送成功。
12. RPA 研究线被写成正式交付默认链路。
13. 未完成项目写回，却声称阶段完成。

## 5. 不能互相替代的证据

| 证据类型 | 能证明什么 | 不能证明什么 |
|---|---|---|
| 文件存在 | 文件已经生成 | 内容正确、功能可用 |
| 静态检查通过 | 关键文本、路径和结构存在 | 浏览器真实可用、接口真实生效 |
| 后端单测通过 | 服务层或接口逻辑可测 | 前端已接入、用户能操作 |
| 前端 typecheck/build | 编译无类型错误 | 页面布局好、功能闭环 |
| 浏览器截图 | 页面能渲染 | 按钮有真实数据变化 |
| 浏览器 smoke | 页面关键交互可走 | 高并发、生产 SLA、真实外部平台可用 |
| dry-run | 本地流程和载荷可验证 | 已真实执行外部动作 |
| demo 模式 | 形态可展示 | 真实客户数据和真实登录可用 |
| 负责人真实登录 smoke | 登录和权限路径可走 | 真实客户环境已部署 |

阶段完成时必须明确说明用了哪些证据，不能用一种证据冒充另一种证据。

## 7. H2W-1：多渠道对话台客服主体验收束

### 7.1 目标

把多渠道对话台做成真实客服软件主工作区：左侧是紧凑会话列表，右侧是大面积真实对话，底部是轻量输入区。AI 决策、外发门禁、质量复盘、知识依据默认不堆在主聊天流里。

### 7.2 进入门禁

1. H2W-0 已通过。
2. 多渠道对话台所有客户可见控件已进入真实性矩阵。
3. 未实现或不适合客户使用的按钮已经隐藏或禁用。

### 7.3 施工内容

1. 左侧会话列表继续瘦身，只保留头像、客户名、渠道、最近一句、等待时间、未读数和小状态。
2. 队列筛选只保留：全部、我的、转人工。
3. 右侧主区只显示消息气泡、轻系统提示和输入区。
4. AI 判断、知识依据、门禁、失败复盘进入抽屉或质量复盘页。
5. 转人工只在左侧小状态和输入区可用状态表达。
6. 点击会话必须刷新右侧消息流。
7. 搜索和筛选必须改变会话列表。

### 7.4 验收门禁

1. `#live` 首屏接近真实客服软件，不是工程说明页。
2. 左侧列表宽度适合大量会话扫描，不是大卡片堆叠。
3. 点击不同会话，右侧消息流真实变化。
4. 全部、我的、转人工三个筛选能改变列表口径或给出真实空状态。
5. 搜索能过滤列表或展示无结果状态。
6. 输入区不会让用户误以为已真实发送外部平台。
7. 页面不出现“AI 自主回复预备”“回复决策进入人工审核”等内部工程口吻。
8. 浏览器 smoke 覆盖搜索、筛选、会话点击、输入区和转人工状态。

### 7.5 停止门禁

1. 对话台仍堆大块 AI 决策卡，停止。
2. 左侧列表无法承载大量会话，停止。
3. 点击会话只改变高亮，不改变消息，停止。
4. 回复框暗示外部平台已发送，停止。
5. 转人工仍以大块重复提醒占据聊天区，停止。
6. 搜索或筛选只是前端样式变化，停止。

## 8. H2W-2：客户知识建设中心

### 8.1 目标

让中小企业客户能自己维护客服知识：录入业务对象、标准问答、流程政策、禁用承诺和转人工规则，经过预检、样题试跑、发布确认后生效。

### 8.2 进入门禁

1. H2W-1 已通过。
2. 知识库运营页新增、编辑、导入、预检入口不再是假按钮。
3. 知识评测已明确不能等同完整客服准确率。

### 8.3 四层知识模型

| 层级 | 客户看到的名称 | 后端对象 | 用途 |
|---|---|---|---|
| 业务对象 | 产品、服务、门店、套餐、活动 | `business_objects` | 判断客户问的对象 |
| 标准问答 | 高频问题、相近问法、标准回复 | `object_knowledge_cards` 或知识卡 | 生成可引用回答 |
| 流程政策 | 售后、退款、发票、物流、保修、预约 | policy 类知识 | 处理流程类问题 |
| 禁用承诺与转人工规则 | 不能承诺、必须人工确认、投诉赔付 | reply policy/rules | 阻断风险回复 |

### 8.4 施工内容

1. 知识库运营页提供四层入口。
2. 新增业务对象可点击、可提交、可落库、可刷新。
3. 标准问答可绑定业务对象。
4. 流程政策有分类、适用范围、生效状态。
5. 禁用承诺和转人工规则能进入自动回复策略，不能只停留在文字说明。
6. 支持客户导入 50-100 条脱敏问题和知识包模板。
7. 发布前有预检和样题试跑。
8. 发布后记录版本、发布人、发布时间和影响范围。

### 8.5 验收门禁

1. 新增业务对象按钮可点击，并能真实保存。
2. 刷新页面后能读取新增业务对象。
3. 新增问答卡能绑定业务对象，并在列表出现。
4. 导入模板 dry-run 能显示新增、跳过、错误和风险。
5. 发布前评测能读取题库或样题。
6. 发布确认能生成版本记录。
7. 禁用承诺和转人工规则至少能被自动回复策略读取或在策略页面显示待接入状态。
8. 客户可见错误提示没有工程黑话。

### 8.6 停止门禁

1. 新增业务对象点不了，停止。
2. 保存后只改前端状态、不写后端，停止。
3. 导入只有上传框，没有后端预检，停止。
4. 发布前没有评测或预检，停止。
5. 知识评测仍写成完整客服准确率，停止。
6. 禁用承诺规则无法进入回复策略或质量复盘，停止。

## 9. H2W-3：线上回执与准确率闭环

### 9.1 目标

建立“问题 -> 回答 -> 人工标签 -> 错因 -> 知识修复 -> 回归验证”的质量闭环。真实平台回执未完成前，指标只能叫本地样本质量、人工标签质量或试点质量，不能叫全量线上准确率。

### 9.2 进入门禁

1. H2W-2 已通过。
2. 有客户题库或脱敏样题。
3. 有人工事实性标签入口。
4. 知识修复能回到知识建设中心。

### 9.3 施工内容

1. 支持最终回复样本导入和采样。
2. 支持人工事实性标签：正确、部分正确、错误、需人工、禁用承诺风险。
3. 支持错因分类：无知识、知识过旧、引用不足、策略错误、模型误判、渠道异常。
4. 支持错因到知识缺口、知识草稿、自动回复策略的跳转。
5. 支持发布后回归验证。
