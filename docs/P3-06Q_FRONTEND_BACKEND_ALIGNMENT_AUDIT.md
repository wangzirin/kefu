# P3-06Q 前后端对齐与前端产品审查

日期：2026-07-01

本轮目标：暂停备案、部署和字段脱敏细化，先审查标准运营版中台的前后端是否对齐，以及当前前端作为智能客服中台的功能性、美观性和商用试点可用度。

## 结论

当前系统已经不是 FAQ 页面，也不是空壳控制台。它具备会话、人工审核、待发送草稿、失败复盘、知识库、质量诊断、工单、联系人、线索、渠道健康、运维指标和 RBAC 的标准运营版骨架，可以作为受控试点中台继续推进。

但它还不能被描述成“成熟可公网商用发布的全渠道智能客服系统”。主要原因不是页面不存在，而是前后端权限契约还未完全闭合，真实渠道配置中心不足，坐席主工作台还没有完成编辑、审核、发送的一屏闭环，生产队列、真实外发、正式告警和高并发压测仍未完成。

| --- | ---: | --- |

| 商用发布度 | 4.2 / 10 | 安全封口、真实渠道、队列、告警、压测都不足 |
| 总体阶段分 | 6.6 / 10 | 可试点，不可正式公网商用 |

## 前后端对齐情况

对齐较好的部分：

- 后端 OpenAPI 有 82 条路径，前端 `client.ts` 中的主要业务接口都能找到对应后端路由。
- 会话、人工审核、outbox、失败复盘、知识库、工单、联系人、线索、渠道、运维指标都有前后端连接。
- RBAC 权限矩阵已成体系，`owner/admin/agent/viewer` 已有明显分层。
- 前端按钮级权限已经开始读取 `user.permissions`，不再完全依赖角色名。
- 构建、类型检查和后端全量测试通过。

主要不对齐：

1. 基础接口仍未鉴权

`backend/app/api/tenants.py` 中租户、渠道、联系人基础接口没有登录或权限依赖：

- `POST /api/tenants`
- `GET /api/tenants`
- `POST /api/tenants/{tenant_id}/channels`
- `GET /api/tenants/{tenant_id}/channels`
- `POST /api/tenants/{tenant_id}/contacts`
- `GET /api/tenants/{tenant_id}/contacts`

这些适合早期 foundation/bootstrap，不适合试点公网或客户环境。

2. 无 token `/api/auth/me` 返回 bootstrap owner

`GET /api/auth/me` 在无 token 时会返回演示 owner。它方便开发演示，但生产或试点环境必须受开关保护，否则语义危险。

3. workflow、inbound worker、reply orchestration 仍主要是“已登录”而不是命名权限

前端按钮已经用 `conversation.manage`、`outbox.*` 等权限控制，但部分后端接口仍是 `require_current_principal`。这会造成前端比后端更严格，后端最终安全边界不一致。

4. 前端登录后会批量拉取所有模块

`App.tsx` 在登录成功后会同时刷新会话、工单、联系人、线索、outbox、审核、失败复盘、发送队列、运维、知识、评测、缺口等资源，没有按 `user.permissions` 分流。owner/admin 问题不大，但 viewer 或更细角色会天然触发多处 403，造成错误状态噪声。

## 前端产品评价

优点：

- 左侧工作域已经收敛成总览、工作台、客户、知识运营、质量复盘、渠道接入、管理运维，结构明显比早期长页更合理。
- 对话台方向正确：左侧会话队列，中间对话与 AI 草稿，右侧客户/AI/知识/运营上下文。
- 质量页已经能表达错因排行、质量漏斗、题库趋势、渠道异常、知识缺口热力、人审矩阵和样本下钻。
- 知识页有文档、分块、发布记录、回滚、检索和分页，已经像运营后台。
- 页面整体风格统一，按钮、卡片、导航、状态标签、分页和空态都有基本规范。

主要短板：

1. 坐席不能在主工作台完成完整回复闭环

对话台的 AI 草稿更像只读预览。坐席缺少直接编辑回复、快捷话术、内部备注、引用确认、二次生成、审批意见和一键进入发送门禁的自然流。

2. 渠道页不像真实接入配置中心

当前渠道页更像“沙盒状态说明 + 健康表 + 失败复盘”。它缺少企业微信、公众号、电商平台的连接器配置表、Webhook URL 验证状态、Token/AESKey/Secret 引用状态、授权状态、回执日志入口和官方配置步骤。

3. 系统设置仍是规划态

导航中仍出现 `规划`、`P3-06UI`、`P3-06F` 等工程阶段标签。对内可以保留，对外演示会显得还在施工。

4. BI 是方向正确，但还不是成熟 BI

质量页有较好的看板结构，但图表多为手写 CSS 表达，缺少时间范围、渠道/团队/坐席筛选、tooltip、图例、趋势对比和点击下钻到修复任务的完整交互。

5. 巨型前端文件影响后续维护

`frontend/src/App.tsx` 约 9463 行，`styles.css` 约 4454 行。当前能跑，但继续叠功能会让页面拆分、权限调整、布局调整和回归验证越来越重。

## 风险分级

P0，上线或客户试点前必须处理：

- 关闭或保护未鉴权的 tenant/channel/contact foundation 接口。
- 无 token bootstrap owner 仅允许开发模式，试点/生产必须关闭。
- workflow、reply orchestration、trusted inbound worker 等接口迁移到命名权限。
- 试点部署安全封口：强 secret、数据库和 Redis 不公网暴露、HTTPS 反代、请求体限制、基础 rate limit。

P1，下一轮产品化优先处理：

- 对话台补“编辑草稿 -> 审核意见 -> 发送前确认 -> outbox 状态”的一屏闭环。
- 渠道接入页升级成真实连接器中心。
- 前端刷新按 `user.permissions` 分模块请求，避免低权限角色天然 403。
- 建立 owner/admin/agent/viewer 四类用户的运行时 contract smoke。
- 客户演示版去掉工程阶段标签、开发演示身份和规划态入口。

P2，暂不需要过度投入：

- 暂不做全渠道真实发送。
- 暂不做 Kubernetes、HPA、service mesh 之类重型高并发架构。
- 暂不做大而全 BI。
- 暂不追求重型 ANN/reranker 优化。
- 字段脱敏和 allowlist 是上线前必需，但本轮先不继续细化到极致。

## 下一步建议

建议下一阶段不要继续扩 UI 页面，而是先做一轮“契约收口 + 坐席闭环”：

1. 后端权限收口第一片：foundation/bootstrap 接口、workflow、inbound worker、reply orchestration。
2. 前端权限请求收口：按权限刷新数据，只请求当前角色能读的资源。
3. 对话台闭环第一片：AI 草稿可编辑、审批意见、引用确认、发送前确认入口在同一屏完成。
4. 渠道连接器中心第一片：先做企业微信/微信客服配置状态、Webhook URL 状态、Token/AESKey 引用状态和测试步骤。
5. 客户演示外观收口：隐藏 P3 阶段标签和开发演示文案，保留内部模式开关。

这一轮完成后，标准运营版才更接近“可以给真实客户试用的智能客服中台”。
