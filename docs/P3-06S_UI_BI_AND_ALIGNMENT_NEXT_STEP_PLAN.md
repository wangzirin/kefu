# P3-06S 客服中台下一步工程优化计划

日期：2026-07-01

本文件只规划下一步，不实施代码修改。目标是把当前智能客服中台从“模块已经不少”推进到“首页像真实运营中台、壳层稳定、数据口径可信、核心页面不再杂乱”的状态。

本轮只保留四个最大问题。其他问题先不展开，避免继续把工程切成过多零散任务。

执行状态补充：P3-06S-01 已完成窄桌面壳层滚动修复。此前 900px 宽度会触发移动布局并导致 body 整页滚动；本轮新增 761px 到 960px 的窄桌面壳层覆盖规则，900x768 已验证为 `bodyScrollY=0`、`sidebarTop=0`、`workspaceScrollTop=900`。390px 手机端仍保持自然滚动且无横向溢出。下一步优先进入 P3-06S-03 总览数据契约收口，再进入 P3-06S-02 运营总览 BI 重做。

## 结论

下一步不建议继续新增功能页，也不建议继续堆更多卡片。优先做四件事：

1. 修复中台壳层滚动：左侧导航必须固定，右侧工作区独立滚动。
2. 重做运营总览：首页要升级成高质感 BI 指挥舱，而不是普通信息卡片堆叠。
3. 收紧总览数据契约：前端展示的每个指标必须有清楚来源、口径、筛选和空态。
4. 精简页面信息架构：保留必要功能，减少长页面堆叠，让“看数据、找问题、处理问题”形成闭环。

## 最大问题 1：壳层滚动仍未达到真实中台标准

### 现象

用户在首页或长页面向下滚动时，左侧黑色导航栏仍会跟着页面整体滑动。对客服中台来说，这个体验是不合格的。标准中台应该是左侧导航固定，右侧白色工作区独立滚动。

### 工程判断

当前代码里已经有 `.app-shell`、`.sidebar`、`.workspace` 的独立滚动设计，但用户实际体验仍然不稳定，说明不能再只看 CSS 是否写了 `height: 100dvh` 或 `overflow: auto`。下一步必须按真实浏览器状态验收：

- body 是否仍然发生滚动。
- 左侧栏 `top` 是否在滚动后保持不变。
- 右侧 `.workspace` 的 `scrollTop` 是否变化。
- 桌面宽度、窄屏宽度和移动端是否进入了正确布局。

### 修复计划

任务编号：P3-06S-01

当前状态：已完成。阶段记录见 `docs/P3-06S-01_LAYOUT_BREAKPOINT_SCROLL_REPAIR.md`。

预计耗时：0.5 到 1 天。

涉及文件：

- `frontend/src/styles.css`
- `frontend/src/App.tsx`
- 可能新增 `docs/p3_06s_ui_qa/` 截图证据目录

施工步骤：

1. 重新定义桌面壳层边界：
   - `html/body/#root` 桌面端禁止页面整体滚动。
   - `.app-shell` 固定在视口高度。
   - `.sidebar` 固定高度并只滚动自身导航。
   - `.workspace` 成为唯一主滚动容器。
2. 重新检查移动断点：
   - 如果窗口宽度被浏览器侧栏压到移动断点，左侧栏不能误进入长页面滚动形态。
   - 移动端允许自然滚动，但要明确不是桌面中台形态。
3. 加一条浏览器滚动验收脚本或手动 QA 记录：
   - 滚动右侧 800px 后，`window.scrollY` 应保持 0。
   - `.sidebar.getBoundingClientRect().top` 应保持 0。
   - `.workspace.scrollTop` 应大于 0。

验收标准：

- 桌面 1440x900：左侧栏不动，右侧内容滚动。
- 桌面 1280x800：左侧栏不动，右侧内容滚动。
- 窄屏 1024x768：不得出现侧栏跟页面一起滚动。
- 移动 390x844：不横向溢出，导航不遮挡内容。

验证命令：

```bash
cd frontend && npm run typecheck
cd frontend && npm run build
```

人工验收：

- 打开 `#overview`。
- 鼠标滚轮向下滚动。
- 截图保存首屏和滚动后状态。

## 最大问题 2：运营总览不够像高质量 BI 指挥舱

### 现象

当前运营总览虽然已经有 ECharts 和一些指标，但视觉层级仍然偏普通：模块偏大、密度不均、经营判断不够强，一眼看过去不像成熟客服系统的“运营驾驶舱”。

首页应该回答四个问题：

- 今天客服压力高不高。
- AI 有没有帮上忙。
- 哪些渠道或会话正在出问题。
- 下一步运营人员应该先处理什么。

### 修复计划

任务编号：P3-06S-02

预计耗时：2 到 3 天。

涉及文件：

- `frontend/src/App.tsx`
- `frontend/src/styles.css`
- 建议新增 `frontend/src/components/ops/`
- 建议新增 `frontend/src/components/charts/`

施工步骤：

1. 重构首页首屏：
   - 顶部改成紧凑核心指标带趋势：进线量、AI 自动解决率、转人工率、平均响应、风险会话。
   - 中间放主视觉图表：会话趋势或运营健康度。
   - 右侧固定“今日优先处理”：高风险审核、待确认草稿、渠道异常、知识缺口。
2. 重做 BI 图表区域：
   - 会话处理漏斗。
   - 渠道进线与异常矩阵。
   - AI 命中率 / 转人工率趋势。
   - 错因或知识缺口 Pareto。
3. 降低无效大卡片：
   - 卡片不只展示数字，要同时展示趋势、状态和跳转动作。
   - 首页不展示长说明文字。
4. 建立视觉基准：
   - 借鉴 shadcn-admin / TailAdmin 的后台密度。
   - 借鉴 Ant Design Pro 的 Analysis / Monitor / Workplace 组织方式。
   - 借鉴 Chatwoot 的客服运营语义，而不是泛后台语义。

验收标准：

- 首屏无需下滑即可看到核心经营状态和下一步动作。
- 首页不是单纯卡片堆叠，至少有 3 类可读图表。
- 每个指标都能点击进入对应页面。
- 图表有空态、加载态、无权限态。
- 桌面截图有“数据中台”质感，移动端不拥挤、不横向滚动。

验证命令：

```bash
cd frontend && npm run typecheck
cd frontend && npm run build
```

人工验收：

- 截图对比当前版和新版本。
- 检查首屏信息密度、图表可读性、跳转动作。

## 最大问题 3：总览数据来源和前后端口径还不够硬

### 现象

当前首页会在“服务端聚合”和“本地样本”之间切换。这个机制适合开发预览，但对商用中台有风险：客户看到一个漂亮图表时，必须知道它是真实数据、演示数据、还是部分回退数据。

### 修复计划

任务编号：P3-06S-03

预计耗时：1.5 到 2 天。

涉及文件：

- `backend/app/api/ops.py`
- `backend/app/schemas/ops.py`
- `backend/tests/test_p3_06r_ops_dashboard_api.py`
- `frontend/src/api/client.ts`
- `frontend/src/App.tsx`

施工步骤：

1. 扩展运营总览 API 响应：
   - 返回 `data_mode`：真实聚合 / 演示样本 / 部分回退。
   - 返回 `range`、`channel_id`、`generated_at`、`source_window`。
   - 返回趋势 buckets、渠道 breakdown、质量错因、待处理动作队列。
2. 前端不再静默混用数据：
   - 服务端数据不可用时明确显示“演示样本”。
   - 渠道筛选或时间范围未被后端支持时，不悄悄显示本地数据当真实结果。
3. 增加契约测试：
   - owner/admin/viewer 可读。
   - agent 不可读。
   - 响应不泄露客户原文、AI 草稿正文、待发送正文。
   - range/channel 参数返回稳定结构。

验收标准：

- 首页每个关键指标都能追溯到后端字段。
- 演示数据和真实聚合数据有明确区分。
- 后端没有返回敏感正文。
- 前端不因为接口缺字段而显示误导性 0 或空白。

验证命令：

```bash
.venv/bin/python -m pytest backend/tests/test_p3_06r_ops_dashboard_api.py -q
cd frontend && npm run typecheck
cd frontend && npm run build
```

## 最大问题 4：页面信息架构仍偏“功能堆叠”，还不是高效运营路径

### 现象

现在页面很多，功能也不少，但用户进入中台后容易变成“看一堆页面”。成熟客服中台首页不应该只是功能入口，它应该形成运营路径：

1. 先看总体风险。
2. 点进异常来源。
3. 进入会话或质量复盘。
4. 修知识库、调规则、确认待发送。
5. 回到首页看到指标变化。

### 修复计划

任务编号：P3-06S-04

预计耗时：1 到 1.5 天。

涉及文件：

- `frontend/src/data/navigation.ts`
- `frontend/src/App.tsx`
- `frontend/src/styles.css`
- 可能新增 `docs/P3-06S_FUNCTION_NECESSITY_MATRIX.md`

施工步骤：

1. 做功能必要性矩阵：
   - 必须保留：首页、对话台、人工审核、待发送、质量复盘、知识库、渠道连接器、联系人/线索、系统运维。
   - 可以合并：重复的状态卡、重复的长说明、只读列表中的弱入口。
   - 暂缓展示：客户不需要理解的工程标签、内部阶段编号、过细的底层队列细节。
2. 重排首页和左侧导航：
   - 首页只放经营状态和行动入口。
   - 对话台承接实时处理。
   - 质量复盘承接错因和知识修复。
   - 渠道连接器承接平台配置和健康排障。
3. 减少长页面连续堆叠：
   - 关键区域用分区、标签或二级面板收纳。
   - 每个页面都要有明确的“下一步动作”。

验收标准：

- 左侧导航不再像工程模块清单，而像客服团队的工作流。
- 首页每个入口都能解释“为什么现在要点这里”。
- 不再出现大量客户看不懂的工程阶段词。
- 不删底层能力，但把底层能力移到合适的管理或诊断区。

## 推荐执行顺序

1. P3-06S-01：先修壳层滚动。这个是基础体验问题，必须最先处理。
2. P3-06S-03：同步收紧数据契约。否则 BI 做漂亮了，也可能展示不可信。
3. P3-06S-02：重做运营总览。用已经明确的数据契约支撑视觉升级。
4. P3-06S-04：最后做信息架构收口。避免在旧页面结构上继续堆样式。
