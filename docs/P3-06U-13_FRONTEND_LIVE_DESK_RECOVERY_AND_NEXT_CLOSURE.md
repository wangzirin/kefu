# P3-06U-13 前端接待台回归修复与下一步收口

日期：2026-07-02

## 1. 本轮判断

前端没有达到“完全完善”。当前可以演示，也具备总览、接待台、质量、知识、渠道、运维等核心工作区，但仍处在产品化收口阶段。

本轮复核发现：

- `npm run typecheck` 通过。
- `npm run build` 通过，但 Vite 仍提示主 chunk 偏大，说明前端仍需要继续拆组件。
- 运营总览浏览器 smoke 通过。
- 多渠道对话台浏览器 smoke 一开始失败：
  - 缺少 `service-inline-context` 轻量上下文条。
  - 下一步动作区没有被识别为紧凑条。
  - P3-06U-11C 静态检查仍按旧路由判断，未兼容 `#rpa-lab`。

这说明前端不是崩坏状态，但“微信式接待台”的关键验收曾经发生回退，需要立即修复。

## 2. 已完成修复

### 2.1 对话台补轻量上下文条

在当前会话消息头下方补入四格上下文：

| 格子 | 用途 |
| --- | --- |
| 客户 | 显示当前客户和咨询主题。 |
| AI 判断 | 显示回复策略模式和下一步动作。 |
| 知识依据 | 显示是否缺口、命中数量，并给出知识入口。 |
| 待发送 | 显示当前草稿状态和外发状态。 |

这样坐席不用在大判断区里找基本信息，界面更接近真实客服工作台。

### 2.2 坐席判断区压缩为可验收的动作条

将坐席判断主区标记为 `service-next-action`，让它被浏览器 smoke 识别为紧凑动作条。当前动作区保留：

- 试算当前会话。
- 带入回复框。
- 领取 / 等客户 / 解决。

这些仍然只是内部坐席动作，不触发真实平台发送。

### 2.3 修正静态验收

P3-06U-11C 的静态检查已兼容 `#rpa-lab`：RPA Lab 不再被误判为必须展示顶部任务路径或运行状态条。

## 5. 下一步收口路线

### P3-06U-14 页面级组件拆分

目标：降低 `App.tsx` 体积，先拆知识运营、质量复盘、渠道接入或运维页面之一。

验收：

- `App.tsx` 明显减少。
- 页面行为不变。
- 原有 smoke 继续通过。

### P3-06U-15 接待台第二轮视觉和信息密度收口

目标：继续压缩坐席判断区，让右侧消息流更像真实 IM，对话消息占据更高比例。

验收：

- 1440 桌面下消息流高度不低于当前。
- 坐席首屏可见客户列表、上下文条、消息流和回复框。
- 不新增管理看板式模块。

### P3-06U-16 真实登录端到端复验

目标：用真实本地测试账号复验 owner/admin/agent/viewer 的默认入口、权限、禁用原因和关键动作。

验收：

- 真实 token 登录。
- 不依赖演示模式。
- 低权限角色不看到无意义按钮墙。

## 6. 边界

本轮不打开真实外发，不接真实抖音、淘宝、京东、拼多多后台，不新增平台自动发送能力。

RPA 研究线继续保持 draft-only；正式商用渠道仍以官方 API、服务商授权和白名单验收为主。
