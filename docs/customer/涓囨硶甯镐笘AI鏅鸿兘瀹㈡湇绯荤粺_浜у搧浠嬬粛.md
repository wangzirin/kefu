# 万法常世 AI 全智能客服系统产品介绍

版本：初版  
适用对象：企业负责人、业务负责人、客服负责人、运营团队、数字化负责人

## 1. 产品定位

万法常世 AI 全智能客服系统是一套面向企业的知识库型智能客服中台。

它不是简单的固定问答，也不是不受控的自动聊天机器人。系统的核心目标是：

- 让常见问题更快得到一致回复。
- 让复杂问题有证据、有人工审核、有审计记录。
- 让企业知识库持续沉淀和更新。
- 让客服团队看到低置信、知识缺口、发送失败和高风险会话。
- 让智能客服从“能回答”升级为“可运营、可复盘、可维护”。

## 2. 适用场景

| 场景 | 价值 |
| --- | --- |
| 官网咨询 | 自动生成基于知识库的回复草稿，提高接待效率 |
| 企业微信/私域客服 | 统一产品、价格、售后和交付口径 |
| 售前咨询 | 快速回答套餐、价格、案例、交付周期 |
| 售后咨询 | 根据售后政策生成可审核回复 |
| 招商加盟 | 收集线索，识别高意向问题 |
| 多坐席团队 | 统一知识、统一审计、统一质量复盘 |
| 知识库运营 | 发现常问但无答案的问题，持续补齐知识 |

## 3. 核心能力

### 3.1 自建知识库

系统支持将产品资料、套餐价格、售后政策、交付流程、合同说明、禁用话术等内容整理为知识库。

知识库不是一次性文件夹，而是可运营资产：

- 支持知识卡片和文档知识。
- 支持分块检索和引用来源。
- 支持题库回归和质量报告。
- 支持知识缺口复盘。
- 支持版本和发布流程逐步完善。

### 3.2 AI 回复草稿

AI 根据用户问题和知识库证据生成回复草稿。草稿进入工作台后，坐席可以看到：

- 用户原始问题。
- AI 回复草稿。
- 引用证据。
- 风险等级。
- 转人工原因。
- 模型状态。
- 审计链路。

### 3.3 人工审核与外发门禁

系统把“AI 生成草稿”和“真正发送给用户”分成两个动作。

这样可以避免：

- AI 在证据不足时直接回答。
- 高风险问题被自动承诺。
- 平台外发误操作。
- 后续无法追溯责任。

### 3.4 坐席工作台

工作台重点展示真正需要处理的任务：

- 待人工审核。
- 待发送确认。
- 发送失败。
- 知识缺口。
- 高风险会话。
- 最近评测运行。
- 渠道健康。

### 3.5 模型路由

不同问题使用不同处理策略：

| 问题类型 | 处理方式 |
| --- | --- |
| 问候、导航、简单 FAQ | 低成本快速模型或规则回复 |
| 普通业务咨询 | 标准模型 + 知识库引用 |
| 退款、赔付、合同、投诉 | 高风险标记 + 人工审核 |
| 证据不足的问题 | 转人工 + 进入知识缺口 |

### 3.6 质量监控

系统通过题库和线上反馈持续监控质量：

- 知识命中率。
- 引用支撑率。
- 转人工率。
- 低置信率。
- 坐席改写率。
- 知识缺口数。
- 渠道失败率。
- 负反馈和投诉。

## 4. 产品版本

### 4.1 Lite 试点版

适合希望先低成本验证智能客服价值的企业。

包含：

- 单入口试点，优先官网或自有入口。
- 基础账号和坐席工作台。
- FAQ 和轻量文档知识库。
- AI 草稿和人工审核。
- 留资和线索记录。
- 50 条起步的脱敏题库。
- 基础质量报告。
- 轻量部署和远程维护支持。

适合：

- 小团队。
- 单业务线。
- 官网或私域试点。
- 先验证效果再扩大投入。

### 4.2 标准运营版

适合计划将智能客服纳入日常运营的企业。

包含：

- 正式账号、角色、团队和权限。
- 坐席工作台和会话证据详情。
- 知识卡片、文档知识、质量评测和知识缺口闭环。
- 模型路由、fallback 和人工审核门禁。
- 渠道接入沙盒和官方渠道逐步接入。
- Outbox、发送确认、失败复盘和审计。
- 部署、备份、恢复、回滚和诊断包。
- 月度质量复盘和运维服务。

适合：

- 多坐席团队。
- 需要长期维护知识库。
- 需要审计和质量复盘。
- 准备逐步接入多个官方渠道。

## 5. 为什么不是普通 FAQ

普通 FAQ 只能回答固定问题。万法常世 AI 全智能客服系统的不同点是：

- 能结合多个知识片段生成草稿。
- 能显示引用证据。
- 能判断低置信和高风险。
- 能把问题进入人工审核。
- 能把失败和知识缺口变成运营任务。
- 能用题库回归判断知识库是否退化。
- 能形成长期可维护的客服中台。

## 6. 安全边界

系统默认遵循以下原则：

- 真实外发默认关闭。
- 高风险问题默认转人工。
- 未授权平台不接入。
- 不使用个人微信外挂、Hook、群控或模拟点击作为正式方案。
- 客户真实数据进入模型前必须脱敏和授权。
- 关键操作保留审计记录。

## 7. 交付流程

| 阶段 | 内容 |
| --- | --- |
| 需求确认 | 明确入口、业务范围、知识资料、风险边界 |
| 知识整理 | 整理产品、价格、售后、禁用话术和题库 |
| 系统部署 | 部署后端、前端、数据库、队列和配置 |
| 试点接入 | 优先接一个官网或官方授权入口 |
| 质量回归 | 使用脱敏题库验证知识命中和草稿质量 |
| 坐席培训 | 讲解审核、改写、知识缺口和复盘流程 |
| 试运营 | 保持人工审核，收集真实问题和反馈 |
| 月度复盘 | 输出质量报告，更新知识库和策略 |

## 8. 适合的采购方式

建议先从 Lite 试点版开始，验证以下问题：

- 常见问题是否能稳定命中知识库。
- 坐席是否愿意使用 AI 草稿。
- 知识缺口是否可被持续补齐。
- 渠道接入和外发策略是否安全。
- 运营团队是否能维护质量。

当试点稳定后，再升级到标准运营版，增加多坐席、更多渠道、质量中心、SLA、工单和更完整的运维服务。
