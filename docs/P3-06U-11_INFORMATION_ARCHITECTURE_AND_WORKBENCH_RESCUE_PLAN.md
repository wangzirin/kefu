## 发现的四个最大问题

### 1. 真实登录态、路由和数据刷新还不够稳定

已确认真实账号可以创建、登录并访问 `/api/auth/me`。但前端存在两个体验问题：

- 登录态初始化后，直接刷新二级路由时可能回到总览。
- 新增会话数据后，单纯切换 hash 不会自动重新拉取会话，需要刷新或点击刷新。

这会影响真实客服使用：平台 webhook 进来新消息后，坐席工作台必须自然出现新会话，不能依赖操作者猜测刷新。

### 2. 多渠道对话台是核心页面，但还不像真正的坐席主界面

有数据时，对话台已有“左侧会话列表 + 右侧会话详情”的基础。但当前仍被顶部任务路径、状态条、说明文案和多个统计卡占据太多空间。坐席打开工作台时，第一眼应该看到：

- 左侧：客户列表、渠道标识、最后一句、等待时长、待审/高风险标签。
- 右侧：像微信一样的对话流、AI 草稿、内部备注、领取/解决/等客户/补知识动作。
- 辅助信息：证据、知识引用、待发送状态放到右侧上下文卡或抽屉，不抢主视觉。

### 3. 工作台下的多个页面存在职责重叠

当前左侧工作台包含：

- 多渠道对话台
- 会话收件箱
- 人工审核
- 待发送草稿
- 工单/SLA

其中“会话收件箱”与“多渠道对话台”重复度最高。建议会话收件箱不再作为平级主页面，而是并入多渠道对话台的筛选队列。

“人工审核”和“待发送草稿”不应完全删除，但不宜默认暴露成普通坐席的主导航。它们更适合成为：

- 对话台当前会话内的动作区。
- 管理员或质检角色的二级队列。
- 真实外发打开前的安全门禁页面。

“工单/SLA”是否保留取决于套餐和客户是否真的需要售后工单。入门版可以先降级为会话中的标签和提醒；标准运营版再保留独立工单页。

### 4. 管理运维和知识运营存在“双层导航”

左侧已经有“运维与告警、模型路由、系统设置”，右侧又渲染一组相同的切换卡，造成重复。

知识运营也有相似问题：知识库运营、知识缺口、知识评测在左侧是三页，但右侧结构和顶部说明很像，用户会感觉没有真正分叉。

建议：

- 管理运维：左侧子项已经足够，右侧不要再出现重复切换卡。
- 知识库运营：聚焦文档、分块、检索、发布。
- 知识缺口：聚焦错因来源、人审失败、评测失败、补知识任务。
- 知识评测：聚焦题集、运行、命中率、回归趋势。

## 是否适合一次性大改

不建议一次性改完。原因：

1. 本轮已发现真实登录、路由刷新、数据刷新和 UI 信息架构同时存在问题。
2. 对话台、审核、outbox、工单之间有后端状态联动，直接删页面容易丢失安全门禁。
3. 总览、知识、运维页面都有组件复用和状态依赖，需要分批验证截图和 API。
4. 当前已经开始有真实账号与样例数据，后续每一批改动都应该用真实登录态验证，而不是只看 demo。

## 推荐施工顺序

### P3-06U-11A：真实本地开发库与登录稳定化

目标：

- 固化本地 SQLite dev 库启动方式。

- seed 后会话总数与前端列表一致。
- 不输出密码、token 或密钥。

### P3-06U-11B：工作台导航收束

目标：

- 工作台默认只保留“多渠道对话台”作为主入口。
- 会话收件箱并入对话台左侧筛选。
- 人工审核、待发送草稿、工单/SLA 降级为对话台内队列、详情抽屉或管理员二级入口。

验收：

- 普通坐席打开工作台不会看到五个重复入口。
- 管理员仍能找到审核、outbox 和工单，不丢失安全门禁。

### P3-06U-11C：微信式多渠道对话台

目标：

- 页面主体改为两栏：左侧会话列表，右侧会话流。
- 去掉顶部冗余任务路径和过多说明。
- 当前会话展示客户消息、AI 草稿、内部备注、知识引用、领取/等客户/解决/补知识动作。
- 保留“真实外发关闭”提示，但不让它占据主操作区。

验收：

- 1440 宽屏首屏能看到完整会话列表和右侧会话内容。
- 坐席无需滚动就能看到客户最后一句、AI 草稿和主要动作。
- 移动端不重叠。

### P3-06U-11D：运营总览 BI 收束

目标：

- 首页首屏只保留关键指标、压力趋势、优先动作。
- 会话处置详情、渠道健康、质量诊断等底部明细改成二级展开或跳转。
- 让首页更像经营看板，而不是一页塞满所有功能。

验收：

- 首屏冲击力提升。
- 风险区域不被遮挡。
- 总览卡片点击能落到正确页面或筛选状态。

### P3-06U-11E：知识与运维真实分叉

目标：

- 知识库运营、知识缺口、知识评测使用不同信息结构。
- 运维与告警、模型路由、系统设置去掉重复内部 tab 卡。

验收：

- 三个知识页从标题、指标、列表、主动作上能一眼区分。
- 三个运维页不再出现左右重复导航。

## 2026-07-02 补充：AI+RPA 研究线进入回复策略验证

用户最新要求是：暂时不把官方 API 作为唯一技术核心，先研究 AI+RPA 闭环能不能作为客服系统的一条可行技术路线。

因此，P3-06U-11 后续推进需要加入一条并行研究线：

```text
P3-06U-12：AI+RPA 回复策略与中台融合
```

这条线不改变当前中台的核心目标：仍然是面向中小企业的一体化客服中台，不先做复杂普通坐席/管理员拆分；也不把研究能力包装成正式平台授权能力。

新的判断：

- 多渠道对话台仍是中台主界面。
- AI+RPA 先作为“副驾驶/填框/研究适配器”，不是正式外发渠道。
- 官方 API / Webhook 线继续保留为正式交付候选。
- 回复策略要统一：无论消息来自官方 webhook、人工粘贴、RPA 观察还是未来渠道连接器，都先进入同一套 `intent / answer_mode / delivery_mode / next_best_action` 决策。
- 质量复盘、知识缺口和 BI 不只看“答得好不好”，还要看策略是否正确：该转人工有没有转、该补知识有没有补、该填草稿有没有填、有没有出现不该承诺的内容。
