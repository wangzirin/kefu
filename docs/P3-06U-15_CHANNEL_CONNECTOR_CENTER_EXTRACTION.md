# P3-06U-15 渠道连接器中心组件拆分

日期：2026-07-02

## 1. 工程控制卡

| 项目 | 内容 |
| --- | --- |
| 本片目标 | 把渠道连接器中心从 `App.tsx` 抽离到独立组件，降低主应用文件复杂度。 |
| 客户可见价值 | 页面行为保持不变，但后续补企业微信、公众号、电商渠道的真实配置状态时，可以在渠道模块内迭代，不继续挤压主壳层。 |
| 是否只是评测 | 否。本片是前端结构性改造，同时补静态门禁。 |
| 停止条件 | P3-06U-15 静态门禁、P3-06U-08 旧渠道门禁、TypeScript 和生产构建通过；不打开真实外发。 |
| 写回位置 | 本文档、`scripts/check_p3_06u_15_channel_connector_extraction.py`、Project_012 执行记录和文件索引。 |

## 2. 本轮改动

### 2.1 新增渠道组件目录

新增：

`frontend/src/components/channels/ChannelConnectorCenterPanel.tsx`

承载原 `#channels` 第一屏的渠道接入状态中心：

- 企业微信 / 微信客服主通道 10 步接入状态。
- 公网 HTTPS 回调 URL、Token、EncodingAESKey、可信 IP 的配置状态。
- 正式商用前置条件。
- 公众号、抖音 / 抖店、小红书、淘宝 / 天猫、京东 / 拼多多的官方授权前置条件。
- 不使用个人号外挂、Hook、模拟点击或商家后台群控的边界提示。

### 2.2 主应用文件减负

`App.tsx` 不再定义 `ChannelConnectorCenterPanel`、`buildChannelConnectorCards` 和渠道中心专用类型，只负责把现有的人审、待发送、失败复盘、发送队列、worker 运行记录、登录状态和权限传入组件。

这次抽离后，`App.tsx` 行数从约 11171 行降到约 10764 行。

### 2.3 行为边界

本片坚持“行为不变，结构变清楚”：

- 没有新增后端接口。
- 没有新增真实渠道配置表。
- 没有读取或展示真实 Token、EncodingAESKey、Secret 明文。
- 没有新增真实外发。
- 没有改变 P3-06U-08 渠道连接器中心的展示口径。
- 企业微信、公众号、电商渠道仍只走官方 API、开放平台或服务商授权。

## 3. 验收方式

| 验收项 | 方式 |
| --- | --- |
| 组件抽离 | `App.tsx` 只 import，不再定义 `ChannelConnectorCenterPanel`。 |
| 渠道中心关键结构 | 静态门禁检查 `data-channel-connector-smoke`、企业微信主卡、配置卡和官方授权卡。 |
| 旧行为回归 | P3-06U-08 旧渠道中心静态门禁继续通过。 |
| 类型契约 | `npm run typecheck`。 |
| 生产构建 | `npm run build`。 |
| 回归边界 | 不改真实外发开关，不接真实企业微信、公众号或电商发送。 |

## 4. 下一步

建议继续做 P3-06U-16：

优先拆知识运营页，或先把渠道中心进一步接入后端只读配置状态。若进入真实平台联调，必须先具备公网 HTTPS 回调、平台后台权限、Token / EncodingAESKey / Secret、测试白名单和明确授权；在这些条件齐备前，渠道页只表达状态、阻塞点和下一步配置动作。
