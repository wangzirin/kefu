# P3-06U-26B 多渠道对话台微信式收束

日期：2026-07-02  
定位：P3-06U-26 工程优化总纲的第二个施工片  
范围：只优化多渠道对话台首屏信息架构和前端布局，不接真实平台账号，不打开真实外发

## 目标

多渠道对话台要更像真实客服每天使用的即时会话工作台，而不是工程状态说明页。

这一片的目标不是增加功能数量，而是把首屏主次关系重新摆正：

```text
左侧会话列表 -> 右侧真实消息流 -> AI 自动回复记录 -> 例外转人工提醒
```

## 本轮改动

### 1. 左侧会话列表继续瘦身

- 主布局已收紧到 `minmax(176px, 204px)`。
- 会话列表继续只承担选择会话、查看平台标识、最后消息、等待时间和异常标签。
- 主筛选只保留“全部 / 我的 / 转人工”。
- 待发送、审核、工单等保留为后台能力，不作为坐席主入口。

### 2. 右侧首屏优先消息流

- 右侧行模型收束为：会话头部、消息流、回复区。
- 平台来源、店铺/入口和坐席状态进入会话头部。
- 旧的上下文条、AI 建议面板、待发控制区不再默认占据右侧首屏。

### 3. AI 自动回复记录靠近输入区

- 普通会话展示“AI 自动回复 / 最近自动回复”，人工无需操作。
- 异常会话展示“转人工提醒 / 人工接管回复”。
- 不再把普通高置信会话设计成必须人工审核或确认发送。

### 4. 转人工保持异常路径

- 低置信、高风险、无知识、超时、渠道异常或发送异常时才进入转人工。
- 转人工是坐席主工作台里的提醒和接管输入，不再叫“待审”。
- 普通高置信会话不再被界面暗示“每条都要写审核意见”或“每条都要确认待发送”。

## 不变边界

- 真实外发继续关闭。
- 不接真实微信、企微、抖音、淘宝、京东、拼多多账号。
- 不点击任何真实发送按钮。
- RPA 仍是研究线和 draft-only 副驾驶，不进入正式默认交付链。
- 本轮不改变后端回复策略状态机、outbox、worker 或权限模型。

## 完成标准

- 桌面端左侧会话列表宽度不超过 218px。
- 右侧聊天区宽度明显大于左侧会话列表。
- 首屏存在消息流、AI 自动回复记录或转人工提醒。
- 队列只暴露“全部 / 我的 / 转人工”。
- 不出现“待审、待发、AI 建议、客户可见回复预案、人工接管 / 异常备注、内部备注、确认发送队列”等旧前台文案。
- 真实外发关闭仍可见。

## 停止条件

- 消息流仍被多个状态卡压到首屏之外。
- 普通高置信会话仍被设计成必须人工写审核意见或确认待发送。
- 真实外发关闭提示被隐藏。
- 页面出现横向溢出或运行时错误。
