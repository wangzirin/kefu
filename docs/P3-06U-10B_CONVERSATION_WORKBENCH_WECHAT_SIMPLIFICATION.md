# P3-06U-10B 多渠道对话台微信式收束

日期：2026-07-02  
范围：`#live` 多渠道对话台  
阶段定位：P3-06U-10 第二片，优先处理坐席主工作流的实用性和信息密度

## 1. 本轮判断

原多渠道对话台已经具备会话列表、AI 回复记录、人审、待发送、知识引用和渠道门禁，但界面承载方式偏“中控台三栏”：顶部有信号条和大队列卡，右侧还有完整上下文侧栏。对坐席来说，核心任务其实更简单：

1. 左侧找到需要处理的客户。
2. 右侧看清客户问题、平台来源和 AI 自动回复记录。
3. 普通问题由 AI 自动回复，人工不逐条审核。
4. 只有低置信、无知识、高风险、超时或渠道异常时进入转人工。

因此本轮不新增后端能力，也不打开真实外发，而是把页面从“三栏管理看板”收束为“微信式双栏接待台”。

## 2. 已做调整

| 区域 | 调整前 | 调整后 |
| --- | --- | --- |
| 顶部 | 标题、数据来源、真实外发、4 个信号卡、8 个队列卡 | 只保留紧凑标题、数据来源、外发关闭和 3 个小型摘要 |
| 队列筛选 | 8 个大卡片横向铺开 | 收进左侧会话列表，只保留“全部 / 我的 / 转人工” |
| 主体布局 | 左队列、中聊天、右上下文三栏 | 左侧会话列表、右侧大对话框双栏 |
| 右侧上下文 | 常驻大侧栏，占用主处理区 | 只保留会话头部、消息流、自动回复记录和转人工接管区 |
| 坐席动作 | 部分动作分散在中栏和右栏 | 普通会话只看自动回复状态；异常会话在同一对话框里接管 |
| 管理信息 | 质量、知识、渠道、待发送信息混在当前页 | 只保留必要入口，深入处理跳到质量、知识、渠道页面 |

## 3. 不做的事

- 不接入真实外部渠道发送。
- 不绕过官方平台授权。
- 不把演示样本包装成真实客户数据。
- 不删除人审、待发送、知识引用和渠道门禁后端能力，但它们不再作为普通坐席的主前台路径。
- 不把质量 BI、知识运营、渠道配置重新塞回接待页。
- 不做移动端适配或移动端 smoke，当前中台只按桌面端工作台推进。

## 4. 验收标准

- `#live` 首屏不再出现独立的大型 `service-desk-signal-strip`。
- `service-desk-layout` 桌面端为两列：会话列表 + 对话框。
- 队列筛选位于左侧会话列表，不再铺满顶部。
- 右侧独立上下文栏不再作为第三列常驻。
- 聊天区仍能看到客户消息、AI 自动回复记录、知识依据和转人工提醒。
- “真实外发关闭”边界仍保留。
- 1440 和 1280 桌面视口无横向溢出。

## 5. 后续建议

下一步可以继续拆 `ConversationWorkbenchPanel.tsx`，把会话列表、聊天区、草稿审核区拆成独立组件；但这一步应在当前双栏交互验证稳定后再做，避免边拆边改导致页面逻辑不稳。
