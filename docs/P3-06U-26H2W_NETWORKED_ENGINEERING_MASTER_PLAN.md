# P3-06U-26H2W 网状工程推进总纲

日期：2026-07-04

## 1. 文档定位

本文件接续 `P3-06U-26H2W_NEXT_STAGE_PHASED_ENGINEERING_PLAN_AFTER_H2W7A.md`，用于把后续工程从线性阶段推进改为网状推进。

它不是客户宣传材料，不是能力完成清单，也不替代每个阶段的阶段卡。它的作用是给后续施工提供统一判断：任何新功能都必须同时回答“前端是否真实、后端是否落账、知识是否可追溯、模型是否有成本、渠道是否合规、本地运维是否可执行、试点是否可验收”。

## 2. 当前固定边界

| 事项 | 当前口径 |
|---|---|
| 产品形态 | 桌面端智能客服中台，不做移动端主线 |
| 真实外发 | 默认关闭，真实外发继续关闭；任何真实平台发送必须另行授权 |
| RPA | 只作为内部 draft-only 研究线，不进入正式默认交付 |
| 官方渠道 | 先做授权矩阵、配置 schema、回执模板和沙箱前置条件，不承诺全平台已接通 |
| 知识评测 | 当前主要是检索、引用、人工标签和报告闭环，不等于完整线上客服准确率 |
| 本地运维 | 已有诊断、更新、备份、恢复 dry-run 和处理单骨架，仍不是无人值守自动更新器 |
| Git 状态 | `standard_ops` 已初始化 Git，但当前没有基线提交，所有工程变更必须保持小步、可审查、可验证 |

## 3. 网状主链路

```text
渠道账号/店铺
  -> 入站消息
  -> 统一消息信封
  -> 回复决策
  -> 知识命中/无引用原因
  -> 模型调用/成本记录
  -> 草稿或转人工
  -> Outbox/外发门禁
  -> 回执/失败复盘
  -> 质量报告
  -> 知识修复
  -> 回归验证
  -> 客户签收/本地更新包
```

这条链路上的每个节点都必须能被同一个 `provenance_id` 或等价事实键串起来。否则后续成本、引用、质量报告、失败复盘和客户签收会各自孤立，不能作为商用试点证据。

## 4. 七条并行校验轴

每个阶段完成前，都必须按以下七条轴校验。

| 校验轴 | 必须回答的问题 | 停止条件 |
|---|---|---|
| 客户日常路径 | 客户能不能登录、接待、维护知识、看质量、生成诊断包 | 页面只展示模块，不知道下一步做什么 |
| 后端事实账 | 是否有表、接口、审计、幂等和数据落盘 | 只有 payload 或前端状态，没有持久事实 |
| 模型/RAG | 是否记录模型、引用、成本、降级和无引用原因 | 无引用高置信、失败伪成功、成本不可追 |
| 渠道/outbox | 是否区分入站、草稿、外发、回执、失败复盘 | 未授权渠道写成已接通或已自动回复 |
| 本地运维 | 是否支持诊断、备份、更新、回滚、客户确认 | 绕过管理员确认、无备份、无回滚 |
| 前端真实性 | 控件是否有真实动作、禁用说明或隐藏 | 假按钮、假筛选、假分页、假发送 |
| 验证证据 | 是否有测试、构建、浏览器 smoke、文档和写回 | 只靠截图、口头说明或 demo 数据封版 |

## 5. 阶段顺序

### H2W-N0：系统事实矩阵

目标：把当前所有核心页面、接口、表、服务、测试、文案的真实状态放进同一张矩阵。

施工内容：

- 复核 `FRONTEND_FUNCTION_REALITY_MATRIX.md`。
- 标注每个能力为：真实可用、只读摘要、演示数据、dry-run、draft-only、未接通。
- 清理客户可见越界文案，例如“已自动回复、已发送、已修复、全渠道接通”。
- 形成后续阶段统一门禁。

停止门禁：

- UI 文案超过后端真实能力。
- 客户可见控件没有真实动作、禁用说明或隐藏策略。
- 只更新文档，不更新门禁脚本或验证证据。

### H2W-7X：回复事实账本

目标：先补统一事实键，避免 H2W-7B、7C、7D 各自落账。

施工内容：

- 为 `reply_decisions` 增加 `provenance_id`。
- 新增 `model_call_records`，记录 provider、model、route、复杂度、状态、token/字符、耗时、估算成本、预算策略和降级动作。
- 新增 `reply_citation_snapshots`，记录知识卡、对象问答卡、文档 chunk、版本、hash、来源 URI、分数和无引用原因。
- 让 RAG/成本治理摘要读取真实 `model_call_records`，不再从 `decision_payload` 猜测模型调用。

停止门禁：

- 成本、引用、质量报告不能追到同一条回复决策。
- 高置信回复没有引用或无引用原因。
- 模型调用记录没有 provider、model、route、状态和成本来源。

### H2W-7B：模型调用成本台账与预算门禁

目标：把模型调用从“可见缺口”升级为真实成本台账和预算降级。

当前状态：第一片已完成。后端已新增预算配置、成本估算服务、预算门禁、`budget_blocked`/`degraded` 状态和成本台账写入；显式 provider 超预算不静默 fallback，自动路由超预算降级为确定性知识草稿并进入人工复核。被拦截或降级的外部模型预计费用只进入 `budget_policy_snapshot`，不计入真实累计成本。

施工内容：

- 在模型网关调用结果写入 `model_call_records`。
- 支持成功、失败、unavailable、budget_blocked、degraded 等状态。
- 预算超限时按高阶模型、标准模型、确定性知识草稿、转人工顺序降级。
- 前端模型路由页展示真实台账摘要，不把空记录写成已治理。

停止门禁：

- 没有持久成本表却宣称成本治理完成。
- 显式 provider 失败被静默 fallback。
- 预算不足仍返回伪成功回答。

### H2W-7C：引用溯源与答案质量

目标：最终回复能追到知识来源；无引用要进入谨慎回复或转人工。

当前状态：第一片已完成。最终答案采样已经同步写入 `reply_citation_snapshots`，有引用 URI 时生成 `final_answer_citation_uri`，无引用时生成 `final_answer_no_citation` 并记录 `final_answer_sample_without_citation`。RAG 治理摘要新增 `answer_quality` 和 `final_answer_quality` 门禁，用来区分检索评测和最终客服答案质量；当前仍不是完整线上准确率。

施工内容：

- 将回复决策、最终回复样本和评测用例绑定引用快照。
- 无引用必须写 `no_citation_reason`。
- 质量评测按事实正确、引用充分、禁用承诺、转人工正确统计。

停止门禁：

- 只评检索命中，不评最终答案。
- 引用只有文件名，没有 chunk/hash/version。
- 无引用答案被标为高置信可自动回复。

### H2W-8A/8B：本地部署和维护闭环

目标：客户本地部署后能完成首任负责人、账号、诊断、备份、更新和回滚演练。

施工内容：

- 8A：空库首次启动、负责人创建、登录、外发关闭、开发 bootstrap 关闭。
- 8B：诊断包、售后接收台、处理单、签名包、备份、应用、回滚、审计。
- 前端把客户可见入口收束为“账号与本地维护”，弱化工程词。

停止门禁：

- 客户不知道管理员账号怎么创建。
- 更新绕过客户管理员确认。
- 诊断包泄露 secrets、原始完整聊天或完整回复。

### H2W-FE：前端真实工作流专项

目标：把前端从“功能展示页”继续收束为真实客服工作台。

施工内容：

- 多渠道对话台继续微信式收束：左侧紧凑会话列表，右侧大面积消息流。
- 去掉“AI 自主回复预备、待审核、待发送”等普通客户首屏不需要的表达。
- 知识运营统一为导入、预检、发布、回归、报告、签收。
- 每个按钮都进入真实性矩阵。

停止门禁：

- 按钮只有图标没有功能。
- 左侧导航和右侧页面是假分页。
- 客户看不懂下一步。

### H2W-10：渠道官方授权矩阵与配置模板

目标：在不打开真实外发的前提下，把官方渠道接入条件、配置 schema 和失败码模板准备好。

施工内容：

- 企业微信、公众号、抖音、淘宝、京东、拼多多分别列官方授权条件。
- 配置低敏账号状态、secret 状态和失败码映射。
- 后端保留外发 kill switch。
- RPA 继续只保留 draft-only 内部研究归档。

停止门禁：

- 未授权渠道显示“已接通”。
- RPA 被包装成正式商用默认能力。
- 无验签、解密、幂等、回执、审计就尝试外发。

### H2W-11：受控试点 rehearsal

目标：用 50-100 条脱敏真实题库跑完整闭环，不用 demo 数据封版。

施工内容：

- 导入真实脱敏题库和客户知识包。
- 跑最终回复样本、人工事实标签、引用覆盖、转人工正确性。
- 输出客户质量报告、签收记录和本地更新包演练。

停止门禁：

- 只用 demo 数据封版。
- 只看检索命中，不看最终答案。
- 没有备份和回滚就执行更新。
