# P3-06U-26H2W 推荐版工程门禁计划

日期：2026-07-03

## 1. 文档定位

本文件是对 `P3-06U-26H2W 工程完善推进与客户知识建设中心计划` 的推荐版重整。

它不是客户宣传文案，也不是阶段汇报。它的用途是作为后续 Codex 目标模式的工程目标文档：每一片都必须有清楚的施工范围、验收门禁、停止门禁和证据要求。

本文件保留原 H2W 的方向：暂缓真实 IM 外发，把精力放在中小企业本地化客服中台、客户知识建设、质量复盘、报告签收、诊断包、更新恢复和成本治理上。

同时新增一条硬规则：所有客户可见前端必须先通过“功能真实性门禁”。凡是只有图标、按钮、文案，但没有真实动作、真实接口、真实数据变化或明确禁用理由的内容，不能进入下一阶段。

## 2. 建议目标模式

如果把本文件设为 Codex 目标，不建议一次性执行完整 H2W。推荐拆成连续目标：

| 目标 | 建议目标名 | 成功率 | 原因 |
|---|---|---:|---|
| 第一目标 | 完成 H2W-0 前端功能真实性与契约门禁 | 80%-88% | 范围清楚，能先消灭假按钮、假流程和越界文案 |
| 第二目标 | 完成 H2W-1 到 H2W-2，对话台与知识建设中心成型 | 72%-82% | 直接影响客户使用，仍可控制范围 |
| 第三目标 | 完成 H2W-3 到 H2W-4，准确率闭环与报告签收 | 65%-75% | 涉及数据口径和导出格式，需更多测试 |
| 第四目标 | 完成 H2W-5 到 H2W-6，云接收台与更新恢复 | 55%-68% | 涉及跨环境、文件包、版本和恢复风险 |
| 第五目标 | 完成 H2W-7，生产级 RAG 与模型成本治理 | 55%-70% | 涉及模型调用、检索质量、成本策略和持续评测 |

不建议把 H2W-1 到 H2W-7 一次性作为一个目标。范围过大时，最容易出现“页面看起来完成，但后端没有闭环”的问题。

## 3. 当前真实边界

必须继续承认以下事实：

1. 真实外发继续关闭。
2. 企业微信、公众号、抖音、淘宝、京东、拼多多等真实平台官方接入验收暂缓。
3. RPA 只作为内部研究线，不进入正式默认交付。
4. 当前知识评测不是完整线上客服准确率，只能作为发布前回归、检索质量和人工标签质量复盘的一部分。
5. 当前客户签收不是正式电子签章。
6. 当前程序更新仍是 dry-run 和签名包演练，不是自动替换运行程序。
7. 当前云接收台还不是完整可运营的线上服务。
8. 当前多渠道对话台仍要防止出现只有图标、没有真实功能的控件。

任何后续文档、页面或测试，如果把以上边界写成“已经完成”，必须停止并修正文案或功能。

## 4. 总施工顺序

推荐顺序如下：

```text
H2W-0 前端功能真实性门禁
  -> H2W-1 多渠道对话台瘦身与真实消息读写口径
  -> H2W-2 客户知识建设中心
  -> H2W-3 线上回执与准确率闭环
  -> H2W-4 XLSX/PDF/DOCX 报告与签收
  -> H2W-5 云接收台
  -> H2W-6 本地更新、恢复和回滚
  -> H2W-7 生产级 RAG 与模型成本治理
```

其中 H2W-0 是所有后续工作的前置门禁。H2W-0 不通过，不允许继续新增客户可见功能。

## 5. 总体验收门禁

整个 H2W 推荐版必须同时满足：

1. 每个客户可见页面都有控件级真实性矩阵。
2. 每个写动作都有前端动作、API client、后端接口、权限校验、数据变化、失败反馈和证据。
3. 每个读动作都有真实接口或明确本地兜底说明，筛选、分页、排序口径可验证。
4. 每个高风险词都不能是假能力：发送、转接、结束、发布、导入、导出、上传、应用、回滚、备份、恢复、同步、自动回复。
5. 多渠道对话台只呈现真实客服消息体验，不堆大块内部工程说明。
6. 知识建设中心能让小微企业客户自己录入、预检、发布和复盘知识。
7. 质量复盘能说明错误原因，并能回到知识修复或策略调整。
8. 报告、签收、诊断包、更新包、恢复包都有版本、时间、操作者和可回溯记录。
9. 真实外发关闭时，任何页面不得暗示已经发送到外部平台。
10. 每一阶段都有静态检查、类型检查、构建检查、浏览器 smoke 或后端测试证据。
11. Project_012 的 `执行记录.md`、`文件索引.md`、`关键决策.md`、`复盘与采坑.md` 已写回。

## 6. 总体停止门禁

出现以下任一情况，必须停止继续新增功能，先修复阻断项：

1. 客户可见按钮点击后无响应，且没有禁用说明。
2. 页面出现“已自动发送、已接通全渠道、已完成线上准确率、已上传云端、已正式签收”等超过真实能力的文案。
3. 前端有写动作，后端没有接口。
4. 后端有接口，但前端按钮没有接入，却对客户展示为可用。
5. 写动作没有权限校验或审计记录。
6. 筛选按钮只改变样式，不改变数据口径，也没有本地兜底说明。
7. 浏览器 smoke 只截首页，没有覆盖关键按钮和关键页面。
8. 使用演示样本冒充真实客户数据。
9. 把 dry-run 写成已执行。
10. 把 draft-only 写成已发送。
11. 把检索评测写成完整客服准确率。
12. 把本地确认记录写成正式电子签章。
13. 把 RPA 研究线写成正式默认交付。
14. 没有写回项目记录就声称阶段完成。

## 7. H2W-0 前端功能真实性门禁

### 7.1 目标

把所有客户可见前端控件逐项核对到真实能力，先消灭假按钮、假文案、假流程。

### 7.2 施工内容

2. 覆盖十个客户可见页面：

   - 质量复盘

2. 矩阵至少覆盖 30 个客户可见控件。

5. 多渠道对话台存在明确的真实外发关闭提示。
6. 回复按钮文案不暗示真实外发。
7. 静态门禁通过。
8. 前端 typecheck 通过。
9. 前端 build 通过。
10. 浏览器 smoke 覆盖所有目标页面。

1. 矩阵没有覆盖多渠道对话台，停止。
2. 矩阵只写页面，不写控件明细，停止。

4. “发送、转接、结束、上传、应用、回滚”等高风险按钮是假动作，停止。
5. 浏览器 smoke 没有覆盖关键控件，停止。

## 8. H2W-1 多渠道对话台真实客服体验

### 8.1 目标

把多渠道对话台做成真实客服软件的主工作区，而不是内部工程面板。

### 8.2 施工内容

1. 左侧会话列表继续瘦身，只保留头像、客户名、渠道、最近一句、等待时间、未读数和状态。
2. 主筛选只保留：全部、我的、转人工。
3. 右侧主区只保留消息气泡、系统弱提示和轻量输入区。
4. AI 判断、知识依据、外发门禁、失败复盘等放入详情抽屉或质量复盘页，不占据主聊天流。
5. 转人工状态以左侧小标签、输入区可用状态、会话状态体现，不再用大块提醒。
6. 消息列表读取真实会话详情接口；没有真实消息时展示空状态，不伪造已接通。
7. 如果仍未接真实平台，必须明确这是本地接待工作台，不暗示已经连到外部平台。

### 8.3 验收门禁

1. `#live` 首屏接近客服软件：左侧列表、右侧聊天、底部输入，不能被大块卡片淹没。
2. 左侧列表宽度适合大量会话扫描，不再是大卡片堆叠。
3. 点击会话后，右侧读取对应消息流。
4. 全部、我的、转人工三个筛选能改变列表数据或有明确本地兜底说明。
5. 转人工会话能保存本地接管回复或进入内部记录，不能伪装外发。
6. 页面没有“AI 自主回复预备”等内部工程口吻。
7. 浏览器 smoke 对列表切换、会话点击、搜索和输入区做检查。

### 8.4 停止门禁

1. 对话台仍出现大量内部决策说明卡，停止。
2. 左侧会话列表无法承载大量会话扫描，停止。
3. 点击会话只改变高亮，不改变右侧消息，停止。
4. 回复框让客户误以为已真实发送外部平台，停止。
5. 转人工仍以大块重复提醒占据聊天区，停止。

## 9. H2W-2 客户知识建设中心

### 9.1 目标

让中小企业客户能自己维护客服知识，而不是依赖工程师改 JSON 或数据库。

### 9.2 四层知识模型

| 层级 | 客户理解 | 后端对象 | 用途 |
|---|---|---|---|
| 业务对象 | 产品、服务、套餐、门店、活动 | `business_objects` | 判断客户问的是什么 |
| 标准问答 | 高频问答、相近问法、标准答案 | `object_knowledge_cards` | 生成可引用回答 |
| 流程政策 | 售后、退款、发票、物流、保修、预约 | policy 类知识卡或扩展字段 | 处理流程类问题 |
| 禁用承诺与转人工规则 | 不能承诺、必须人工确认、投诉赔付 | reply policy/rules | 阻断风险回复 |

### 9.3 施工内容

1. 知识库运营页提供四层入口。
2. 新增业务对象必须可点击、可提交、可落库、可刷新。
3. 标准问答必须能绑定业务对象。
4. 流程政策必须有明确分类、适用范围和生效状态。
5. 禁用承诺和转人工规则必须能被自动回复策略读取或至少进入待接入清单。
6. 支持客户导入 50-100 条脱敏问题和知识包模板。
7. 发布前必须走预检和样题试跑。
8. 发布后必须记录版本、发布人、发布时间和影响范围。

### 9.4 验收门禁

1. 新增业务对象按钮可点击，并能真实保存。
2. 编辑业务对象后刷新页面仍能读取最新值。
3. 新增问答卡能绑定业务对象，并能在知识库列表看到。
4. 导入模板能通过 dry-run 显示新增、跳过、错误和风险。
5. 知识评测能读取题库或样题，输出发布前结论。
6. 禁用承诺和转人工规则至少能进入策略配置页，不能只停留在文案。
7. 所有客户可见错误提示必须能被非工程用户理解。

### 9.5 停止门禁

1. 新增业务对象点不了，停止。
2. 保存后只改前端状态、不写后端，停止。
3. 导入只有上传框，没有后端预检，停止。
4. 发布前没有评测或预检，停止。
5. 知识评测仍写成完整客服准确率，停止。

## 10. H2W-3 线上回执与准确率闭环

### 10.1 目标

建立“问题 -> 回答 -> 人工标签 -> 错因 -> 知识修复 -> 回归验证”的闭环。

### 10.2 施工内容

1. 最终回复样本导入和采样。
2. 人工事实性标签：正确、部分正确、错误、需人工、禁用承诺风险。
3. 错因分类：无知识、知识过旧、引用不足、策略错误、模型误判、渠道异常。
4. 质量复盘页展示准确率、转人工率、错因分布、修复后变化。
5. 错因能跳转到知识缺口或自动回复策略。
6. 线上回执暂未接真实平台时，必须把指标命名为“本地样本质量”或“人工标签质量”，不能写成完整线上准确率。

### 10.3 验收门禁

1. 能导入或采样最终回复样本。
2. 人工标签能保存到后端。
3. 质量复盘能读取标签结果。
4. 每个错误样本能进入错因分类。
5. 错因能生成知识修复任务或策略调整建议。
6. 修复后能运行回归题集并比较前后变化。

### 10.4 停止门禁

1. 没有真实样本或人工标签，却展示准确率，停止。
2. 错因不能落库，只在前端显示，停止。
3. 无法从错误样本跳转到知识修复，停止。
4. 把本地样本质量写成线上全量准确率，停止。

## 11. H2W-4 报告导出与签收

### 11.1 目标

把质量报告、知识发布报告、客户签收记录做成可交付、可归档、可追溯的正式材料。

### 11.2 施工内容

1. 支持 XLSX 导出数据明细。
2. 支持 PDF 或 DOCX 导出客户可读报告。
3. 报告包含版本、时间范围、知识包版本、样本量、指标定义和边界说明。
4. 签收记录包含签收版本、签收时间、签收人标识、适用范围。
5. 签收记录不能写成电子签章，除非接入真实电子签服务。

### 11.3 验收门禁

1. 至少一种结构化数据导出可用，优先 XLSX。
2. 至少一种客户可读报告导出可用，优先 DOCX 或 PDF。
3. 导出文件能打开，格式正常。
4. 报告指标和系统页面口径一致。
5. 签收记录能列表查看、详情查看、按版本追溯。

### 11.4 停止门禁

1. 只生成 HTML 预览，却声称支持 PDF/DOCX，停止。
2. CSV 被写成 XLSX，停止。
3. 签收记录没有版本和适用范围，停止。
4. 本地确认被写成正式电子签章，停止。

## 12. H2W-5 云接收台

### 12.1 目标

为客户本地部署提供远程运维入口：客户本地生成诊断包，我方云端接收、登记、处理、回传更新建议或更新包。

### 12.2 施工内容

1. 云端接收台 API：上传诊断包、质量报告、更新请求。
2. 云端接收台页面：列表、详情、下载、处理状态、备注。
3. 客户本地端：生成包、预检包、上传包、上传记录。
4. 包内容脱敏、体积限制、格式校验。
5. 不做长期远控，不做静默控制客户电脑。

### 12.3 验收门禁

1. 本地能生成诊断包。
2. 云端能接收包并登记。
3. 上传失败能重试或显示可理解错误。
4. 云端能查看包元数据，不展示敏感原文。
5. 每次上传有租户、版本、时间、包类型和处理状态。

### 12.4 停止门禁

1. 没有云端接收接口，却写成“已上传云端”，停止。
2. 上传包包含明文密钥、密码、Token，停止。
3. 云端无法区分客户、版本和包类型，停止。
4. 以远控代替诊断包机制，停止。

## 13. H2W-6 本地更新、恢复和回滚

### 13.1 目标

让客户本地应用能安全接受知识包、策略包和程序包更新，并支持预检、应用、失败回滚和操作记录。

### 13.2 施工内容

1. 签名知识包导入与回滚。
2. 签名策略包导入与回滚。
3. 程序更新包 dry-run 到真实更新的分阶段路线。
4. 本地 SQLite 备份、校验、恢复 dry-run。
5. 更新前强制备份。
6. 更新后健康检查。
7. 失败时回滚并记录原因。

### 13.3 验收门禁

1. 更新包必须有签名或校验摘要。
2. 应用前必须预检。
3. 应用前必须生成备份。
4. 应用后必须能看到版本变化或数据变化。
5. 失败时必须回滚到更新前状态。
6. 每次操作必须进入操作记录。

### 13.4 停止门禁

1. 没有签名或校验就允许应用，停止。
2. 没有备份就应用更新，停止。
3. 失败不能回滚，停止。
4. dry-run 被写成真实程序更新，停止。
5. 更新过程没有审计记录，停止。

## 14. H2W-7 生产级 RAG 与模型成本治理

### 14.1 目标

把“能回答”升级为“可解释、可控成本、可回归、可降级”的客服知识检索和模型调用体系。

### 14.2 施工内容

1. 生产级检索路线：关键词、向量、混合检索、重排、引用溯源。
2. 每次回答记录引用来源、置信度、转人工理由、模型调用成本。
3. 高置信 FAQ 不调模型。
4. 普通问题走默认模型。
5. 高风险、低置信、无知识、禁用承诺命中时转人工。
6. 模型预算：日预算、月预算、单次上限、降级策略。
7. 对客户显示为“自动回复策略”，不直接暴露复杂模型路由术语。

### 14.3 验收门禁

1. 每条自动回复都有引用来源或无引用原因。
2. 每条模型回复都有模型名、token 或费用估算。
3. 超预算后能降级或转人工。
4. 高风险问题不会自动承诺。
5. 评测集能覆盖事实性、引用充分、禁用承诺、转人工正确性。
6. 客户能在前端理解“直接回复、谨慎回复、转人工、暂不回复”的策略。

### 14.4 停止门禁

1. 无引用却高置信回答，停止。
2. 没有成本记录却展示成本治理，停止。
3. 高风险问题仍自动外发，停止。
4. 客户界面只写模型名，不解释策略，停止。
5. 把模型回答质量当成知识库质量，停止。
