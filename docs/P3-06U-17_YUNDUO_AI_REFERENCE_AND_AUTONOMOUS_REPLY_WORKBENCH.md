# P3-06U-17 云朵AI视频参考与自动回复工作台收束

日期：2026-07-02

## 本轮目标

本轮只处理一个核心问题：多渠道对话台不能继续呈现为“每条消息都等坐席生成、审核、备注、再发送”的人工审核工具。智能客服的主路径应该是：

```text
渠道入站消息 -> 统一会话/消息存储 -> 意图与风险判断 -> 知识检索 -> 模型生成 -> 高置信自动回复
                                                      -> 低置信/高风险/无知识/渠道异常转人工
```

人工审核、内部备注、发送队列和工单仍保留，但它们是异常处理和安全门禁，不是普通问题的默认路径。

## 对我们的直接结论

### 1. 多渠道对话台左侧会话列表应该更窄

当前左侧会话列表不需要占到 300px 左右。真实客服场景里，左侧主要承载：

- 客户头像或首字。
- 客户昵称。
- 渠道标识。
- 最后一条消息。
- 等待时间。
- 待审、待发、超时、异常等少量状态。

本轮已将桌面端会话列从 `260-304px` 收窄为 `224-252px`，把主要空间让给右侧大对话区。

### 2. 所有渠道消息必须进入统一收录层

多渠道对话台不能只是展示样本。正式工程里，不管消息来自官网、企业微信、公众号、抖音、淘宝、京东、拼多多、小红书，最终都要进入统一结构：

| 字段组 | 必须保留的信息 |
|---|---|
| 租户与渠道 | `tenant_id`、`channel_id`、`channel_type`、`channel_name` |
| 平台原始标识 | 平台会话 ID、平台消息 ID、平台用户 ID、店铺 ID、商品 ID、订单 ID |
| 统一会话 | `conversation_id`、`contact_id`、会话状态、分配坐席、SLA 状态 |
| 消息正文 | 方向、文本、附件、图片/视频/订单卡片、发送时间、接收时间 |
| 原始载荷 | 脱敏后的 `raw_payload_json`、签名校验结果、幂等键 |
| AI 决策 | 意图、知识命中、置信度、回复策略、是否自动回复、是否转人工 |
| 外发状态 | outbox 状态、渠道发送状态、回执、失败原因、重试记录 |

本轮界面已在 AI 辅助抽屉增加“渠道入站”卡片，明确每个会话都要保留渠道、会话和联系人标识。

### 3. 默认路径应该是 AI 自主回复

用户把知识运营、产品知识、FAQ、规则、转人工条件配置好之后，普通高置信问题应该由 AI 自动回复。人工处理只在这些情况下出现：

- 无知识命中。
- 置信度低。
- 赔付、投诉、辱骂、法务、医疗、金融等高风险。
- 客户连续追问仍未解决。
- 图片、视频、订单卡片等结构化上下文暂时无法可靠解析。
- 渠道发送失败或平台回执异常。
- 命中客户自定义的必须转人工规则。

因此本轮把回复区从“坐席回复草稿/等待生成/内部备注/审核意见”改为“AI 自动回复预案/异常人工门禁”。内部备注被折叠到“人工接管 / 异常备注”，避免普通坐席误以为所有消息都要人工写审核意见。

### 4. 云朵AI值得借鉴的核心能力

| 能力 | 我们的转化方式 |
|---|---|
| 多平台账号列表 | 渠道连接器中心 + 对话台渠道入站卡 |
| 每店铺自动回复开关 | 每渠道/每业务对象的自动回复策略和 kill switch |
| 售前、售后、转人工、AI 自动设置 | 回复策略分层，不再只靠一个提示词 |
| 商品学习 | 业务对象知识库：商品、服务、套餐、课程都可挂知识 |
| 新建问答、批量导入、批量导出 | 知识运营台加入表格导入、清洗、发布前回归 |
| AI 协助中心 | 保留为折叠式 AI 辅助抽屉，不做过多浮窗 |
| 已回复隐藏/消息通知 | 队列清洁、已处理自动降权、未处理优先 |
| 会话区联动订单/商品 | 对话台右侧或抽屉展示订单、商品、客户历史 |

### 5. 不应照搬的部分

- 不照搬其蓝色重边框风格。我们的中台应更稳、更克制、更像正式运营软件。
- 不把平台接入能力说成已经合规可用。视频只能证明一个产品界面，不证明它的接入方式、授权方式、风控稳定性和平台许可。
- 不把“自动回复”理解为无门禁乱发。自动回复必须受知识命中、模型路由、敏感词、渠道授权、外发开关和质量回归控制。
- 不让配置页变成巨大的规则迷宫。中小企业需要默认模板、少量关键开关和可解释的异常路径。

## 本轮已改动的产品方向

| 位置 | 改动 |
|---|---|
| 多渠道对话台 | 左侧会话列表收窄，右侧对话区成为主视觉 |
| AI 辅助抽屉 | 新增“渠道入站”上下文卡，保留渠道、会话和联系人标识 |
| 回复区 | 改为“AI 自动回复预案”，默认强调自动回复，异常才人工确认 |
| 人工备注 | 从常驻表单降级为折叠的“人工接管 / 异常备注” |
| 操作文案 | “批准进入待发送”改为“人工确认放行”，“确认待发送”改为“确认发送队列” |
| 安全边界 | 继续明确真实外发依赖官方渠道授权、白名单测试和外发开关 |

## 下一步建议

下一步不要继续只磨视觉。应该进入两条并行线：

1. **知识运营线**：把“商品学习”升级成我们的“业务对象知识库”，先支持商品/服务/套餐三个对象类型，支持批量导入、清洗、发布前回归。
2. **回复策略线**：把自动回复策略落到后端字段和状态机里，明确 `auto_reply`、`manual_gate`、`draft_only`、`blocked`、`failed` 等状态。

完成这两条线后，多渠道对话台才真正从“漂亮客服界面”变成“可持续自动回复的客服系统”。
