## 本轮目标

P3-03 的目标不是新增一轮离线评测，而是把已经存在的后端链路整理成坐席日常可用的工作台主屏。坐席打开页面后，应该能立刻知道：

1. 哪些会话需要人工审核。
2. 哪些草稿已经批准但还没有发送确认。
3. 哪些发送或回执需要失败复盘。
4. 哪些问题暴露了知识缺口。
5. 哪些会话属于高风险投诉、赔付、合同、法务或平台规则问题。
6. 最近一次知识评测是否暴露风险。
7. 渠道和外部写入当前是否安全。

## 已落地的主屏队列

前端 `frontend/src/App.tsx` 新增/使用 `WorkbenchCommandCenter` 和 `buildWorkQueues`，主屏展示七类队列：

| 队列 | 数据来源 | 作用 |
| --- | --- | --- |
| 待人工审核 | `human-review-inbox` | 坐席核对低置信、无知识命中、模型不可用和高风险草稿 |
| 待发送确认 | `outbox-drafts` 中 `pending_confirmation` | 避免审核通过后直接外发 |
| 发送失败 | `delivery-failure-reviews` | 集中处理回执、限流、权限和内容拒绝问题 |
| 知识缺口 | 最近评测运行的失败原因 | 提醒知识运营补文档、补证据或复核标注 |
| 高风险会话 | 人审任务里的 `high/critical` 风险等级 | 让赔付、合同、投诉和法务类问题优先处理 |
| 最近评测运行 | 知识评测运行摘要 | 展示最近质量门禁，而不是继续扩展评测链 |
| 渠道健康 | 失败复盘和发送队列异常 | 明确真实外部写入仍关闭，避免误判平台已接通 |

## 已落地的会话证据详情

前端 `ReviewEvidenceDetail` 把单个待审核任务聚合成证据面板，包含：

| 信息 | 目的 |
| --- | --- |
| 原始入站 | 坐席先看用户到底问了什么 |
| AI 草稿 | 坐席只审核草稿，不盲目发送 |
| 引用证据 | 展示知识命中、来源和摘要 |
| 风险等级 | 区分低风险问题和投诉/赔付/法务问题 |
| 转人工原因 | 说明为什么不能自动回复 |
| 模型/provider 状态 | 说明 provider、model、route 和状态 |
| Outbox 状态 | 展示是否已生成草稿、队列任务或失败复盘 |
| 审计链 | 串起入站消息、workflow、人审、outbox 和外部写入关闭状态 |
