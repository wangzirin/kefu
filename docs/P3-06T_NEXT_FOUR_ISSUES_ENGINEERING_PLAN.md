## 问题一：壳层滚动必须返修验收

### 当前问题

桌面端理应是左侧黑色导航固定，右侧白色工作区独立滚动。用户实测出现“页面向下滑时，左侧导航也跟着滑”的情况，这说明当前壳层滚动仍不能按完成处理。

当前 CSS 里已经有 `html/body/#root overflow: hidden`、`.app-shell height: 100dvh`、`.sidebar height: 100dvh`、`.workspace overflow-y: auto` 的设计，但实际体验仍可能被断点、浏览器窗口宽度、外层容器或移动布局覆盖。

### 工程目标

让桌面中台始终满足：

- `window.scrollY = 0`
- `.workspace.scrollTop` 承担主滚动
- `.sidebar` 视觉位置不随右侧内容移动
- 721px 到 960px 的窄桌面窗口不误进入移动长页面模式
- 手机端保留自然滚动，但不影响桌面中台形态

### 施工范围

- `frontend/src/styles.css`
- `frontend/src/App.tsx`
- 新增或更新浏览器滚动验收脚本与截图目录

### 施工步骤

1. 重新核验 `.app-shell`、`.sidebar`、`.workspace` 的高度链路。
2. 检查 `@media (max-width: 960px)` 和 `@media (min-width: 761px) and (max-width: 960px)` 是否互相覆盖。
3. 固化桌面、窄桌面、移动端三套滚动规则。
4. 写一个最小浏览器验收脚本：
   - 打开首页。
   - 滚动 `.workspace` 900px。
   - 读取 `window.scrollY`、`.workspace.scrollTop`、`.sidebar.getBoundingClientRect().top`。
5. 保存滚动前、滚动后截图。

### 验收标准

- 1440x900、1280x800、1024x768、900x768：左侧栏不跟随页面下滑。
- 390x844：无横向溢出，导航不遮挡内容。
- 验收结果不能只写“已修复”，必须有数值和截图。

### 预计时间

0.5 到 1 天。

## 问题二：运营总览要重做成真正的 BI 首页

### 当前问题

当前首页已经有图表和指标，但整体更像“信息卡片集合”，还不像成熟客服系统的运营指挥舱。主要问题是：

- 首屏不够惊艳。
- 卡片太大，信息密度偏低。
- 图表和经营动作之间关联弱。
- 一眼看不出“今天最该处理什么”。
- 缺少成熟 BI 的时间范围、渠道对比、趋势、异常、错因复盘组合感。

### 工程目标

首页必须回答四个问题：

1. 今天服务压力高不高。
2. AI 有没有真正减少人工压力。
3. 哪些渠道或会话正在出问题。
4. 运营人员下一步应该先处理什么。

### 施工范围

- `frontend/src/App.tsx`
- `frontend/src/styles.css`
- 建议新增 `frontend/src/components/ops/`
- 建议新增 `frontend/src/components/charts/`

### 推荐布局

首屏结构改为：

1. 顶部经营状态条：进线量、AI 自动解决率、转人工率、平均响应、风险会话。
2. 左侧主图：会话趋势、渠道压力或运营健康度。
3. 右侧动作栏：高风险审核、待确认草稿、渠道异常、知识缺口。
4. 第二屏图表组：处理漏斗、渠道异常矩阵、AI 命中趋势、错因 Pareto。

### 视觉方向

- 借鉴 shadcn-admin / TailAdmin 的清爽后台密度。
- 借鉴 Ant Design Pro 的 Analysis / Monitor / Workplace 信息分区。
- 借鉴 Tremor 的指标卡和图表组合节奏。
- 借鉴 Grafana 的时间窗口、数据源、告警状态表达。
- 保留当前品牌色，但降低大块卡片和长段解释文字。

### 验收标准

- [x] 1440x900 首屏无需下滑即可看出运营健康度、风险、优先动作。
- [x] 至少 3 个核心图表，不只是数字卡片。
- [x] 每个指标都带状态或可点击动作。
- [x] 保留图表加载态；空态继续沿用现有样本/本地汇总提示。
- [x] 截图看起来更接近可对外演示的数据中台，而不是内部测试页。

### 预计时间

已完成第一片。后续如继续精修，应拆分图表组件、补 drill-down、补真实历史聚合和缓存。

## 问题三：首页数据口径要继续收紧

### 当前问题

首页会显示服务端聚合、本地汇总或样本数据。开发阶段可以这样做，但商用演示时风险很大：客户看到图表时必须清楚知道数据从哪里来、时间范围是什么、是否被渠道筛选、是不是样本。

### 工程目标

所有首页指标必须可追溯：

- 数据模式：真实聚合 / 本地汇总 / 演示样本
- 时间窗口：今日 / 近 7 天 / 近 30 天
- 渠道筛选：全部渠道 / 指定渠道
- 生成时间：后端生成时间
- 敏感边界：不得返回客户原文、AI 草稿正文、待发送正文

### 施工范围

- `backend/app/api/ops.py`
- `backend/app/schemas/ops.py`
- `backend/tests/test_p3_06r_ops_dashboard_api.py`
- `frontend/src/api/client.ts`
- `frontend/src/App.tsx`

### 施工步骤

1. 后端响应补齐并稳定：
   - `data_mode`
   - `data_source`
   - `source_window`
   - `filters`
   - `generated_at`
2. 前端不静默混用数据：
   - 当前筛选没有服务端数据时，必须明确显示本地汇总或样本。
   - 不能把样本数据包装成真实经营数据。
3. 增加契约测试：
   - owner/admin/viewer 可读。
   - agent 不可读。
   - 不返回敏感正文。
   - range/channel 参数结构稳定。

### 验收标准

- 首页每个指标都能追溯到接口字段。
- 演示数据、服务端聚合、本地汇总有明确区分。
- 后端测试覆盖权限、字段和敏感信息边界。
- 前端切换时间范围/渠道时不出现误导性 0 或旧数据。

### 预计时间

1 到 1.5 天。

## 问题四：页面信息架构要按运营闭环重排

### 当前问题

现在页面数量不少，但仍像“功能模块列表”。成熟客服中台不是让运营人员自己找功能，而是要把路径排清楚：

1. 首页发现问题。
2. 进入对话台处理客户。
3. 进入审核/待发送处理风险。
4. 进入质量复盘找到错因。
5. 进入知识库修复知识。
6. 回到首页看指标是否改善。

### 工程目标

把左侧导航和首页入口重排成客服运营路径，而不是工程模块清单。

### 施工范围

- `frontend/src/data/navigation.ts`
- `frontend/src/App.tsx`
- `frontend/src/styles.css`
- 建议新增 `docs/P3-06T_FUNCTION_NECESSITY_MATRIX.md`

### 施工步骤

1. 做功能必要性矩阵：
   - 必须保留：运营总览、对话台、人工审核、待发送、质量复盘、知识库、渠道连接器、联系人/线索、系统运维。
   - 可以合并：重复状态卡、重复长说明、只读列表中的弱入口。
   - 暂缓前置：内部阶段编号、过细底层队列、客户不需要理解的工程标签。
2. 重排左侧导航：
   - 总览
   - 接待处理
   - 质量复盘
   - 知识运营
   - 渠道连接
   - 客户资产
   - 系统运维
3. 首页所有入口必须能落到一个可执行页面：
   - 高风险 -> 审核
   - 待发送 -> 待发送
   - 渠道异常 -> 渠道连接器
   - 知识缺口 -> 知识库/质量复盘

### 验收标准

- 左侧导航不再像工程清单。
- 首页入口都能解释“为什么现在点这里”。
- 不删除底层能力，只把底层能力放回管理/诊断区。
- 客户演示时不会出现大量内部阶段词。

### 预计时间

1 到 1.5 天。
