## 1. 纠偏后的最终结论

上一轮把重点放到了 MemoryMesh / OCA，这是错误重心。对当前商业化主线来说，应该回答的是：

```text
SAG 对“万法常世 AI 全智能客服系统”的知识库、回复质量、人工审核和商业套餐有没有作用？
```

结论是：

**有作用，但不能替代当前客服中台，也不能替代现有 RAG。SAG 更适合作为智能客服知识库的“复杂知识检索增强层”，用于多文档、多政策、多步骤、多实体关系的客服问题。**

它最适合解决这些问题：

- 售后规则跨多份文档，例如发货、签收、退换货、质保、特殊品类限制一起判断。
- 投诉、退款、赔付、合同、发票、法务类问题，需要找到多条政策证据。
- 多渠道运营 SOP，例如企业微信、公众号、抖音、小红书、淘宝不同回复边界。
- 复杂售前方案，例如价格、套餐、实施周期、私有化、接口范围、SLA 同时被问到。
- 客服知识长期演化，例如旧政策被新政策替换、某类错答反复出现、某个客户/产品有历史上下文。
- 坐席审核时需要解释“系统为什么推荐这条回复、依据是哪几段”。

它不适合解决这些问题：

- 不是渠道接入方案。企业微信、公众号、淘宝、抖音、小红书仍要走官方 API / 服务商授权。
- 不是坐席工作台。会话收件箱、人工审核、outbox、失败复盘仍是我们自己的中台能力。
- 不是所有 FAQ 的替代品。简单问候、营业时间、物流话术、退换货固定规则，用 FAQ / BM25 / 向量检索更便宜。
- 不是直接商用成品。SAG 仓库是工作台/服务参考，不是可以原样包装交付客户的客服系统。

一句话：

**智能客服的主线应是“客服中台 + RAG 知识库 + 模型路由 + 人工审核 + 渠道 outbox”，SAG 只进入“复杂知识检索增强层”。**

## 2. 当前智能客服原系统解读

当前 `standard_ops` 已经不是空壳。它的真实结构如下：

```text
渠道入站 / 会话
  -> Conversation / Message
  -> ReplyOrchestrator
      -> classify_intent
      -> retrieve_knowledge
      -> draft_reply
      -> risk_check
  -> HumanReviewTask
  -> OutboxDraft
  -> DeliveryJob / ChannelReceipt / FailureReview
```

知识库现状：

| 模块 | 当前能力 | 商用评价 |
|---|---|---|
| `knowledge_cards` | 结构化 FAQ / 标准话术 / 标签 / 别名 | 适合 Lite 和标准客服基础问答 |
| `knowledge_documents` | 导入文档原文、自动切 chunk | 已进入文档级知识库雏形 |
| `knowledge_document_chunks` | chunk 内容、引用位置、embedding 元数据 | 已有 RAG 基础，但还不是强生产 RAG |
| `knowledge_searches` | `lexical_bm25_v1` 卡片检索 | 适合短问题和标准 FAQ |
| `knowledge_document_searches` | `hybrid_bm25_vector_rerank_v1` 文档片段检索 | 适合文档片段召回，但多跳弱 |
| embedding | deterministic 本地 embedding、OpenAI-compatible provider smoke、pgvector exact path | 工程链路有了，真实语义质量还没完成 |
| pgvector | exact cosine、ANN dry-run、本地 ANN smoke | 未切换生产知识表 ANN query path |
| evaluation | 知识评测集、命中率、引用覆盖、期望词覆盖 | 已有检索评测入口，但还没有完整答案事实性评估 |

模型与回复现状：

| 模块 | 当前能力 | 商用评价 |
|---|---|---|
| `ReplyOrchestrator` | `manual`、`knowledge_search`、`document_rag`、`model_assisted` | 核心编排已成型 |
| `document_rag` | 用最高相关 chunk 生成带引用草稿 | 目前偏“检索片段回复”，还不是完整生成式 RAG |
| `model_assisted` | 先搜知识卡片，再调用模型网关生成草稿 | 当前主要接结构化卡片，不是文档 RAG + 模型一体化 |
| 模型路由 | simple / standard / premium / deterministic fallback | 有商用治理雏形 |
| 人审规则 | 无知识命中、低置信、中高风险进入人工审核 | 商业安全线正确 |
| outbox | 已批准回复进入内部待发送、确认、dry-run、队列骨架 | 仍未真实外发 |

关键缺口：

1. 当前知识检索更像“卡片检索 + 文档 chunk 检索”，还没有事件、实体、关系层。
2. `document_rag` 当前拿 top chunk 直接生成草稿，多文档、多政策、多步骤问题容易漏证据。
3. `model_assisted` 目前主要用知识卡片，不是“文档检索结果 + 引用证据 + 模型生成”的完整链路。
4. 评测目前偏检索命中和引用覆盖，还没有端到端答案事实性、幻觉率、转人工合理性评测。
5. 客服知识还没有“旧政策替换新政策”“某类问题反复错答”“知识缺口自动归类”的长期结构。

这正是 SAG 思路能帮上的地方。

## 3. SAG 对客服系统真正有用的点

SAG 的核心不是“换一个向量库”，而是新增一层结构化知识：

```text
chunk -> event
chunk -> entities
event <-> entities
```

放到智能客服里，可以翻译成：

| SAG 概念 | 客服系统里的含义 | 用途 |
|---|---|---|
| chunk | 文档片段、FAQ、政策段落、工单摘要 | 原始证据和引用来源 |
| event | 一条可执行/可解释的客服事项 | 比如“七天无理由退货需满足未拆封” |
| entity | 商品、订单、渠道、政策、客户类型、时间、服务项 | 支撑别名和跨文档连接 |
| edge | 事项与实体的关系 | 支撑多跳检索和证据链 |

示例：

```text
客户问：小红书买的套装拆封后过了 8 天还能退吗？

普通 RAG 可能命中：
- 退换货政策 chunk
- 小红书渠道说明 chunk
- 套装商品说明 chunk

SAG 增强后应该能串起来：
- entity: 小红书渠道
- entity: 套装商品
- event: 小红书订单售后需按平台规则处理
- event: 拆封商品不适用无理由退货
- event: 超过 7 天进入人工售后确认
```

这类问题如果只靠 top1 chunk，很容易漏掉其中一个条件；SAG 的价值就是提高“完整证据链召回”。

## 4. 跑分报告对客服系统的真实含义

本地报告里最重要的结果不是“所有检索都变好”，而是：

| 数据集 | 关键结果 | 对客服系统的解释 |
|---|---|---|
| HotpotQA | FullRecall@5 从 44.80% 到 66.40% | 多跳证据更容易一起召回 |
| 2WikiMultiHopQA | FullRecall@5 从 32.40% 到 46.00% | 跨实体/跨文档关系有帮助 |
| MuSiQue | Recall@5 和 NDCG@5 退化，FullRecall@5 小幅升 | 简单一跳扩展会引入噪声，必须有 rerank 和失败复盘 |
| 中文 synthetic memory | 样本太小，只能说明排序可能改善 | 不能证明中文客服知识库已验证 |

对客服系统的判断：

1. SAG 对“复杂售后 / 多政策 / 多渠道规则 / 方案报价 / 合同争议”有明显验证价值。
2. SAG 对“你好、多少钱、发货多久、怎么联系人工”这类问题没有必要进入主链。
3. 当前报告只能支持我们做 shadow 验证，不能支持马上对客户承诺“更智能、更准确、全场景提升”。
4. 商用系统应使用“分层检索路由”，而不是所有问题都跑 SAG。

## 5. 改进后的客服知识库架构

推荐目标架构：

```text
客户消息
  -> 意图 / 风险 / 复杂度分类
  -> 检索路由
       simple_faq: knowledge_cards / BM25
       document_policy: hybrid_bm25_vector_rerank_v1
       complex_multi_policy: semantic_event_graph
       high_risk: 检索证据 + 人工审核
  -> 候选证据合并
  -> 引用与事实约束
  -> 模型生成草稿
  -> 风险检查
  -> 自动完成或人工审核
  -> outbox 确认发送
```

新增的 SAG-like 层不建议直接命名为 `sag_*`，更适合产品化命名为“语义事件索引”：

| 建议表 | 作用 |
|---|---|
| `knowledge_semantic_events` | 从文档 chunk、FAQ、历史工单中抽取的客服事项 |
| `knowledge_semantic_entities` | 商品、政策、渠道、客户类型、时间、风险、服务项等实体 |
| `knowledge_semantic_event_entities` | 事项与实体的关系 |
| `knowledge_semantic_aliases` | 实体别名、简称、常见问法 |
| `knowledge_retrieval_traces` | 每次检索命中的 chunk/event/entity 路径 |
| `knowledge_answer_evaluation_cases` | 面向客服答案质量的评测题 |
| `knowledge_gap_clusters` | 无命中/低置信/错答问题聚类 |

每个 event/entity 必须继承：

- `tenant_id`
- `document_id`
- `chunk_id`
- `source_uri`
- `status`
- `visibility`
- `risk_level`
- `created_by_id`
- `extractor_model`
- `extractor_prompt_version`
- `confidence`
- `content_hash`

## 6. 与当前工程的具体接入点

### 6.1 知识导入阶段

当前：

```text
create_knowledge_document
  -> split chunks
  -> embedding
  -> write KnowledgeDocumentChunk
```

建议增强为：

```text
create_knowledge_document
  -> split chunks
  -> embedding
  -> write KnowledgeDocumentChunk
  -> optional semantic event extraction
       -> extract events
       -> extract entities
       -> write event/entity/relation
       -> write extraction trace and cost
```

工程注意：

- 不要照搬 SAG “每个 chunk 只保留一个 event”的限制。客服政策文档里一个 chunk 可能包含多条规则，应允许 1-N 个 event。
- 抽取必须可重跑。模型、prompt、版本变了，event/entity 要能重建。
- 先只对 `active` 文档抽取，不对 draft 文档抽取。
- 对客户隐私、订单号、手机号、地址等字段，默认不抽实体或做脱敏占位。

### 6.2 检索阶段

当前：

```text
search_knowledge_cards
search_knowledge_documents
```

建议新增：

```text
search_semantic_events
hybrid_customer_service_retrieval
```

路由规则：

| 问题类型 | 走哪条检索 | 自动回复策略 |
|---|---|---|
| 问候、导航、营业时间 | `knowledge_cards` | 可自动回复 |
| 单一政策问题 | `knowledge_documents` | 高置信可自动回复 |
| 多条件售后问题 | `hybrid + semantic_events` | 默认人工审核或高置信辅助草稿 |
| 投诉、赔付、发票、合同 | `hybrid + semantic_events` | 必须人工审核 |
| 知识缺口 | 记录 `knowledge_gap` | 不让模型硬答 |

### 6.3 ReplyOrchestrator 阶段

当前模式：

- `knowledge_search`
- `document_rag`
- `model_assisted`

建议新增两个模式：

| 新模式 | 用途 |
|---|---|
| `rag_model_assisted` | 文档 chunk 检索 + 引用证据 + 模型生成草稿 |
| `semantic_event_assisted` | 复杂问题走 event/entity 多跳检索，再生成带证据链草稿 |

推荐链路：

```text
入站消息
  -> classify_intent
  -> classify_complexity
  -> route_retrieval
  -> retrieve_evidence
  -> generate_grounded_draft
  -> citation_check
  -> risk_check
  -> human_review or completed
```

关键差异：

- 模型不能只拿一条 top chunk，要拿结构化证据包。
- 证据包应包含 `citation`、`event_summary`、`entity_path`、`matched_terms`、`confidence`。
- 高风险问题即使命中充分，也只生成内部草稿，不自动外发。

### 6.4 前端运营台

不要做复杂图谱大屏作为第一优先级。客服运营更需要：

| 面板 | 应显示什么 |
|---|---|
| 知识检索详情 | 命中的 FAQ、chunk、event、引用和置信度 |
| 证据链 | 为什么推荐这条回复，经过哪些政策/商品/渠道实体 |
| 低置信复盘 | 哪些问题没命中、命中了错误证据、需要补知识 |
| 知识优化 | 把错答问题转成新知识卡片、文档修订或别名补充 |
| 评测报表 | 复杂题 FullRecall@5、引用覆盖、人工审核率、错答率 |

真正对坐席有价值的不是“图很炫”，而是能回答：

- 这条草稿依据哪几条政策？
- 哪一条政策最关键？
- 有没有冲突或过期知识？
- 为什么需要人工审核？
- 这个问题是否代表知识库缺口？

## 7. 商用套餐里的使用位置

SAG-like 检索不应该全套餐强塞。

| 商用档位 | 是否引入 SAG-like | 原因 |
|---|---|---|
| Lite 入门验证版 | 不建议实时引入 | FAQ + 轻量文档 RAG 足够，成本低，交付快 |
| 标准接待版 | 可做离线评测和知识优化，不建议默认实时 | 客户开始有文档和知识运营，但复杂度仍需控制 |
| 增强转化版 | 可引入 shadow 检索和复杂问题辅助草稿 | 售前方案、价格、权益、渠道政策容易多跳 |
| Professional / 标准运营版 | 建议引入 hybrid candidate | 有知识运营、评测、人审、outbox，能承接复杂证据链 |
| Enterprise / 强合规版 | 可生产化，但要权限、脱敏、审计、私有化模型策略 | 多租户、敏感数据、SLA 和复杂知识库才值得投入 |

商业上最合理的包装不是“我们用了 SAG”，而是：

- 复杂问题证据链增强。
- 多文档政策联合判断。
- 低置信与知识缺口复盘。
- 坐席审核可解释。
- 知识库持续优化。
- 高风险问题不自动硬答。

客户不需要知道 SAG 这个词；客户需要知道系统能更可靠地回答复杂问题，并且有证据可查。

## 8. 预计优化效果

以下是工程预测，不是已经在我们客服系统完成验证的事实。

| 指标 | Lite / FAQ 场景 | 标准文档 RAG 场景 | 复杂政策/售后/方案场景 |
|---|---:|---:|---:|
| 命中率 | 变化小，0-3 个百分点 | 可能 +3 到 +8 个百分点 | 可能 +8 到 +20 个百分点 |
| 完整证据召回 | 基本无提升 | 可能 +5 到 +12 个百分点 | 可能 +10 到 +25 个百分点 |
| 错答率 | 变化小 | 若引用检查完善，可下降 5%-15% | 若人审配合，可下降 10%-30% |
| 转人工率 | 变化小 | 初期可能上升，优化后下降 | 初期上升，成熟后下降 5%-15% |
| 坐席复核效率 | 小幅提升 | 中等提升 | 明显提升 |
| 入库成本 | 不建议增加 | +2 到 +4 倍 | +4 到 +8 倍 |
| 查询延迟 | 不建议增加 | +100ms 到 +500ms | +300ms 到 2s，取决于 rerank/模型 |
| 存储体积 | 不建议增加 | +2 到 +4 倍 | +3 到 +8 倍 |

最现实的收益：

- 入门包不会因此立刻更好卖，因为入门客户更看重快、便宜、能接待。
- 标准运营版会更有壁垒，因为复杂知识和审核证据是成熟客服系统的分水岭。
- 企业客户会更在意“错答可控、证据可查、知识可持续优化”，这正是 SAG-like 层的商业价值。

## 9. 风险和不能做的承诺

不能对外承诺：

- 不能说接入 SAG 后客服全场景准确率提升。
- 不能说它可以替代坐席。
- 不能说它可以自动处理投诉、赔付、合同、法务。
- 不能说它可以解决微信、淘宝、抖音等渠道接入。
- 不能说当前标准运营版已经完成生产级 SAG/RAG。

必须保留的商业安全线：

- 高风险问题必须进入人工审核。
- 无引用依据的问题不能让模型硬答。
- 客户隐私和订单信息不能随便进入实体索引。
- 不同租户的实体和知识不能互相混。
- 旧政策、过期政策必须能下线或降权。
- 所有自动草稿必须能追溯到知识来源。

  - `full_evidence_recall@5`
  - `citation_precision`
  - `unsupported_answer_rate`
  - `human_review_correctness`

只生成旁路 JSON，不影响真实回复。

- 复杂题 `full_evidence_recall@5` 至少提升 10 个百分点。

- citation 缺失或知识不足时进入人工审核。

### P2-25：新增 `semantic_event_assisted`

通过 P2-23 后再接入主编排。

任务：

- 复杂问题路由到 event/entity 检索。
- 合并 chunk、event、entity 证据。
- rerank 后给模型。
- 高风险默认人审。
- 前端展示证据链和知识缺口。

### P2-26：商用验收包

面向客户试点前必须形成验收资料：

- 题库质量报告。
- 引用覆盖报告。
- 转人工策略报告。
- 错答复盘报告。
- 成本与延迟报告。
- 知识缺口清单。
- 可上线 / 不可上线边界。

## 11. 明确取舍

当前我们不应该做：

- 不把 SAG 作为 Lite 入门包默认能力。
- 不把 SAG 原项目直接包装给客户。
- 不把智能客服从当前中台切到 SAG 工作台。
- 不为了图谱而图谱，先做证据链和评测。

当前我们应该做：

- 保留现有客服中台架构。
- 保留 FAQ / 文档 RAG / 模型路由 / 人审 / outbox。
- 在知识层新增 semantic event shadow。
- 先评测，再接主链。
- 把它包装成“复杂知识证据链增强”和“知识优化闭环”，而不是卖技术名词。

最终一句话：

**SAG 对我们的智能客服有用，但它应该增强客服知识库的复杂问题证据召回，不应该改写整个客服系统架构。商业上先做评测和 shadow，等复杂问题确实提升后，再把它放进增强版、专业版和企业版。**
