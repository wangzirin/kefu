# P3-06U-26H2W-0 前端功能真实性与前后端契约门禁计划

日期：2026-07-03

## 1. 文档定位

本文件是 `P3-06U-26H2W` 之前必须先执行的前置门禁。

它解决一个核心问题：客服中台前端不能出现“看起来能点击、看起来有能力、但没有真实功能闭环”的界面。尤其是多渠道对话台、知识运营、质量复盘、渠道接入和管理运维这些客户高频使用区域，必须先完成真实能力审计，再继续堆新功能。

如果把后续工作放入 Codex 目标模式，建议先把本文件作为第一个目标，而不是直接把整份 H2W 大计划一次性设为目标。

推荐目标名称：

```text
完成 P3-06U-26H2W-0 前端功能真实性审计与前后端契约门禁，并把结果写回总控计划
```

推荐成功率判断：

| 执行方式 | 成功率 | 判断 |
|---|---:|---|
| 整份 H2W 一次性执行 | 45%-55% | 范围太大，容易出现半成品和假功能 |
| H2W 拆阶段执行，但没有本门禁 | 60%-70% | 能推进，但前端承诺仍可能超过后端能力 |
| 先执行 H2W-0，再分阶段执行 H2W | 75%-85% | 最稳，能先消灭假按钮、假流程和文案越界 |

## 2. 本轮必须承认的真实现状

当前 H2W 文档方向正确，但它仍是工程路线，不是可直接执行的细粒度验收规格。

当前多渠道对话台已经比上一版更接近真实客服软件，但仍存在若干高风险点：

1. 页面中存在只有图标或按钮外观、没有真实动作的控件。
2. 某些后端能力已经存在，但前端没有接上。
3. 某些前端文案容易让人误以为已经能真实自动回复、真实转接、真实结束会话或真实发送附件。
4. 有些页面“看起来分页或分区”，但真实数据、真实接口、真实动作还没有完全对齐。
5. 真实外发、真实官方渠道、真实线上回执仍未开放，客户界面不能暗示已经完成。

已知需要重点复查的前端区域：

| 页面 | 当前风险 | 本门禁处理方式 |
|---|---|---|
| 多渠道对话台 | 顶部操作、底部工具按钮可能只是图标 | 每个按钮必须映射真实 handler、真实接口或禁用理由 |
| 知识库运营 | 新增、编辑、导入、发布入口必须真的能走通 | 检查前端动作、后端接口、数据落库、错误提示和回滚 |
| 知识缺口 | 筛选、分页、修复动作必须和后端一致 | 检查服务端筛选、分页和修复闭环 |
| 知识评测 | 不能把检索评测写成完整客服准确率 | 检查文案和指标口径 |
| 运营总览 | 筛选按钮必须和数据口径对齐 | 检查时间、渠道、服务端聚合或本地兜底说明 |
| 渠道接入 | 不能把官方平台允许写成本系统已接通 | 检查授权状态、沙盒状态、真实外发关闭提示 |
| 管理运维 | 三个左侧入口必须是真页面 | 检查运维与告警、自动回复策略、账号安全是否独立落点 |

## 4. 功能真实性矩阵

每一个客户可见按钮、链接、输入框、菜单、筛选、导出、上传、发送、转接、结束、发布、回滚、刷新，都必须进入矩阵。

矩阵字段固定如下：

| 字段 | 说明 |
|---|---|
| 页面 | 例如多渠道对话台、知识库运营、质量复盘 |
| 区域 | 例如顶部操作区、会话列表、底部输入区 |
| 控件名称 | 客户看到的文案或图标含义 |
| 控件类型 | 按钮、链接、输入框、筛选、菜单、上传、导出 |
| 当前可见角色 | 负责人、管理员、坐席、只读账号 |
| 前端文件 | 具体组件文件 |
| 前端动作 | `onClick`、表单提交、路由跳转、状态更新等 |
| API client | 对应前端 API 函数，没有则写无 |
| 后端接口 | 对应后端路由，没有则写无 |
| 数据变化 | 读数据、写数据、导出文件、审计事件、无变化 |
| 权限要求 | 对应后端权限或前端角色限制 |
| 成功反馈 | 页面状态、toast、列表刷新、文件下载、审计记录 |
| 失败反馈 | 权限不足、接口失败、字段缺失、配置缺失 |
| 当前状态 | 真实可用、只读可用、禁用合理、仅前端、仅后端、应隐藏 |
| 处理结论 | 保留、接真实动作、禁用、隐藏、移到后续阶段 |
| 验收证据 | 静态检查、接口测试、浏览器截图、日志摘要 |

当前状态解释：

| 状态 | 允许进入客户版 | 要求 |
|---|---|---|
| 真实可用 | 允许 | 必须有前端动作、后端接口、数据变化和 smoke |
| 只读可用 | 允许 | 必须明确是只读，不暗示可写 |
| 禁用合理 | 允许 | 必须展示清楚禁用原因 |
| 仅前端 | 不允许 | 必须接后端、禁用或隐藏 |
| 仅后端 | 不允许对客户宣传 | 可以保留内部任务，不进入客户主界面 |
| 应隐藏 | 不允许 | 从客户可见页面移除 |

## 5. 总验收门禁

### 5.1 文档门禁

必须通过：

1. 本文件存在，标题为 `P3-06U-26H2W-0 前端功能真实性与前后端契约门禁计划`。
2. `FRONTEND_FUNCTION_REALITY_MATRIX.md` 存在。
3. 矩阵至少覆盖以下页面：
   - 多渠道对话台
   - 知识库运营
   - 知识缺口
   - 知识评测
   - 运营总览
   - 质量复盘
   - 渠道接入
   - 运维与告警
   - 自动回复策略
   - 账号安全
4. 矩阵中不得出现空白的前端文件、处理结论和验收证据。
5. 所有 `仅前端` 和 `应隐藏` 项必须有后续处理状态，不能只记录不处理。

停止门禁：

- 如果矩阵没有覆盖多渠道对话台，停止。
- 如果矩阵只写页面名称、不写控件级明细，停止。
- 如果存在客户可见控件但没有进入矩阵，停止。

### 5.2 静态代码门禁

必须检查：

1. 客户可见 `button` 必须满足以下之一：
   - 有真实 `onClick` 或提交动作。
   - 有真实路由跳转。
   - 有 `disabled` 并显示禁用原因。
   - 是纯展示按钮样式，但语义上不是按钮。
2. 图标按钮必须有可理解的 `aria-label` 或可见文字。
3. 带有以下高风险词的按钮必须有真实动作或明确禁用：
   - 发送
   - 转接
   - 结束
   - 发布
   - 导入
   - 导出
   - 上传
   - 应用
   - 回滚
   - 备份
   - 恢复
   - 同步
   - 自动回复
4. 前端不得出现与当前后端能力不一致的文案：
   - 自动回复中
   - 已自动发送
   - 已接通全渠道
   - 已完成线上准确率
   - 程序包可直接应用
   - 已上传云端
   - 已正式签收
5. 真实外发关闭时，页面不能暗示消息已经真实发到微信、抖音、淘宝、京东、拼多多等平台。

停止门禁：

- 发现客户可见按钮没有动作、没有禁用、没有隐藏，停止。
- 发现“发送 / 转接 / 结束 / 上传 / 应用 / 回滚”等高风险按钮是假动作，停止。
- 发现前端文案超过后端真实能力，停止。
- 发现真实外发关闭但页面写成已自动回复或已发送，停止。

### 5.3 前后端契约门禁

每个写动作必须证明：

1. 前端有 API client。
2. 后端有接口。
3. 后端有权限校验。
4. 后端有数据变化或审计事件。
5. 前端成功后会刷新真实数据。
6. 前端失败后能展示可理解错误。

每个读动作必须证明：

1. 前端请求真实接口，或明确标识本地兜底。
2. 筛选、分页、排序口径和后端一致。
3. 空数据、加载失败、无权限有明确状态。

停止门禁：

- 前端存在写动作，但后端没有接口，停止。
- 后端有接口，但前端按钮没有接入，停止进入下一阶段。
- 写动作没有权限校验，停止。
- 写动作没有审计或数据变化证据，停止。
- 筛选按钮只是改 active 样式，不影响数据口径，也没有兜底说明，停止。

### 5.4 浏览器操作门禁

必须用浏览器 smoke 覆盖：

1. 负责人登录后访问所有目标页面。
2. 多渠道对话台点击会话、搜索、切换全部/我的/转人工。
3. 多渠道对话台顶部操作按钮逐项验证：
   - 已实现的必须产生页面变化、接口请求或审计结果。
   - 未实现的必须禁用或隐藏。
4. 底部输入区验证：
   - 若真实外发关闭，不能让用户误以为已发送到外部平台。
   - 若只是人工接管本地记录，必须写清楚。
5. 知识库运营验证新增、编辑、导入、预检、导入反馈。
6. 知识缺口验证筛选、分页、修复入口。
7. 知识评测验证指标口径，不能写成完整准确率。
8. 渠道接入验证分层、授权状态、真实外发关闭状态。
9. 管理运维三页验证左侧入口对应真实页面。

停止门禁：

- 浏览器里任何客户可见按钮点击后无响应，且没有禁用说明，停止。
- 浏览器只能看到 UI，但无法证明数据变化或接口调用，停止。
- 登录后目标页路由丢失或错误回到非目标页，停止。
- smoke 只截首页、不覆盖关键控件，停止。

### 5.5 数据真实性门禁

必须区分：

1. 真实服务端数据。
2. 本地演示样本。
3. 只读聚合。
4. dry-run。
5. draft-only。
6. 外发关闭。

页面上不能把这些状态混写。

停止门禁：

- 把 dry-run 写成已执行，停止。
- 把 draft-only 写成自动发送，停止。
- 把本地演示样本写成真实客户数据，停止。
- 把检索评测写成完整客服准确率，停止。
- 把本地确认记录写成正式电子签章，停止。

## 6. 多渠道对话台专项门禁

多渠道对话台是本阶段最高优先级。

### 6.1 允许保留的主界面内容

1. 左侧会话列表。
2. 搜索框。
3. 队列筛选：全部、我的、转人工。
4. 客户头像、昵称、渠道、最近消息、等待时长、未读数、小状态。
5. 右侧消息气泡。
6. 系统弱提示。
7. 轻量输入区。

### 6.2 不应堆在主聊天区的内容

1. 大块 AI 决策卡。
2. 大块转人工提醒。
3. 大块系统门禁说明。
4. 大块知识依据说明。
5. 发送失败复盘详情。
6. 模型路由解释。
7. 内部工程状态。

这些信息可以进入详情抽屉、质量复盘、知识运营或运维页面，但不应占据主聊天流。

### 6.3 顶部操作按钮要求

| 按钮 | 当前建议 |
|---|---|
| 历史记录 | 如果没有历史记录抽屉，先隐藏 |
| 设为星标 | 如果后端没有星标字段，先隐藏或禁用 |
| 转接 | 必须有目标人员/团队选择，否则先隐藏 |
| 结束 | 可以接后端 `resolve`，否则先禁用并说明 |

停止门禁：

- 顶部按钮只有图标和文字，没有动作，停止。
- 转接没有目标选择就直接显示为可点击，停止。
- 结束会话没有后端状态变化或审计事件，停止。

### 6.4 底部工具按钮要求

| 按钮 | 当前建议 |
|---|---|
| 表情 | 没有表情选择器时隐藏 |
| 图片 | 没有附件上传后端模型时隐藏 |
| 附件 | 没有文件上传、存储和审计时隐藏 |
| 发送 | 真实外发关闭时，只能用于本地人工接管记录或禁用 |

停止门禁：

- 图片和附件按钮没有上传模型却可点击，停止。
- 发送按钮让用户误以为已发到外部平台，停止。
- 输入内容没有保存、没有审计、没有状态反馈，停止。

## 7. 知识运营专项门禁

知识运营必须服务小微企业客户自己维护知识库。

### 7.1 四层知识入口

页面必须让客户能理解并维护：

1. 业务对象：产品、服务、套餐、门店、活动。
2. 标准问答：问题、标准答案、相近问法、适用对象。
3. 流程政策：售后、退款、发票、物流、保修、预约。
4. 禁用承诺和转人工规则：不能承诺、必须人工确认、投诉赔付。

### 7.2 真实功能要求

| 功能 | 验收 |
|---|---|
| 新增业务对象 | 表单可提交，后端落库，列表刷新 |
| 编辑业务对象 | 修改后能读取到最新值 |
| 新增问答卡 | 能绑定业务对象，导入后能被检索或评测引用 |
| 知识包预检 | 能显示新增、跳过、错误和风险 |
| 知识包导入 | 导入后有批次记录和审计 |
| UI 到服务端持久化 | 负责人从浏览器登录后完成新增业务对象、绑定问答卡、导入知识文档、预检更新包、导入更新包，并由后端 API 读取同一租户数据确认落库 |
| 回滚 | 当前只允许已有后端支持的知识更新包回滚，不得泛化成整库恢复 |
| 评测 | 明确是发布前回归或检索评测，不是完整线上准确率 |

停止门禁：

- “新增业务对象”点不了，停止。
- “编辑知识草稿”不能保存或没有后端数据，停止。
- 导入只改变前端状态、不写后端，停止。
- 评测结果文案超过实际口径，停止。

4. 不新增复杂多租户审批。
