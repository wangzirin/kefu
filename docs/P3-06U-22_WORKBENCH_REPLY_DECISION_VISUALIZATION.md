# P3-06U-22 工作台回复决策可视化

## 定位

本阶段把 P3-06U-20/P3-06U-21 已经落地的 `reply_decisions` 接到接待工作台，让坐席在多平台对话台里直接看到每条会话的后端回复判断。

它解决的是“后端已经判断了，前端看不见”的问题，不打开真实外发，不代表微信、抖音、淘宝、京东、拼多多已经完成自动回复。

## 本轮完成

- 前端 API 类型补齐 `TrustedInboundWorkerItem.reply_decision_id`、`knowledge_gap_id`、`outbox_draft_id`。
- `App.tsx` 新增 `replyDecisionState`、`refreshReplyDecisions()`，正式登录后跟随会话资源刷新最新回复决策。
- 演示工作区新增多平台会话的 `replyDecisions` 样本，覆盖自动回复预备、人工确认、知识缺口、策略阻断、仅生成草稿。
- `ConversationWorkbenchPanel` 按会话聚合最新 `reply_decision`。
- 多平台对话台新增可见的“回复决策”状态条，展示：
  - 回复决策状态
  - 决策原因
  - 业务对象
  - 知识依据
  - 下一步动作
  - 外发边界
- 会话时间线新增“回复决策”事件，让客户入站、后端判断、人审、待发送、失败复盘处在同一条处理线里。
- 左侧会话列表继续收窄，右侧聊天区保留主视觉，避免回到管理卡片堆叠。

## 多平台对话台当前真实水平

当前多平台对话台已经具备“中台接待台”的基本骨架：

- 左侧是会话列表。
- 每条会话展示渠道、客户、等待时间、预览和异常标签。
- 右侧是聊天式消息流、AI 回复预案和人工接管区。
- 同一页可以看到回复决策、知识缺口、人审、待发送和外发关闭边界。

但它还不是完整云朵AI级别的商用平台中心。

仍缺：

- 真实平台账号、店铺、客服账号、授权状态和服务商身份管理。
- 真实平台消息同步，不是演示数据。
- 商品、订单、售后、物流等平台业务对象的真实映射。
- 批量学习商品资料、批量改知识卡、批量配置自动回复规则。
- 平台原生限制展示，例如抖音私信互关限制、商家后台客服权限、淘宝/京东/拼多多服务市场授权状态。
- 实时未读、声音提醒、坐席在线状态和排队转接。
- 真实发送、回执、失败重试和渠道限流闭环。

## 和云朵AI视频参考的对应关系

已经吸收的点：

- 以会话为核心，而不是以工程模块为核心。
- 平台来源必须在会话列表和聊天区可见。
- AI 自动回复是主路径，人工处理只处理异常。
- 需要把知识缺口、回复判断、外发边界直接放到坐席能看到的位置。
- 多平台能力不能只在“渠道配置页”出现，也要进入日常接待界面。

尚未达到的点：

- 视频里的平台账号/店铺感更强，我们现在只是会话层展示渠道。
- 视频里的操作入口更偏客服日常批量运营，我们现在还偏“工程型中台”。
- 视频里的自动回复、商品学习、平台连接看起来是一体化产品流，我们现在后端能力仍是分阶段拼接。
- 视频里很多状态更像真实客户运营数据，我们现在仍主要是 demo 和受控样本。

## 当前边界

- 不真实外发。
- 不绕过平台规则。
- 不做个人号 Hook、群控、模拟点击正式交付。
- 不把 RPA 研究能力包装成正式渠道能力。
- 不把演示多平台样本说成真实接入。

## 验收

已新增静态验收：

```bash
.venv/bin/python scripts/check_p3_06u_22_workbench_reply_decision_visualization.py
```

建议同时运行：

```bash
cd frontend
npm run typecheck
npm run build
```

前端浏览器验收重点：

- 工作台打开后仍是微信式两栏结构。
- 左侧会话列表不能过宽。
- 右侧对话区必须明显大于左侧列表。
- 每个会话顶部能看到“回复决策”状态条。
- 页面必须显示真实外发关闭边界。
- 无横向滚动、无文字溢出、无旧三栏管理视图回潮。
