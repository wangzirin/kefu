## 1. 阶段结论

P3-06T-03 已经把运营总览从“大指标卡堆叠”提升到“经营信号 + 健康环 + 风险组成 + 趋势主图 + 优先动作”的数据指挥舱雏形，但这不等于前端已经成熟。

用户复审后的校准结论是：当前前端不能按 7.6 分理解。按“真实客户能否顺畅使用、前后端逻辑是否严丝合缝、每个页面是否服务实际运营任务”来评价，当前前端更接近 `6.0 / 10`。

这个判断是合理的。原因是当前系统已经有很多能力，但还存在三个根本问题：

1. 前端仍有“功能模块列表”的影子，尚未完全变成客户日常运营路径。
2. 部分页面展示了状态和数据，但还没有把后端契约、权限、动作、失败态和下一步处理路径完全对齐。
3. 视觉已经比早期更像中台，但实用性、信息架构、角色化体验和真实登录端到端验证还没有达到可正式对外商用的程度。

因此，P3-06T-04“信息架构收口”不再作为一个孤立小阶段推进，而是被吸收到本阶段：

> **P3-06U：前后端契约对齐 + 实用型前端产品化优化。**

本阶段不是继续单纯做视觉精修，也不是继续堆新页面，而是把前端重新拉回三个问题：

- 后端真实有什么能力、什么权限、什么状态、什么边界？
- 客户和坐席每天到底要完成哪些任务？
- 页面能不能让人少思考、少迷路、少误操作，并在出问题时知道怎么修？

### 3.2 为什么不能按 7 分以上计算

当前前端存在“看起来已经很完整”的错觉。真实商用评估不能只看页面数量、图表数量和按钮数量，还要看下面这些问题是否闭环：

- 页面上每一个动作是否都有明确后端 API、权限、状态机和审计记录。
- 后端每一个核心状态是否都有清楚的前端解释、操作入口和异常处理。
- 用户是否能从首页自然进入最该处理的任务，而不是自己在左侧菜单里找功能。
- 坐席、管理员、只读管理层看到的页面是否符合岗位，而不是同一套页面简单隐藏按钮。
- 演示数据、本地汇总、真实服务端聚合是否明确区分。
- 失败、空状态、无权限、配置缺失、渠道未验证、模型不可用时是否有清楚提示和下一步。
- 客户实际运营时是否能完成“发现问题 -> 处理会话 -> 修复知识 -> 复盘效果”的闭环。

这些问题没有完全解决前，前端不能按成熟商用产品评分。

## 4. 本阶段两个最高原则

### 4.1 原则一：前端和后端必须完美对齐

这里的“对齐”不是指页面能请求接口，也不是指功能列表名称一致，而是四层对齐。

| 层级 | 必须达到的标准 | 不合格表现 |
| --- | --- | --- |
| 功能对齐 | 前端每个页面、按钮、表格、筛选器都有对应后端契约 | 页面展示了后端没有的能力，或后端已有能力前端无入口 |
| 权限对齐 | 每个动作按后端 RBAC 权限显示、禁用或解释 | 前端按钮可见但后端 403，或低权限用户看到不该看的经营数据 |
| 状态对齐 | 会话、审核、outbox、渠道、知识、工单状态在前端有统一解释 | 后端状态很多，前端只显示“处理中/完成”，导致无法运营 |
| 逻辑对齐 | 页面操作顺序符合后端状态机和业务审计链 | 前端允许跳步、重复操作、误以为审核通过就是已发送 |

本阶段的硬规则：

- 后端契约是源头，前端不能凭想象发明生产状态。
- 前端不能把样本、演示、本地汇总包装成真实后台能力。
- 每个按钮都必须回答：谁能点、点了调什么 API、成功后进入什么状态、失败后怎么处理、是否写审计。
- 每个后端核心状态都必须回答：谁能看、在哪里看、下一步能做什么、不能做时为什么。
- 真实外发、真实渠道和真实客户数据永远不能被前端文案暗示为已打开。

### 4.2 原则二：前端必须从实用性出发

中台前端不是作品集页面，也不是功能展览馆。它要服务真实岗位。

本系统的前端首先服务四类人：

| 角色 | 真实任务 | 前端必须帮他节省什么 |
| --- | --- | --- |
| 坐席 | 看客户问题、判断 AI 草稿、改写、确认、转人工、记录备注 | 少切页、少找证据、少误发、少漏单 |
| 运营管理员 | 看今天压力、风险、知识缺口、渠道异常、坐席负载 | 快速发现问题，能一键进入修复路径 |
| 知识库负责人 | 看错因、补知识、发布、回归、回滚 | 知道为什么错、改哪里、改完有没有变好 |
| 老板/管理层 | 看 AI 是否减负、转人工是否上升、客户投诉是否变多、渠道是否健康 | 一眼判断系统是否有价值和风险 |

本阶段前端设计必须服从这些任务，不再追求“页面多、模块全”。凡是不服务真实任务的页面和信息，都要合并、降级或隐藏到二级入口。

## 6. 前后端契约对齐模型

本阶段先建立一张总矩阵，矩阵不完成，不继续做大面积视觉精修。

### 6.1 页面契约矩阵字段

每个页面都要登记以下字段：

| 字段 | 说明 |
| --- | --- |
| 页面名称 | 用户看到的页面名称，不使用阶段编号命名 |
| 路由 | hash 路由或实际前端路径 |
| 主要用户 | owner/admin/agent/viewer 哪些角色会用 |
| 业务目的 | 这个页面帮用户完成什么任务 |
| 后端 API | 页面读取、写入、刷新、导出的接口 |
| 权限 | 读取权限、写入权限、审批权限、导出权限 |
| 数据状态 | 真实服务端、演示样本、本地汇总、离线夹具 |
| 关键动作 | 页面上可触发的业务动作 |
| 后端状态机 | 动作前后会改变哪些状态 |
| 审计 | 是否写会话事件、租户审计、发送尝试、worker 记录 |
| 空态 | 没有数据时怎么显示 |
| 错态 | API 失败、权限不足、配置缺失时怎么显示 |
| 禁用态 | 不能操作时是否解释原因 |
| 下一步 | 用户看完这个页面后最自然的下一步 |

## 7. 信息架构重排方向

现阶段不建议继续把左侧导航做成“总览、会话、审核、待发送、联系人、线索、知识、质量、渠道、运维”的平铺清单。成熟客服中台应该围绕任务闭环。

推荐将工作区收束为 6 个一级域：

| 一级域 | 面向角色 | 包含内容 | 设计目标 |
| --- | --- | --- | --- |
| 运营总览 | 管理员、老板、运营 | 经营信号、风险、趋势、优先动作 | 一眼知道今天有没有问题 |
| 接待工作台 | 坐席、管理员 | 实时会话、用户列表、消息流、AI 草稿、人工接管、待发送 | 像真实客服 IM 一样工作 |
| 质量复盘 | 运营、知识负责人 | 错因 BI、低置信、投诉、转人工上升、知识缺口 | 找到为什么答错 |
| 知识运营 | 知识负责人、管理员 | 知识卡片、文档、缺口修复、发布、回归、回滚 | 把问题修回知识库 |
| 渠道接入 | 管理员、交付、售后 | 官网、企业微信、公众号、电商平台连接器状态 | 知道渠道接到哪一步 |
| 管理运维 | 管理员、内部售后 | 账号权限、队列、worker、告警、指标、审计 | 保证系统可维护 |

这不是简单改菜单名称。每个一级域必须有明确任务。

例如“质量复盘”不是一张统计表，而是：

1. 哪些问题 AI 没答好。
2. 是知识缺失、检索不准、模型乱说、渠道上下文不够，还是客户问题超范围。
3. 应该补哪条知识、改哪个 FAQ、加哪个禁答策略。
4. 改完后用哪些题回归。
5. 回归结果是否改善。

## 8. 阶段施工计划

### P3-06U-01 前后端契约与页面路径盘点

目标：先把系统现在真实有什么、页面展示什么、权限允许什么、用户能做什么彻底盘清。

施工内容：

- 建立 `docs/P3-06U-01_FRONTEND_BACKEND_CONTRACT_MATRIX.md`。
- 逐页列出路由、API、权限、数据模式、关键动作、后端状态、空态、错态和下一步。
- 查找三类问题：
  - 孤儿页面：前端展示了没有后端契约或没有真实业务意义的内容。
  - 孤儿能力：后端已有能力但前端没有自然入口。
  - 错位能力：前端动作顺序、文案或权限和后端状态机不一致。

预计时间：0.5 - 1 天。

验收标准：

- 至少覆盖 10 个核心工作区。
- 每个按钮级关键动作都有 API/权限/状态说明。
- 明确列出第一批必须修的 8 - 12 个前后端错位点。
- 不改业务代码，只出矩阵和结论。

### P3-06U-02 角色化任务路径重排

目标：把系统从“功能列表”改成“岗位任务路径”。

施工内容：

- 为 owner、admin、agent、viewer 设计默认首页和主要路径。
- 重新定义每类用户登录后的首要任务：
  - 坐席默认进入接待工作台。
  - 管理员默认进入运营总览。
  - 知识负责人优先看质量复盘和知识缺口。
  - 只读管理层看经营总览和质量趋势，不进入写动作。
- 更新导航文案、排序和入口层级。
- 明确哪些内部工程页面放进二级或管理运维，不在客户主流程前置。

预计时间：1 天。

验收标准：

- 每个角色有 3 - 5 条核心任务路径。
- 左侧导航不再像工程模块清单。
- 低权限角色不会看到无意义的禁用按钮墙。
- 用户从首页进入高风险审核、待发送、知识缺口、渠道异常的路径清楚。

### P3-06U-03 接待工作台实用性重构

目标：把对话台做得更接近真实客服系统，而不是普通详情页。

推荐布局：

| 区域 | 内容 |
| --- | --- |
| 左侧主导航 | 保持全局工作区 |
| 中左会话列表 | 客户头像、渠道标记、最后消息、风险状态、未读、等待时长、AI/人工状态 |
| 中间消息区 | 客户消息、AI 草稿、坐席回复、系统事件、引用证据提示 |
| 右侧上下文 | 客户画像、历史咨询、开放工单、线索、知识引用、风险和建议动作 |
| 底部动作区 | 改写草稿、转人工、批准、确认待发送、内部备注 |

施工内容：

- 强化会话列表和消息区的视觉层级。
- 明确渠道来源：官网、企业微信、公众号、抖音、小红书、淘宝等都只能在官方接入状态下展示真实状态；未接入时展示为计划/样本。
- 把 AI 草稿、引用证据、风险原因和人工动作放在坐席可直接操作的位置。
- 对“审核通过”和“真实发送”做强区分，避免误解。

预计时间：1.5 - 2 天。

验收标准：

- 坐席能在一个页面完成看问题、看证据、改写、批准、确认待发送。
- 会话列表支持清晰区分渠道、状态、风险和等待时长。
- 页面不依赖长段说明文字解释怎么用。
- 真实外发关闭时，按钮和文案必须明确说明只进入 outbox 或发送计划。

### P3-06U-04 运营总览到处理路径打通

目标：运营总览不是漂亮首页，而是进入工作的入口。

施工内容：

- 首页“优先动作”必须能跳到对应过滤后的工作区：
  - 高风险会话 -> 人工审核/接待工作台风险队列。
  - 待发送草稿 -> 待发送工作区。
  - 知识缺口 -> 质量复盘或知识运营。
  - 渠道异常 -> 渠道接入中心。
- 每个跳转携带上下文：时间范围、渠道、风险类型或状态过滤。
- 首页指标和目标页数据口径要一致，不能首页显示 10 个风险，目标页过滤后没有对应项且不解释。

预计时间：1 天。

验收标准：

- 首页至少 4 类优先动作可点击进入目标工作区。
- 目标工作区能保留或展示来自首页的筛选条件。
- 无数据时显示“本时间窗口暂无对应任务”，而不是空白。

### P3-06U-05 真实登录与角色端到端前端 smoke

目标：证明前端不是只在开发演示身份下好看，而是真实角色能用。

施工内容：

- 为 owner/admin/agent/viewer 建立可重复浏览器 smoke。
- 验证每个角色：
  - 登录成功。
  - 默认落地页正确。
  - 导航可见性正确。
  - 核心页面 API 请求不出现非预期 403。
  - 禁用动作解释清楚。
  - 退出后 token 清理。
- 截图保存到 `output/p3_06u_role_smoke/`。

预计时间：1 天。

验收标准：

- owner/admin/agent/viewer 四类角色至少各 1 条 smoke 路径。
- 所有 smoke 都必须使用真实登录 token，不使用开发演示入口替代。
- 403 只能出现在预期被禁权限，且前端有解释。

### P3-06U-06 质量复盘 BI 与知识修复闭环

目标：把质量复盘从“报告展示”变成“错因定位和知识修复工作台”。

推荐模块：

| 模块 | 说明 |
| --- | --- |
| 错因分布 | 知识缺失、检索失败、模型越界、渠道上下文不足、客户需求超范围 |
| 低置信趋势 | 最近 7/30 天低置信、转人工、投诉、改写率 |
| 知识缺口排行 | 哪些主题最需要补知识 |
| 修复建议 | 补 FAQ、改话术、加禁答、补文档、调整路由 |
| 回归结果 | 修改后题库是否改善 |

施工内容：

- 质量复盘页面用 BI 图表表达错因和优先级。
- 每个错因能生成或关联知识缺口。
- 知识缺口能进入知识运营草稿。
- 知识发布后可回到题库回归。

预计时间：1.5 - 2 天。

验收标准：

- 用户能从质量复盘找到错因，而不是只看“准确率下降”。
- 至少三类错因能产生对应修复路径。
- 页面展示回归前后变化，哪怕第一片先用已有评测结果。

### P3-06U-07 知识运营台产品化

目标：让客户能维护知识库，而不是只能让工程师改数据。

施工内容：

- 知识卡片、文档、缺口、草稿、发布、回滚重新组织为一条工作流。
- 知识编辑页面要清楚显示：
  - 适用问题。
  - 标准答案。
  - 禁止承诺。
  - 引用来源。
  - 生效范围。
  - 版本和审核状态。
- 引入发布前回归门禁的清楚解释：哪些题会变好，哪些可能变差。

预计时间：1.5 天。

验收标准：

- 客户能从“错因”进入“补知识”再进入“发布前回归”。
- 发布、回滚、失效、审核状态都有清楚前端状态。
- 没有客户看不懂的工程字段前置。

### P3-06U-08 渠道连接器中心实用化

目标：把渠道接入从文档和脚本变成客户/交付都能看懂的状态中心。

施工内容：

- 为每个渠道展示状态：
  - 未配置。
  - 已创建客服账号。
  - 已绑定接待人员。
  - 已获取链接/二维码。
  - 回调 URL 待配置。
  - URL 验证通过。
  - 已收到入站消息。
  - 已生成 AI 草稿。
  - 已进入人工审核。
  - 白名单发送测试通过。
- 第一优先级：企业微信/微信客服。
- 公众号、抖音、小红书、淘宝/天猫、京东、拼多多只展示官方授权前置条件和未接入状态，不做外挂暗示。

预计时间：1 - 1.5 天。

验收标准：

- 用户知道自己卡在哪一步。
- 页面能展示公网 HTTPS 回调、Token、EncodingAESKey 的配置位置和状态，但不泄露密钥明文。
- 未备案、未公网、未可信 IP、未官方授权等阻塞条件有清楚解释。

### P3-06U-09 前端状态体系统一

- 统一数据口径标签样式。
- 统一“不自动真实外发”的提示位置和文案。

预计时间：1 天。

验收标准：

- 核心页面不再出现空白表格、无解释禁用按钮或工程调试口吻。
- 用户能判断当前是演示、本地汇总还是真实数据。
