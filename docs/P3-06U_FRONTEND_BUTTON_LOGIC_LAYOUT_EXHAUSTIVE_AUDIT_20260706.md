## 结论

当前前端已经具备“可演示、可受控试跑”的基本形态，但还不能说是客户拿到后顺手可用的成品。最大问题不是单纯美观，而是若干关键路径的业务语义没有闭合：工作台不能让坐席清楚地替代 AI 回复，试点准备页对非工程人员不直观，联系人和线索模块在真实渠道数据未接入前显得偏重，部分页面存在裁切、禁用按钮原因不清和客户可见工程词。

本轮自动化审计覆盖 13 个桌面页面、281 个可见控件，实际安全点击 49 个控件，记录 33 个禁用控件、2 个风险跳过控件，生成 26 张前后截图。危险动作没有强行点击，避免误改数据或触发不可逆流程。

## 用户问题直接回答

### 试点准备是什么意思

现在的“试点准备”实际含义是“本地试运行交付前检查页”。它检查本地环境、负责人、知识资料、复测确认、质量报告、诊断包、备份更新和交付档案是否齐备。

问题是这个名字不够直观。客户或非工程使用者看到“试点准备”，不知道自己应该点什么、上传什么、何时算可以开始。建议改名为“本地试运行准备”或“交付准备”，首屏只保留三个动作：

1. 导入资料
2. 运行复测
3. 导出交付档案

其它证据、阻断项、安装器候选、边界说明折叠到二级区域。

### 保存接管回复是什么意思

现在“保存接管回复”不是“发送给客户”。它只是转人工后，把坐席改写的回复保存成本地接管记录或草稿。它在没有关联人工审核任务时还会被禁用，所以用户想替代 AI 回复时会发现发不了。

这是 P0 级产品逻辑问题。工作台应该拆成三种明确状态：

1. 本地试跑：按钮叫“发送到本地会话”，发送后立刻在右侧消息流出现一条坐席消息，但不外发平台。
2. 未接真实渠道：按钮叫“保存人工回复草稿”，旁边明确写“未外发，需要人工复制到平台或等待官方渠道接入”。
3. 官方渠道已通过白名单和回执：按钮才允许叫“发送给客户”或“加入发送队列”。

当前不能继续用“保存接管回复”承担这三种语义。

### 联系人中心和线索跟进有没有实际价值

有价值，但不是当前试跑阶段的主价值。

联系人中心的价值在于沉淀客户身份、渠道来源、最近问题和服务历史；线索跟进的价值在于把报价、试点、部署、采购类咨询转成可跟进商机。问题是我们现在还没有真实渠道数据和 CRM 同步，所以它们看起来像一个未接数据的销售后台。

建议短期降级为“客户资料与商机记录”轻入口：

- 联系人只保留客户基础信息、最近会话、标签和备注。
- 线索只保留来源会话、阶段、下一步动作和负责人。
- 不做复杂画像、预算、批量销售动作和重 CRM 化页面。
- 在工作台会话右侧或抽屉里能创建线索，比单独大页面更符合小微企业试跑。

### 多渠道对话台现在能不能用

能看会话，但还不好用，不能算完整客服 IM 工作台。

当前优点是：左侧有会话列表，右侧有消息流和输入区，能显示渠道和转人工状态。

当前问题是：真实发送路径没有闭合，输入区默认禁用，发送按钮语义不清，AI 草稿、人工接管、真实外发关闭之间的关系不够直观。它更像“会话查看器”，还不是“坐席接待台”。

## 高优先级问题清单

### P0 接待工作台缺少真正人工回复路径

证据：`#live` 中“保存接管回复”按钮禁用，且该按钮不是用户理解的发送动作。

影响：客户或坐席无法完成“我替 AI 回复一句话并看到这句话进入会话”的基本动作。

整改：

- 增加本地试跑消息发送：写入本地 conversation message，不触发外部平台。
- 保留人工草稿：真实渠道未接入时不允许误写为已发送。
- `onConfirmOutboxDraft` 不能只传入组件而不暴露可用路径。
- 文案改为“发送到本地会话 / 保存人工草稿 / 加入发送队列”三态。

### P0 UI 对真实外发和已处理状态存在软误导

证据：代码中 `auto_reply_ready` 映射为“已处理”，会话列表显示“在线接待”，渠道身份可手工登记授权健康，但后端真实外发仍关闭。

影响：客户可能误以为系统已经在平台里自动回复。

整改：

- `auto_reply_ready` 改成“AI 已生成建议”或“可自动草稿”。
- 未接真实渠道前不显示“已处理”，改为“本地已记录”。
- 渠道账号状态改成“人工登记：待官方验证”，不能像平台验证通过。

### P1 试点准备页概念和信息层级不直观

证据：首屏同时出现今日任务路径、阻断项、证据卡片、真实外发关闭、封包候选等信息。

影响：客户不知道第一步要做什么，用户会把工程封版页当成业务功能页。

整改：

- 改名为“本地试运行准备”。
- 首屏只放当前状态、下一步动作、三项主任务。
- 工程证据、边界、档案列表折叠。
- 删除或隐藏客户看不懂的内部阶段词。

### P1 联系人中心和线索跟进存在横向裁切

证据：自动化审计抓到 `profile-list-column` 与 `lead-list-column` 横向裁切，`scrollWidth > clientWidth`。

影响：用户已经肉眼看到左侧列表被割掉，筛选和列表内容不完整。

整改：

- 列表列宽从固定/过宽网格改成真正可压缩布局。
- 列表项减少字段，长文本统一截断。
- 过滤器不要横向塞满，改成顶部一行主要筛选 + 更多筛选抽屉。
- 线索详情放右侧或抽屉，列表只保留姓名、来源、阶段、下一步。

### P1 联系人和线索模块当前价值过重

证据：当前联系人、线索有 26 个以上客户可见控件，但真实渠道和 CRM 数据还不完整。

影响：小微企业试跑时会觉得复杂，且很多动作没有真实业务闭环支撑。

整改：

- 短期合并为“客户与线索”。
- 仅保留从会话生成客户、从会话生成线索、更新阶段、写备注。
- 复杂画像、预算、批量跟进、销售系统感内容后置。

### P1 运维与告警仍有客户可见工程词

证据：`#ops` 出现 `outbox`。

影响：客户不懂工程词，且容易把待发送队列理解成真实外发能力。

整改：

- `outbox` 改为“发送草稿队列”或“外发演练队列”。
- 工程字段只进入诊断包，不出现在客户页面。

### P1 多渠道对话台仍不像完整客服 IM

证据：截图显示右侧消息区空白比例很高，只有一条客户消息和一段建议回复，底部按钮禁用。

影响：不像真实抖音、微信、淘宝客服后台那样可以自然接待、查看历史、快速回复。

整改：

- 右侧只保留消息流、输入框、发送按钮、必要状态。
- AI 建议以“建议回复”卡片贴近输入区，可一键填入输入框。
- 转人工只作为左侧会话状态，不单独堆很多说明。
- 消息流展示方向、头像、时间、发送状态、引用来源。

## 中低优先级问题

### P2 多页禁用按钮缺少明确原因

影响页面：试点准备、接待工作台、联系人中心、线索跟进、知识库运营、知识缺口、知识评测、账号与本地维护。

整改：每个禁用按钮必须满足其一：

- 直接隐藏。
- 显示禁用原因。
- 提供下一步动作。

### P2 试点准备存在无标签图标按钮

整改：补 `aria-label`，或改成带文字按钮。

### P2 质量复盘控件数量过多

证据：`#quality` 可见控件 60 个，但实际安全点击只有 1 个。

整改：质量复盘应聚焦错因、知识修复、转人工边界和报告导出，减少不可立即操作的证据型按钮。

### P2 知识运营三页仍偏证据墙

整改：知识库运营、知识缺口、知识评测分别压成明确动作：

- 知识库运营：导入、预检、发布。
- 知识缺口：查看缺口、补知识、进入回归。
- 知识评测：选择题库、运行复测、查看报告。

## 子智能体交叉结论

### Agent A：产品逻辑审查

- 外发闭环在主工作台断开。
- “转人工”语义过载，混入低置信、未分配、SLA、知识缺口、发送失败等多种含义。
- 试点准备太像实施后台，不像客户可执行流程。
- 联系人和线索应在真实渠道数据沉淀后增强，不应抢主链路。

### Agent B：布局与交互审查

- 联系人和线索列表存在横向裁切。
- 多渠道对话台存在多层滚动风险，应固定为左侧列表滚动、右侧消息流滚动、输入框固定。
- 线索跟进列表项太重，应该改成列表 + 详情，不要每条塞太多字段和按钮。
- 空态、错误态和禁用态不一致。

### Agent C：真实能力边界审查

- 当前不能把 AI 草稿或本地记录写成真实平台已回复。
- 渠道授权状态如果只是人工登记，不能展示成平台已验证。
- 待发送草稿和发送队列现在是外发演练能力，不是正式发送能力。
- 质量确认和线下确认不能被误读成正式客户验收。

## 下一轮修复建议

### 第一优先级：工作台回复链路救火

目标：让用户在本地试跑里真的能回复一条消息。

验收：

- 选中一条转人工会话。
- 输入人工回复。
- 点击“发送到本地会话”。
- 消息流出现坐席气泡。
- 页面明确显示“未外发平台”。
- 如果真实渠道未接入，不出现“发送给客户”。

### 第二优先级：试点准备改名和首屏收束

目标：让客户知道这页是干什么、下一步做什么。

验收：

- 页面名改为“本地试运行准备”。
- 首屏只显示准备状态、缺什么、下一步动作。
- 工程证据折叠。
- 不出现 H2W、P3、dry-run、outbox、provider、sandbox 等工程词。

### 第三优先级：客户和线索减重

目标：从未接 CRM 的销售后台，改成客服衍生的客户记录。

验收：

- 取消横向裁切。
- 列表项一屏可扫。
- 线索详情从会话创建或右侧抽屉进入。
- 没有真实数据时显示清楚空态，不堆复杂筛选。

### 第四优先级：渠道和运维边界文案清理

目标：客户不会误以为已经接通真实平台自动回复。

验收：

- 渠道页只写“配置准备、官方条件、当前缺口、负责人”。
- 人工登记状态必须标注“待官方验证”。
- 运维页不出现 `outbox` 等工程词。

### 第五优先级：二次逐按钮回归

目标：修完后重新跑同一脚本，确保 P0/P1 清零。

验收：

- `scripts/audit_p3_06u_frontend_button_logic_layout.mjs` 重新运行。
- P0 为 0。
- P1 为 0 或只剩明确接受的非阻断项。
- 关键页面截图重新归档。
