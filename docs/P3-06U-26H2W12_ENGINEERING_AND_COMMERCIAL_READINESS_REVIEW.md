# H2W-12 工程推进与商用成熟度评估

日期：2026-07-04

## 1. 结论先行

当前系统已经进入“受控本地试点可用”的后半段，但还没有达到“可以公开面向市场承诺全渠道自动回复”的成熟商用状态。

| 评估口径 | 当前评分 | 结论 |
|---|---:|---|
| H2W 网状工程主线完成度 | 79/100 | 主体工程骨架、知识维护、质量报告、本地维护和事实账本已成型，剩余是客户真实证据与生产化硬门槛 |
| 中小企业本地受控试点可用度 | 76/100 | 可以做共创客户、试点客户、内部演示和低风险本地试用 |
| 完整商用客服中台成熟度 | 64/100 | 不建议公开大规模售卖为“成熟全渠道智能客服系统” |
| 全渠道自动回复商用成熟度 | 38/100 | 真实官方渠道、回执、失败重试、审计和白名单未闭环，不能承诺 |
| 前端产品化成熟度 | 70/100 | 主界面已明显改善，但仍需逐页真实工作流 QA 和客户语言打磨 |
| 后端事实链成熟度 | 74/100 | 事实账、质量、运维已有基础，但生产检索、队列、高并发和真实渠道仍不足 |

一句话判断：可以进入“受控试点市场”，不可以进入“公开规模化商用市场”。

## 2. 市场基线对照

当前主流智能客服产品的公开能力叙事已经不只是 FAQ，而是“AI agent + 知识库 + 坐席协作 + 多渠道 + 质量/分析 + 工作流动作”。这意味着我们进入市场时，不能把页面好看或本地演练通过当成成熟商用。

参考来源：

- [Intercom Fin AI Agent](https://www.intercom.com/help/en/articles/7120684-fin-ai-agent-explained)：强调 AI agent 处理复杂问题、跨渠道、与 helpdesk 协作。
- [Zendesk AI](https://www.zendesk.com/service/ai/)：公开能力包含 AI agents、copilot、knowledge base、ticketing、QA、reporting。
- [Freshworks Freddy AI Agent](https://www.freshworks.com/freshdesk/omni/freddy-ai-agent/)：强调 autonomous support、持续优化和服务团队协同。
- [Salesforce Agentforce](https://www.salesforce.com/agentforce/)：强调基于业务知识执行任务、支持客户与员工服务。
- [美洽 AI 客服](https://www.meiqia.com/chatbot)：公开强调 7x24 自动处理常见问题、人机协同、知识库。
- [Udesk 全渠道客服系统](https://www.udesk.cn/feature_udesk.html)：公开强调多渠道、工单、呼叫中心、知识库和数据报表。
- [智齿科技官网](https://www.zhichi.com/)：公开强调全渠道智能客服、客服机器人、人工在线客服、呼叫中心和工单系统。

市场基线带来的硬要求：

1. 真实渠道不能只停在说明页，至少要有一个官方 sandbox 可复现。
2. 知识库不能只会导入，还要能评测、修复、发布、回归和追溯引用。
3. AI 回复不能只看检索命中，还要看最终答案事实性、禁用承诺、转人工正确和线上回执。
4. 客户本地交付不能只给源码或页面，要有首任负责人、备份、诊断、更新、回滚和审计。
5. 前端不能像工程控制台，必须服务真实客服工作流。

### 3.1 已经完成的硬能力

| 模块 | 完成情况 | 证据 |
|---|---|---|

| 多渠道对话台 | 已收束为桌面客服 IM 形态，主屏只保留会话列表、消息流和转人工状态 | H2W0、H2V、真实性矩阵 |

| 质量报告 | 已有最终答案样本、人工标签、质量报告、导出、签收记录、本地演练证据汇入 | H2N 到 H2U，H2W-11J/K |

### 3.2 已经完成但还不能当成商用完成的能力

| 能力 | 当前真实状态 | 不能怎么宣传 |
|---|---|---|
| 客户标准答案确认 | H2W-11M 已生成返回模板，客户返回文件不存在 | 不能写“客户已确认答案” |
| 质量准确率 | 本地确定性 rehearsal 指标好，但不是线上模型和真实渠道数据 | 不能写“线上准确率已验证” |

| 本地更新 | 知识/策略包能应用，程序包仍是 dry-run | 不能写“无人值守自动升级” |

## 5. 前端是否完全处理好

没有完全处理好。

已经明显改善的部分：

1. 左侧导航已经从工程阶段路线收束为客服工作台结构。
2. 多渠道对话台已经从 AI 决策大面板改为更像客服 IM 的工作区。
3. 知识运营三页已经分叉，知识库运营、知识缺口、知识评测职责更清楚。
4. 运维与告警、模型路由、账号安全已经拆为真实页面。
5. 大量无动作按钮已隐藏、禁用或写入真实性矩阵。

仍需继续处理的部分：

1. 多渠道对话台还需要真实消息流、历史抽屉、转接、结束、附件等后续片。
2. 部分页面仍偏长，尤其渠道接入和运维页面，要继续分层。
3. 前端客户语言仍需持续清理，避免“门禁、回执、dry-run、provider”等工程词漏出。
4. 总览 BI 可以用于演示，但离成熟产品的数据洞察仍差缓存、真实数据和钻取。
5. 页面已经不做移动端，但桌面 1180、1280、1440 三档还要持续做视觉验收。

前端当前可以给试点客户看，但建议定位为“受控试点版”，不要包装成成熟 SaaS 工作台。

## 6. 功能性是否足够商用

要分两种商用：

### 可以做的商用

适合做：

- 首批共创客户。
- 本地部署试点。
- 官网客服或单入口低风险试点。
- 知识库治理、质量复盘、本地维护能力展示。
- 收费但边界清晰的试点项目。

前提：

1. 合同或方案写清“真实外发关闭或单渠道授权后开放”。
2. 不承诺全渠道自动回复。
3. 不承诺线上准确率已达某个比例。
4. 客户先确认标准答案和禁用承诺。
5. 先用脱敏题库和白名单环境试点。

### 不适合做的商用

暂不适合：

- 公开宣传“成熟全渠道智能客服”。
- 承诺企业微信、公众号、抖音、淘宝、京东、拼多多都能自动回复。
- 承诺无人值守高并发客服。
- 承诺生产级 RAG 和正式线上准确率。
- 承诺客户安装后自动联网诊断、自动更新、自动恢复。

## 7. 剩余硬缺口

| 优先级 | 缺口 | 为什么关键 | 建议阶段 |
|---|---|---|---|
| P0 | 客户真实确认标准答案 | 没有客户确认，就没有准确率口径 | H2W-11N |
| P0 | 真实 50-100 条脱敏题库 | 当前样本仍是受控演练，不是客户真实业务 | H2W-11O |
| P0 | 线上回执与失败复盘接真实渠道 | 没有真实回执，就不能证明线上回复闭环 | H2W-10A/11P |
| P1 | 生产级检索证据 | 当前检索还不是完整生产 RAG | H2W-7D |
| P1 | 前端逐页客户可用性最终 QA | 防止隐藏假按钮、重复页面和不懂下一步 | H2W-FE2 |
| P1 | 本地安装包和程序更新器 | 客户本地交付不能只靠源码和人工操作 | H2W-8C |
| P2 | 高并发与队列压测 | 商用后并发、重试、限流会成为稳定性问题 | H2W-9 |
| P2 | 安全审计与权限硬化 | 本地部署也需要凭据、审计、备份、权限安全 | H2W-Sec |
| P2 | 正式报告签收 | 当前本地确认不是电子签章或合同级签收 | H2W-11Q |

## 8. 下一阶段推进方案

### 第一组：不要等渠道，先把准确率口径做实

1. H2W-11N：客户确认结果导入实战。
   - 输入：客户或业务负责人返回的 H2W-11M 模板。
   - 输出：确认通过、修订请求、拒绝项和下一版标准答案包。
   - 停止：没有客户返回文件不能继续伪造。

2. H2W-11O：真实 50-100 条脱敏题库导入。
   - 输入：客户真实问题、标准答案、必含词、禁用词、引用来源、转人工规则。
   - 输出：真实题库评测集和风险分布。
   - 停止：样本少于 50 条时不能写成正式准确率验收。

3. H2W-11P：最终答案抽样与人工事实标签。
   - 输入：系统生成的最终客服回复样本。
   - 输出：事实正确、引用充分、禁用承诺、转人工正确。
   - 停止：只看检索命中不算。

### 第二组：继续把前端打成真实客服工作台

1. H2W-FE2：逐页按钮点击与工作流复审。
   - 覆盖总览、工作台、知识、质量、渠道、运维、账号。
   - 每个按钮必须有真实动作、禁用说明或隐藏。

2. H2W-FE3：多渠道对话台第二片。
   - 增加真实会话详情加载。
   - 做历史记录抽屉。
   - 转接、结束、附件仍保持隐藏，直到后端闭环。

3. H2W-FE4：渠道接入页减噪。
   - 第一屏只保留当前客户要做的官方配置动作。
   - 其它平台说明移入分层详情。

### 第三组：一个真实官方渠道沙箱

建议只选一个：企业微信/微信客服。

条件：

- 客户或我们自己的企业主体可配置公网 HTTPS。
- 有可信 IP、Token、EncodingAESKey。
- 有测试白名单。
- 后端验签、解密、幂等、入站消息、草稿、回执全部走审计。

停止：

- 没有官方授权不做真实外发。
- 没有回执不写已接通。

### 第四组：生产化底座

1. H2W-7D：pgvector/混合检索/重排/引用版本证据。
2. H2W-8C：本地安装包、版本检查、程序更新器、服务重启和健康检查。
3. H2W-9：队列、worker、限流、失败重试、压测。
4. H2W-Sec：secrets、权限、审计、备份恢复和日志脱敏安全审查。

## 9. 推荐销售口径

当前可以这样说：

> 我们可以为中小企业提供一套本地化部署的 AI 客服试点系统，先支持知识库维护、自动回复策略、人工接管、质量复盘、报告留档和本地运维。真实微信、企微、电商平台自动回复需要根据客户官方账号授权逐步接入，先从一个渠道白名单试点开始。

当前不能这样说：

> 我们已经完成全渠道自动回复，可以直接接入微信、抖音、淘宝、京东、拼多多并无人值守自动回复。

## 10. 当前最终判断

如果目标是“拿出去找第一批共创客户、做低风险本地试点”，当前已经接近可用，但仍建议先完成 H2W-11N/O/P 和 H2W-FE2。

如果目标是“作为成熟商用客服公开进入市场”，当前还不够。最短路径不是继续加页面，而是补五件硬证据：

1. 客户真实确认的标准答案。
2. 真实 50-100 条脱敏题库。
3. 一个官方渠道沙箱的真实入站、草稿、回执闭环。
4. 生产级检索与引用证据。
5. 前端逐页真实工作流验收。
