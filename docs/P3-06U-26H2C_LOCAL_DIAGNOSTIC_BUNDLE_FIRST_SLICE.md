# P3-06U-26H2C 本地诊断包生成第一片

## 1. 阶段定位

本片解决客户本地应用交付后的第一个运维闭环问题：客户现场出现知识命中下降、回复质量异常、渠道状态异常或后台任务异常时，客户管理员可以在本地工作台生成一份只读诊断包，再按授权流程发给我方排查。

本片不是远程控制，不是自动上传，也不是程序更新器。它只做“本地生成、客户可见、客户主动发送”的第一片能力。

## 2. 已完成内容

### 后端

- 新增 `GET /api/tenants/{tenant_id}/diagnostic-bundle`。
- 权限要求：当前用户必须属于同一租户，并具备 `ops.metrics.read` 权限。
- 返回内容包括：
  - 诊断包版本、生成时间、文件名。
  - 租户基础状态。
  - 运行环境摘要。
  - 基础资源计数。
  - 知识库状态摘要。
  - 质量评测状态摘要。
  - 渠道账号与连接器状态摘要。
  - 队列、worker、近期错误、近期变更。
  - 安全扫描结果。
- 诊断包不会主动调用外部平台，不会触发模型调用，不会触发真实外发。

### 前端

- 在“管理运维 -> 系统设置”加入“本地诊断包”卡片。
- 管理员点击“生成并下载”后，会从后端读取诊断包并下载 JSON 文件。
- 下载文件名沿用后端返回的 `filename`，格式类似 `wanfa-diagnostic-tenant-YYYYMMDD-HHMMSS.json`。
- 无 `ops.metrics.read` 权限的用户看不到导出入口。

### 安全边界

诊断包默认排除：

- secrets、token、cookie、密码、私钥、EncodingAESKey、Bearer 值、API Key。
- 原始完整客户聊天内容。
- 外部客户真实唯一标识。
- 完整渠道 payload。
- 草稿回复全文。
- 知识文档原文。

安全扫描会检查疑似凭据值；如发现疑似凭据，诊断包状态会变为 `needs_review`，并提示人工复核。

## 3. 为什么这样设计

客户本地部署后，我方不能默认长期进入客户电脑，也不能把客户聊天、店铺授权、渠道密钥直接带回我方。诊断包的正确定位是“问题摘要和运行证据”，不是“客户数据库拷贝”。

本片先把可安全导出的运维证据跑通，后续再做：

- 客户授权上传。
- 我方运维后台接收。
- 知识更新包回传。
- 策略更新包回传。
- 签名程序更新包。
- 备份和回滚。

## 6. 下一片建议

下一步建议进入 P3-06U-26H2D：知识更新包导入第一片。

优先做三件事：

1. 定义知识更新包 JSON 格式。
2. 在导入前自动备份当前知识状态。
3. 导入后跑知识评测回归，并给出是否可发布的结果。

客户授权上传可以稍后做，因为先把“客户把诊断包发给我们，我们回传知识修复包，客户本地导入”的最小闭环跑通，更符合本地应用试点节奏。
